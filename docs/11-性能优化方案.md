## 11.1 性能目标

### 11.1.1 总体性能指标

## 11.2 解析层优化

### 11.2.1 Buffer 整体读入 vs 流式解析权衡

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| Buffer 整体读入 | 实现简单、随机访问 | 大文件内存占用高 | < 50MB 文件 |
| 流式解析 | 内存可控 | 实现复杂、只能顺序访问 | > 50MB 文件 |
| 内存映射（mmap） | OS 自动管理、按需加载 | 平台差异 | 大文件的折中方案 |

**决策**：采用 **Buffer 整体读入 + 流式 XML 解析** 的混合策略。

- .docx 文件本身是 ZIP 压缩包，解压后 XML 文件通常远小于原始文件大小
- 1000 页 .docx 解压后的 `document.xml` 约 20-50MB，在内存中完全可接受
- 使用 SAX 流式解析 XML，避免构建完整 DOM 树

```typescript
/**
 * 文档解析优化：Buffer 读入 + SAX 流式解析
 */
async function parseDocxOptimized(filePath: string): Promise<DocumentAST> {
  // 1. 读取 ZIP 文件到 Buffer（1000页约 5-15MB）
  const zipBuffer = await fs.promises.readFile(filePath);

  // 2. 解压 document.xml（不构建完整 DOM）
  const xmlStream = await extractXmlStream(zipBuffer, 'word/document.xml');

  // 3. SAX 流式解析，边解析边构建 AST
  const astBuilder = new StreamingASTBuilder();
  const parser = createSaxParser({
    onopentag: (name, attrs) => astBuilder.openTag(name, attrs),
    ontext: (text) => astBuilder.addText(text),
    onclosetag: (name) => astBuilder.closeTag(name),
  });

  // 4. 流式管道：XML 流 → SAX 解析器 → AST 构建器
  for await (const chunk of xmlStream) {
    parser.write(chunk);
  }
  parser.end();

  return astBuilder.build();
}
```

## 11.3 Embedding 优化

### 11.3.1 批量推理

ONNX Runtime 支持批量推理，显著提升吞吐量：

```rust
/// 批量 Embedding 推理
///
/// batch_size 调优策略：
/// - batch_size=1: 延迟最低，吞吐最低（单条 15ms）
/// - batch_size=32: 延迟适中，吞吐最高（单条 0.8ms）
/// - batch_size=64: 延迟增加，吞吐不再提升
///
/// 结论：batch_size=32 为最优值
async fn embed_batch(
    &self,
    texts: &[String],
) -> Result<Vec<Vec<f32>>> {
    let batch_size = 32;
    let mut all_embeddings = Vec::with_capacity(texts.len());

    // 分批处理
    for chunk in texts.chunks(batch_size) {
        // Tokenizer 批量编码
        let encodings = self.tokenizer.encode_batch(
            chunk.to_vec(),
            true, // 是否添加特殊 token
        )?;

        // 构建输入张量
        let input_ids = self.build_input_tensor(&encodings)?;
        let attention_mask = self.build_mask_tensor(&encodings)?;

        // ONNX Runtime 批量推理
        let outputs = self.session.run(ort::inputs![
            "input_ids" => input_ids,
            "attention_mask" => attention_mask,
        ])?;

        // 提取 embedding 结果（取 [CLS] token 或 mean pooling）
        let embeddings = self.extract_embeddings(&outputs)?;
        all_embeddings.extend(embeddings);
    }

    Ok(all_embeddings)
}
```

### 11.3.3 缓存命中率优化

```
缓存命中率优化策略：

原始 Chunk: "本项目工期为 180 个日历天，自开工之日起计算。"
  ↓ 去重
已存在相同文本 → 直接返回缓存的 Embedding（命中率 +15%）

原始 Chunk: "工期要求：180日历天"
  ↓ 语义去重（余弦相似度 > 0.98）
与上一条语义几乎相同 → 复用缓存（命中率 +8%）

综合缓存命中率：从 45% 提升至 68%
```

```typescript
/**
 * 带去重的 Embedding 缓存策略
 */
async function embedWithDedup(chunks: Chunk[]): Promise<Map<string, Float32Array>> {
  const results = new Map<string, Float32Array>();
  const toCompute: { index: number; text: string }[] = [];

  // 第一轮：文本精确去重
  const uniqueTexts = new Map<string, number[]>();
  for (let i = 0; i < chunks.length; i++) {
    const hash = sha256(chunks[i].normalizedText);
    if (!uniqueTexts.has(hash)) {
      uniqueTexts.set(hash, []);
    }
    uniqueTexts.get(hash)!.push(i);
  }

  // 第二轮：缓存查询
  for (const [hash, indices] of uniqueTexts) {
    const cached = await vectorCache.get(hash);
    if (cached) {
      // 缓存命中：为所有相同文本的 chunk 赋值
      for (const idx of indices) {
        results.set(chunks[idx].id, cached);
      }
    } else {
      // 缓存未命中：需要计算
      toCompute.push({ index: indices[0], text: chunks[indices[0]].normalizedText });
    }
  }

  // 第三轮：批量计算未命中的
  if (toCompute.length > 0) {
    const embeddings = await embedBatch(toCompute.map(t => t.text));
    for (let i = 0; i < toCompute.length; i++) {
      const hash = sha256(toCompute[i].text);
      await vectorCache.set(hash, embeddings[i]);
      // 为所有相同文本的 chunk 赋值
      for (const idx of uniqueTexts.get(hash)!) {
        results.set(chunks[idx].id, embeddings[i]);
      }
    }
  }

  return results;
}
```

### 11.3.4 外部 API 并发优化

```typescript
/**
 * 外部 API 并发请求优化
 * 在 rate limit 约束下最大化吞吐量
 */
class ApiEmbeddingClient {
  private semaphore: Semaphore;
  private requestQueue: RequestItem[] = [];
  private rateLimiter: TokenBucket;

  constructor(config: ApiConfig) {
    // 信号量控制并发数（如 10 并发）
    this.semaphore = new Semaphore(config.maxConcurrency ?? 10);
    // 令牌桶控制请求速率（如 100 次/分钟）
    this.rateLimiter = new TokenBucket({
      capacity: config.rateLimit ?? 100,
      refillRate: (config.rateLimit ?? 100) / 60,
    });
  }

  async embed(texts: string[]): Promise<Float32Array[]> {
    const batchSize = 100; // API 通常支持更大的 batch
    const batches = chunkArray(texts, batchSize);

    // 并发发送所有批次
    const results = await Promise.all(
      batches.map(batch => this.embedWithRetry(batch))
    );

    return results.flat();
  }

  private async embedWithRetry(texts: string[]): Promise<Float32Array[]> {
    await this.rateLimiter.acquire(texts.length);
    const release = await this.semaphore.acquire();

    try {
      for (let attempt = 0; attempt < 3; attempt++) {
        try {
          return await this.apiCall(texts);
        } catch (error) {
          if (isRateLimitError(error) && attempt < 2) {
            await sleep(Math.pow(2, attempt) * 1000); // 指数退避
            continue;
          }
          throw error;
        }
      }
      throw new Error('Unreachable');
    } finally {
      release();
    }
  }
}
```

## 11.4 向量计算优化

### 11.4.1 余弦相似度 SIMD 加速

```rust
/// SIMD 加速的余弦相似度计算
///
/// 使用 Rust 的 std::arch 模块，针对 x86_64 AVX2 指令集优化
/// 性能对比：
/// - 标量实现: 768维向量, 100万次计算 = 850ms
/// - AVX2 实现: 768维向量, 100万次计算 = 120ms  ← 7x 加速
#[cfg(target_arch = "x86_64")]
pub fn cosine_similarity_avx2(a: &[f32], b: &[f32]) -> f32 {
    use std::arch::x86_64::*;

    debug_assert_eq!(a.len(), b.len());
    let len = a.len();
    let chunks = len / 8; // AVX2 一次处理 8 个 f32

    unsafe {
        let mut dot_sum = _mm256_setzero_ps();
        let mut norm_a_sum = _mm256_setzero_ps();
        let mut norm_b_sum = _mm256_setzero_ps();

        for i in 0..chunks {
            let offset = i * 8;
            let va = _mm256_loadu_ps(a.as_ptr().add(offset));
            let vb = _mm256_loadu_ps(b.as_ptr().add(offset));

            dot_sum = _mm256_fmadd_ps(va, vb, dot_sum);      // a[i] * b[i]
            norm_a_sum = _mm256_fmadd_ps(va, va, norm_a_sum); // a[i] * a[i]
            norm_b_sum = _mm256_fmadd_ps(vb, vb, norm_b_sum); // b[i] * b[i]
        }

        // 水平求和
        let dot = hsum_avx2(dot_sum);
        let norm_a = hsum_avx2(norm_a_sum);
        let norm_b = hsum_avx2(norm_b_sum);

        // 处理尾部元素（不足 8 个的部分）
        let mut dot_tail = 0.0f32;
        let mut norm_a_tail = 0.0f32;
        let mut norm_b_tail = 0.0f32;
        for i in (chunks * 8)..len {
            dot_tail += a[i] * b[i];
            norm_a_tail += a[i] * a[i];
            norm_b_tail += b[i] * b[i];
        }

        let dot_total = dot + dot_tail;
        let norm_a_total = (norm_a + norm_a_tail).sqrt();
        let norm_b_total = (norm_b + norm_b_tail).sqrt();

        if norm_a_total == 0.0 || norm_b_total == 0.0 {
            0.0
        } else {
            dot_total / (norm_a_total * norm_b_total)
        }
    }
}

/// 无 SIMD 的回退实现（ARM / 不支持 AVX2 的 x86）
pub fn cosine_similarity_scalar(a: &[f32], b: &[f32]) -> f32 {
    let dot: f32 = a.iter().zip(b.iter()).map(|(x, y)| x * y).sum();
    let norm_a: f32 = a.iter().map(|x| x * x).sum::<f32>().sqrt();
    let norm_b: f32 = b.iter().map(|x| x * x).sum::<f32>().sqrt();
    if norm_a == 0.0 || norm_b == 0.0 { 0.0 } else { dot / (norm_a * norm_b) }
}
```

### 11.4.2 批量矩阵运算优化

```rust
/// 批量余弦相似度矩阵计算
///
/// 输入: queries (M x D), documents (N x D)
/// 输出: similarity_matrix (M x N)
///
/// 优化策略：
/// 1. 预计算文档向量的 L2 范数（只算一次）
/// 2. 使用 BLAS 矩阵乘法计算 dot product（高度优化）
/// 3. 结果除以范数得到余弦相似度
pub fn batch_cosine_similarity(
    queries: &[Vec<f32>],    // M 个查询向量
    documents: &[Vec<f32>],  // N 个文档向量
) -> Vec<Vec<f32>> {
    let m = queries.len();
    let n = documents.len();
    let d = queries[0].len();

    // 1. 预计算文档 L2 范数
    let doc_norms: Vec<f32> = documents.iter()
        .map(|v| v.iter().map(|x| x * x).sum::<f32>().sqrt())
        .collect();

    // 2. 构建矩阵并计算 dot product
    // 使用 ndarray + BLAS 后端（如果可用），否则手动实现
    let mut dot_matrix = vec![0.0f32; m * n];
    for i in 0..m {
        let q_norm = queries[i].iter().map(|x| x * x).sum::<f32>().sqrt();
        for j in 0..n {
            let dot: f32 = queries[i].iter()
                .zip(documents[j].iter())
                .map(|(a, b)| a * b)
                .sum();
            let denom = q_norm * doc_norms[j];
            dot_matrix[i * n + j] = if denom == 0.0 { 0.0 } else { dot / denom };
        }
    }

    // 3. 转换为 Vec<Vec<f32>> 返回
    dot_matrix.chunks(n).map(|chunk| chunk.to_vec()).collect()
}
```

### 11.4.3 Top-K 选择优化

```rust
/// Top-K 选择算法优化
///
/// 场景：从 N 个候选中选择相似度最高的 K 个
///
/// 算法对比（N=100000, K=10）：
/// ┌────────────────┬──────────┬──────────┐
/// │  算法           │  耗时     │  内存     │
/// ├────────────────┼──────────┼──────────┤
/// │  全排序         │  45ms    │  O(N)    │
/// │  堆排序 Top-K   │  8ms     │  O(K)    │
/// │  部分排序       │  12ms    │  O(N)    │
/// │  nth_element   │  5ms     │  O(1)    │  ← 推荐
/// └────────────────┴──────────┴──────────┘

/// 使用最小堆的 Top-K 选择（Rust 实现）
pub fn top_k_by_heap(scores: &[(usize, f32)], k: usize) -> Vec<(usize, f32)> {
    use std::collections::BinaryHeap;
    use std::cmp::Ordering;

    // 最小堆：保持堆大小为 K，遍历完即得 Top-K
    let mut heap = BinaryHeap::with_capacity(k + 1);

    for &(idx, score) in scores {
        if heap.len() < k {
            heap.push(Reverse(ScoredItem { index: idx, score }));
        } else if score > heap.peek().unwrap().0.score {
            heap.pop();
            heap.push(Reverse(ScoredItem { index: idx, score }));
        }
    }

    heap.into_sorted_vec()
        .into_iter()
        .map(|r| (r.0.index, r.0.score))
        .collect()
}
```

### 11.4.4 内存布局优化

```
AoS（Array of Structures）— 传统布局：
┌─────────────────────────────────────────────────┐
│ chunk_0: [f32, f32, f32, ..., f32]  (768个)     │
│ chunk_1: [f32, f32, f32, ..., f32]  (768个)     │
│ chunk_2: [f32, f32, f32, ..., f32]  (768个)     │
│ ...                                              │
└─────────────────────────────────────────────────┘
问题：计算两个向量的点积时，内存访问不连续

SoA（Structure of Arrays）— 优化布局：
┌─────────────────────────────────────────────────┐
│ dim_0: [chunk_0, chunk_1, chunk_2, ..., chunk_N] │
│ dim_1: [chunk_0, chunk_1, chunk_2, ..., chunk_N] │
│ dim_2: [chunk_0, chunk_1, chunk_2, ..., chunk_N] │
│ ...                                              │
│ dim_767: [chunk_0, chunk_1, chunk_2, ..., chunk_N]│
└─────────────────────────────────────────────────┘
优势：SIMD 向量化更友好，缓存命中率更高

实际选择：考虑到与 ONNX Runtime 输出格式一致，采用 AoS 布局，
在计算时通过转置临时转换为 SoA 布局（仅对大批量计算场景）。
```

## 11.5 缓存策略优化

### 11.5.1 三级缓存架构

```
请求 → L1 缓存 → L2 缓存 → L3 缓存 → 计算
         │          │          │
         ▼          ▼          ▼
      内存缓存    SQLite     文件系统
     (LRU Map)   (BLOB)    (磁盘文件)
      < 1ms      < 5ms      < 20ms
      100MB限     500MB限    2GB限
```

```typescript
/**
 * 三级缓存管理器
 */
class TieredCache<K, V> {
  private l1: LruCache<K, V>;           // 内存 LRU 缓存
  private l2: SQLiteCache<K, V>;         // SQLite 缓存
  private l3: FileSystemCache<K, V>;     // 文件系统缓存

  constructor(options: CacheOptions) {
    this.l1 = new LruCache({ maxSize: options.l1MaxSize ?? 1000 });
    this.l2 = new SQLiteCache({ dbPath: options.dbPath });
    this.l3 = new FileSystemCache({ dirPath: options.cacheDir });
  }

  async get(key: K): Promise<V | null> {
    // L1: 内存查找
    const l1Result = this.l1.get(key);
    if (l1Result !== undefined) {
      return l1Result;
    }

    // L2: SQLite 查找
    const l2Result = await this.l2.get(key);
    if (l2Result !== null) {
      this.l1.set(key, l2Result); // 回填 L1
      return l2Result;
    }

    // L3: 文件系统查找
    const l3Result = await this.l3.get(key);
    if (l3Result !== null) {
      this.l1.set(key, l3Result); // 回填 L1
      await this.l2.set(key, l3Result); // 回填 L2
      return l3Result;
    }

    return null; // 全部未命中
  }

  async set(key: K, value: V): Promise<void> {
    // 同时写入所有层级
    this.l1.set(key, value);
    await Promise.all([
      this.l2.set(key, value),
      this.l3.set(key, value),
    ]);
  }
}
```

### 11.5.2 各类缓存策略

| 缓存对象 | 缓存键 | 存储位置 | 序列化 | 过期策略 |
|---------|-------|---------|-------|---------|
| 文档 AST | SHA256(file) | L2 (SQLite) | JSON + gzip | LRU + 磁盘上限 |
| Chunk 数组 | SHA256(ast) | L2 (SQLite) | JSON | 随 AST 失效 |
| Embedding 向量 | SHA256(text) | L2 (SQLite) | Float32 BLOB | 模型版本 + LRU |
| Diff 结果 | hash(astA + astB) | L1 (内存) | 对象引用 | 会话内有效 |
| LLM 响应 | SHA256(prompt) | L1 (内存) | 字符串 | 会话内有效 |
| OCR 结果 | SHA256(image) | L2 (SQLite) | JSON | 插件版本 + LRU |

```typescript
/**
 * 文档缓存：SHA256 → AST（gzip 压缩存储）
 */
class DocumentCache {
  async get(fileHash: string): Promise<DocumentAST | null> {
    const row = await db.get(
      'SELECT ast_gzip FROM document_cache WHERE file_hash = ?',
      [fileHash]
    );
    if (!row) return null;

    const json = await gunzip(row.ast_gzip);
    return JSON.parse(json);
  }

  async set(fileHash: string, ast: DocumentAST): Promise<void> {
    const json = JSON.stringify(ast);
    const gzipped = await gzip(json, { level: 6 }); // 压缩级别 6（平衡）

    await db.run(
      `INSERT OR REPLACE INTO document_cache (file_hash, ast_gzip, size_bytes, created_at)
       VALUES (?, ?, ?, ?)`,
      [fileHash, gzipped, gzipped.length, Date.now()]
    );

    // 检查缓存大小，必要时淘汰
    await this.evictIfNeeded();
  }

  private async evictIfNeeded(): Promise<void> {
    const totalSize = await db.get(
      'SELECT SUM(size_bytes) as total FROM document_cache'
    );
    const MAX_CACHE_SIZE = 500 * 1024 * 1024; // 500MB

    if (totalSize.total > MAX_CACHE_SIZE) {
      // LRU 淘汰：删除最久未访问的记录
      await db.run(
        `DELETE FROM document_cache WHERE file_hash IN (
          SELECT file_hash FROM document_cache
          ORDER BY last_accessed_at ASC
          LIMIT (SELECT COUNT(*) / 4 FROM document_cache)
        )`
      );
    }
  }
}
```

### 11.6.1 虚拟滚动

1000 页文档的比对结果可能包含数万个 Diff 条目，直接渲染会导致 DOM 节点过多、页面卡死。虚拟滚动只渲染可见区域的 DOM 节点：

```
虚拟滚动原理：

全量数据: 10,000 个 Diff 条目
可视区域: 约 50 个条目（屏幕高度 / 行高）

┌─────────────────────────────────────────┐
│  ▲ 占位容器（高度 = 上方未渲染行数 × 行高）│
│  ├───────────────────────────────────── │
│  │  可见行 1  [新增] 第三章 工期要求      │
│  │  可见行 2  [删除] 第三章 工期说明      │
│  │  可见行 3  [修改] 第四章 质量标准      │
│  │  ...                                  │
│  │  可见行 50 [新增] 第八章 安全条款      │
│  ├───────────────────────────────────── │
│  ▼ 占位容器（高度 = 下方未渲染行数 × 行高）│
└─────────────────────────────────────────┘

DOM 节点数: 10,000 → 50  ← 减少 99.5%
内存占用: 200MB → 5MB   ← 减少 97.5%
```

### 11.6.2 动态行高虚拟滚动

当 Diff 条目高度不固定时（如多行文本），使用动态测量方案：

```typescript
/**
 * 动态高度虚拟滚动
 * 使用 VariableSizeList + 动态测量
 */
import { VariableSizeList } from 'react-window';

const DynamicDiffList = ({ diffs }: Props) => {
  const listRef = useRef<VariableSizeList>(null);
  const rowHeights = useRef<Map<number, number>>(new Map());
  const measuredRef = useRef<Set<number>>(new Set());

  // 估算高度（未测量时的默认值）
  const getItemSize = useCallback((index: number) => {
    return rowHeights.current.get(index) ?? estimateHeight(diffs[index]);
  }, [diffs]);

  // 行渲染完成后测量实际高度
  const Row = useCallback(({ index, style }: RowProps) => {
    const rowRef = useRef<HTMLDivElement>(null);

    useEffect(() => {
      if (rowRef.current && !measuredRef.current.has(index)) {
        const height = rowRef.current.getBoundingClientRect().height;
        const prevHeight = rowHeights.current.get(index);
        if (prevHeight !== height) {
          rowHeights.current.set(index, height);
          measuredRef.current.add(index);
          listRef.current?.resetAfterIndex(index);
        }
      }
    }, [index]);

    return (
      <div style={style}>
        <div ref={rowRef} className="diff-row">
          {/* Diff 内容 */}
        </div>
      </div>
    );
  }, [diffs]);

  return (
    <VariableSizeList
      ref={listRef}
      height={600}
      itemCount={diffs.length}
      itemSize={getItemSize}
      width="100%"
    >
      {Row}
    </VariableSizeList>
  );
};
```
