## 1. 文档目的

本文档（软件设计说明书）是 Nova BidLens 产品的核心技术文档，旨在为开发团队、测试团队、产品团队及后续维护人员提供完整、准确、可执行的软件设计蓝图。

**本文档要解决的核心问题：**

- 开发人员如何理解系统全局架构和各模块的详细设计？
- 新加入项目的工程师如何快速建立对系统的整体认知？
- 测试团队如何基于设计文档编写测试用例和验收标准？
- 产品团队如何理解技术边界和实现约束？
- 后续维护人员如何定位问题和进行功能扩展？

**文档定位：**

本文档介于《架构设计文档》（高层决策）和《代码实现》（具体编码）之间，是连接"做什么"和"怎么做"的桥梁。架构设计文档回答了"Why"（为什么选择这个方案），本文档回答了"How"（具体如何实现）。

## 4. 术语表

本文档中使用的专业术语及其定义。按字母顺序排列。

| # | 术语 | 英文 | 定义 |
|---|------|------|------|
| 1 | **Adapter 模式** | Adapter Pattern | Parser 层采用的设计模式。每种文档格式（Word/PDF/Markdown 等）对应一个适配器实现，统一接口为 `DocumentParser`，使新增格式支持无需修改核心逻辑。 |
| 2 | **AST** | Abstract Syntax Tree | 抽象语法树。在本文档中特指 Document AST——将文档内容解析为树形结构的统一内部表示，是整个系统的核心数据模型。 |
| 3 | **BGE-M3** | BGE-M3 | BAAI 发布的多语言多粒度 Embedding 模型，支持中英文等 100+ 语言，向量维度 1024，是 BidLens 推荐的本地 Embedding 模型。 |
| 4 | **Bridge 层** | Bridge Layer | Rust 引擎中负责 stdio JSON-RPC 通信的入口模块（`bidlens-engine/src/main.rs`），将 JSON 请求转换为 Rust 核心函数调用，再将结果序列化为 JSON 响应。逻辑上与核心 crate 解耦，便于独立测试。 |
| 5 | **Chunk** | Chunk | 文档分块。将 Document AST 按照一定策略（Section 边界、段落边界、Token 上限）切分为若干语义单元，每个 Chunk 是 Embedding 推理和向量检索的最小单位。典型大小 256-512 Token。 |
| 6 | **Crate** | Crate | Rust 的编译单元，类似于其他语言中的"包"或"库"。BidLens Rust 引擎采用 Workspace 多 Crate 结构，每个 Crate 职责单一。 |
| 7 | **Diff AST** | Diff Abstract Syntax Tree | 比对引擎的输出数据结构。仅存储匹配关系和差异引用（不含原始 AST），渲染时从 document_cache 取原始内容。包含 Match、Unmatched、Statistics 三大部分。 |
| 8 | **Document AST** | Document Abstract Syntax Tree | 文档抽象语法树。BidLens 的核心数据模型，采用 JSON 格式，完整表示文档的层级结构（文档→章节→段落→文本片段）、格式信息、批注、修订等。达到 L5 全保真级别。 |
| 9 | **Embedding** | Embedding | 向量化。将文本（Chunk）通过 AI 模型转换为固定维度的浮点数向量的过程。生成的向量在语义空间中表示文本含义，语义相近的文本向量距离更近。 |
| 10 | **Electron** | Electron | 基于 Chromium 和 Node.js 的跨平台桌面应用框架。BidLens 使用 Electron 构建桌面客户端，主进程负责系统级操作，渲染进程运行 React UI。 |
| 11 | **GGUF** | GPT-Generated Unified Format | llama.cpp 使用的模型文件格式，量化后体积小、加载快。BidLens 本地 LLM 模型采用此格式，推荐 Qwen/DeepSeek 系列。 |
| 12 | **GTE** | General Text Embedding | 阿里达摩院发布的通用文本 Embedding 模型，与 BGE-M3 同为 BidLens 支持的本地 Embedding 模型选项。 |
| 13 | **HNSW** | Hierarchical Navigable Small World | 分层可导航小世界图，一种高效的近似最近邻（ANN）向量索引算法。BidLens 第一阶段使用 SQLite 暴力搜索，后续版本将引入 HNSW 索引提升大规模检索性能。 |
| 14 | **IPC** | Inter-Process Communication | 进程间通信。在 Electron 中特指主进程（Main Process）与渲染进程（Renderer Process）之间的通信机制，基于 `ipcMain` / `ipcRenderer`。 |
| 15 | **JSON-RPC** | JSON Remote Procedure Call | 基于 JSON 的远程过程调用协议。BidLens 中 Node.js 层与 Rust 子进程之间采用 stdio（标准输入输出）传输的 JSON-RPC 协议进行通信，每条消息一行 JSON，以 `\n` 结尾。 |
| 16 | **L5 全保真** | Fidelity Level 5 | BidLens 定义的最高文档保真级别。L1（基础格式）→ L2（段落格式）→ L3（页眉页脚）→ L4（绝对定位）→ L5（批注+修订）。用户可在设置中选择保真级别，降低级别可提升解析性能。 |
| 17 | **llama.cpp** | llama.cpp | Georgi Gerganov 开发的轻量级 LLM 推理引擎，支持 GGUF 格式模型在 CPU/GPU 上高效推理。BidLens 本地 LLM 推理基于此引擎。 |
| 18 | **LRU 缓存** | Least Recently Used Cache | 最近最少使用缓存淘汰策略。BidLens 在模型管理中采用 LRU 策略：当内存不足以加载新模型时，自动卸载最久未使用的模型。 |
| 19 | **Match** | Match | Diff AST 中的核心概念，表示文档 A 的一组节点与文档 B 的一组节点之间的对应关系。Match 类型包括 1:1（一对一）、1:N（一对多/拆分）、N:1（多对一/合并）、N:M（碎片匹配）。 |
| 20 | **Myers Diff** | Myers Diff Algorithm | Eugene Myers 1986 年提出的差分算法，是 Unix `diff` 命令的基础算法。BidLens 的文本 Diff 基于此算法改进，针对中文字符和混合文本优化。 |
| 21 | **ONNX Runtime** | Open Neural Network Exchange Runtime | 微软开源的跨平台 AI 推理引擎。BidLens 使用 ONNX Runtime 运行本地 Embedding 模型（BGE-M3/GTE）和 OCR 模型（PaddleOCR），支持 CPU/GPU 加速。 |
| 22 | **Overlap** | Overlap | Chunk 之间的重叠区域。相邻 Chunk 共享一定比例（默认 10%）的文本，防止语义在分块边界处被切断，提高 Embedding 质量和检索召回率。 |
| 23 | **PaddleOCR** | PaddleOCR | 百度飞桨开源的 OCR 工具包。BidLens 通过 ONNX Runtime 运行 PaddleOCR 模型，用于扫描件（图片）的文字识别。 |
| 24 | **Parser** | Parser | 文档解析器。将各种格式的文档文件（.docx/.pdf/.md 等）解析为统一的 Document AST。在 Node.js 层实现，采用适配器模式，每种格式一个适配器。 |
| 25 | **Preload 脚本** | Preload Script | Electron 中在渲染进程加载网页前执行的脚本，用于安全地暴露 Node.js API 给渲染进程。BidLens 通过 Preload 脚本暴露 IPC 通信接口，遵循最小权限原则。 |
| 26 | **Rerank** | Re-ranking | 精排/重排序。在 Embedding 粗排（Top-K 候选）之后，使用更精确的模型（如 Cross-Encoder）对候选匹配对重新打分，提高匹配准确率。精确模式下启用，额外耗时 10-30 秒。 |
| 27 | **Renderer 进程** | Renderer Process | Electron 中运行 Web 页面的进程。BidLens 的 React 前端运行在此进程中，负责 UI 渲染、用户交互、状态管理。与主进程通过 IPC 通信。 |
| 28 | **shadcn/ui** | shadcn/ui | 基于 Radix UI 和 TailwindCSS 的可复用 UI 组件库。不是传统意义上的 npm 包，而是将组件源码直接复制到项目中，便于完全自定义。BidLens 的 UI 基于此组件库构建。 |
| 29 | **SQLite** | SQLite | 轻量级嵌入式关系数据库。BidLens 使用 SQLite 存储文档缓存、Embedding 缓存、比对历史、模型注册信息和用户配置，无需独立数据库服务。 |
| 30 | **stdio** | Standard Input/Output | 标准输入输出。BidLens 中 Node.js 通过 `child_process.spawn` 启动 Rust 子进程，两者通过 stdin/stdout 传输 JSON-RPC 消息进行通信。 |
| 31 | **TanStack Query** | TanStack Query (React Query) | React 的异步状态管理库。BidLens 使用 TanStack Query 管理与 Rust 引擎的 RPC 调用、模型下载进度、比对任务状态等异步数据，提供缓存、乐观更新、重试等能力。 |
| 32 | **TextRun** | Text Run | Document AST 中的行内文本节点，表示一段具有相同样式属性的连续文本。一个段落（Paragraph）通常包含多个 TextRun，例如"**加粗文字**普通文字"包含两个 TextRun。 |
| 33 | **Token** | Token | AI 模型处理文本的基本单位。中文通常一个字为 1-2 个 Token，英文一个单词为 1-3 个 Token。BidLens 的 Chunk 大小以 Token 计量（256-512 Token）。 |
| 34 | **Top-K** | Top-K | 从向量检索结果中取相似度最高的 K 个候选。BidLens 在 Embedding 匹配阶段对每个 Chunk 取 Top-K（默认 K=5）个最相似的候选，再通过阈值过滤确定最终匹配。 |
| 35 | **Workspace** | Cargo Workspace | Rust 的多包管理工作区。BidLens Rust 引擎使用 Workspace 管理多个 Crate（document-ast、diff-engine、embedding、vector、chunk、common、bridge），共享依赖版本和编译目标。 |
| 36 | **Zustand** | Zustand | 轻量级 React 状态管理库。BidLens 使用 Zustand 管理全局 UI 状态（如当前比对配置、选中的文档、UI 偏好设置），相比 Redux 更简洁，无 Boilerplate。 |
| 37 | **余弦相似度** | Cosine Similarity | 衡量两个向量方向相似性的指标，值域 [-1, 1]。BidLens 使用余弦相似度计算两个 Chunk 向量的语义相似程度，值越接近 1 表示语义越相似。 |
| 38 | **向量检索** | Vector Search / Similarity Search | 在向量数据库中查找与查询向量最相似的 Top-K 个向量的过程。BidLens 第一阶段在 SQLite 中存储向量并暴力搜索，后续引入 HNSW 索引。 |
