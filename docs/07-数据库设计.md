# 7. 数据库设计

## 7.1 数据库架构总览

BidLens 使用 **SQLite** 作为本地持久化存储，承载文档缓存、向量缓存、比对历史、模型管理和系统配置五大核心数据域。SQLite 以单文件形式嵌入应用，无需额外部署数据库服务，完美契合桌面应用场景。

### 7.1.1 技术选型依据

| 维度 | SQLite | LevelDB/RocksDB | PostgreSQL |
|------|--------|-----------------|------------|
| 部署复杂度 | 零依赖，单文件 | 零依赖，KV 存储 | 需要独立服务 |
| SQL 支持 | 完整 SQL 子集 | 无 SQL，纯 KV | 完整 SQL |
| 并发性能 | WAL 模式多读单写 | 高并发写入 | 高并发读写 |
| 数据类型 | 动态类型 + BLOB | 仅 bytes | 丰富类型 |
| 事务支持 | 完整 ACID | 原子批量写入 | 完整 ACID |
| 适用场景 | 桌面应用，中小数据量 | 高写入吞吐 | 企业级应用 |

### 7.1.2 存储引擎配置

```sql
-- 每次连接建立时执行的 PRAGMA 配置
PRAGMA journal_mode = WAL;           -- Write-Ahead Logging，提升并发性能
PRAGMA synchronous = NORMAL;         -- WAL 模式下 NORMAL 即可保证数据安全
PRAGMA cache_size = -8000;           -- 8MB 页缓存（负值单位为 KB）
PRAGMA foreign_keys = ON;            -- 启用外键约束
PRAGMA temp_store = MEMORY;          -- 临时表使用内存存储
PRAGMA mmap_size = 268435456;        -- 256MB 内存映射（提升大表读取性能）
PRAGMA busy_timeout = 5000;          -- 写锁等待超时 5 秒
PRAGMA user_version = 1;             -- Schema 版本号，用于迁移管理
```

### 7.1.3 数据库文件位置

```
Windows:  %APPDATA%/BidLens/data/bidlens.db
macOS:    ~/Library/Application Support/BidLens/data/bidlens.db
Linux:    ~/.config/BidLens/data/bidlens.db
```

数据库文件预计大小：
- **初始状态**：< 1MB（仅 Schema + 配置）
- **典型使用**：100-500MB（含文档缓存和向量缓存）
- **重度使用**：1-3GB（大量历史比对和模型元数据）

## 7.3 表结构详细设计

### 7.3.1 `document_cache` — 文档缓存表

**用途**：缓存已解析的文档 AST，避免重复解析。相同文件（SHA256 匹配）直接复用缓存。

```sql
CREATE TABLE IF NOT EXISTS document_cache (
    doc_hash      TEXT PRIMARY KEY,              -- SHA256 哈希，文档唯一标识
    filename      TEXT NOT NULL,                 -- 原始文件名（仅用于显示）
    format        TEXT NOT NULL,                 -- 文件格式: 'docx' | 'pdf' | 'wps'
    size_bytes    INTEGER NOT NULL,              -- 原始文件大小（字节）
    ast_blob      BLOB NOT NULL,                 -- gzip 压缩后的 AST JSON
    ast_size      INTEGER NOT NULL,              -- 原始 AST JSON 大小（未压缩，字节）
    page_count    INTEGER,                       -- 文档页数（可选，解析时估算）
    access_count  INTEGER NOT NULL DEFAULT 1,    -- 访问次数，用于 LRU 淘汰
    last_accessed_at TEXT NOT NULL,              -- 最后访问时间 ISO 8601
    created_at    TEXT NOT NULL DEFAULT (datetime('now')),  -- 创建时间
    updated_at    TEXT NOT NULL DEFAULT (datetime('now'))   -- 更新时间
);

-- 索引：按最后访问时间排序（LRU 淘汰查询）
CREATE INDEX IF NOT EXISTS idx_document_cache_last_access
    ON document_cache(last_accessed_at ASC);

-- 索引：按文件格式分组统计
CREATE INDEX IF NOT EXISTS idx_document_cache_format
    ON document_cache(format);
```

**字段详细说明**：

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| doc_hash | TEXT | PK, NOT NULL | SHA256(file_content)，64 字符十六进制字符串 |
| filename | TEXT | NOT NULL | 原始文件名，仅用于 UI 显示，不影响缓存命中 |
| format | TEXT | NOT NULL, CHECK IN ('docx','pdf','wps','doc') | 文件格式枚举 |
| size_bytes | INTEGER | NOT NULL, CHECK > 0 | 原始文件字节数 |
| ast_blob | BLOB | NOT NULL | gzip 压缩的 AST JSON，典型大小 50KB-5MB |
| ast_size | INTEGER | NOT NULL | 未压缩 AST 大小，用于预估内存占用 |
| page_count | INTEGER | NULL | 文档页数，PDF 可精确获取，DOCX 估算 |
| access_count | INTEGER | NOT NULL, DEFAULT 1 | 每次命中缓存时 +1 |
| last_accessed_at | TEXT | NOT NULL | ISO 8601 时间戳，每次访问时更新 |

### 7.3.2 `embedding_cache` — 向量缓存表

**用途**：缓存文档各 chunk 的 Embedding 向量，避免重复推理。向量以 Float32 数组 BLOB 存储。

```sql
CREATE TABLE IF NOT EXISTS embedding_cache (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    doc_hash      TEXT NOT NULL,                  -- 关联 document_cache.doc_hash
    chunk_index   INTEGER NOT NULL,               -- chunk 在文档中的序号（从 0 开始）
    chunk_text    TEXT NOT NULL,                   -- chunk 原文（用于调试和 rerank）
    chunk_tokens  INTEGER NOT NULL,               -- chunk token 数量
    embedding     BLOB NOT NULL,                   -- Float32 向量，dim * 4 字节
    embedding_dim INTEGER NOT NULL,                -- 向量维度（如 768, 1024）
    model_id      TEXT NOT NULL,                   -- 使用的 embedding 模型 ID
    start_node_id TEXT,                            -- AST 起始节点 ID
    end_node_id   TEXT,                            -- AST 结束节点 ID
    created_at    TEXT NOT NULL DEFAULT (datetime('now')),

    -- 同一文档的同一 chunk 用同一模型只缓存一份
    UNIQUE(doc_hash, chunk_index, model_id)
);

-- 索引：按 doc_hash 查询（最频繁的查询模式）
CREATE INDEX IF NOT EXISTS idx_embedding_cache_doc_hash
    ON embedding_cache(doc_hash, model_id);

-- 索引：按模型查询（模型清理时使用）
CREATE INDEX IF NOT EXISTS idx_embedding_cache_model
    ON embedding_cache(model_id);
```

**BLOB 格式说明**：
```
embedding 字段存储格式：IEEE 754 Float32 小端序数组
- 维度 768:  BLOB 大小 = 768 * 4 = 3072 字节
- 维度 1024: BLOB 大小 = 1024 * 4 = 4096 字节

写入 (Rust → SQLite):
  let vec: Vec<f32> = embedding_result;
  let bytes: &[u8] = bytemuck::cast_slice(&vec);
  stmt.execute(params![..., bytes, ...])?;

读取 (SQLite → Rust):
  let bytes: &[u8] = row.get("embedding")?;
  let vec: &[f32] = bytemuck::cast_slice(bytes);
```

### 7.3.3 `compare_history` — 比对历史表

**用途**：记录每次比对任务的结果摘要，支持历史查看和统计分析。

```sql
CREATE TABLE IF NOT EXISTS compare_history (
    id              TEXT PRIMARY KEY,                -- UUID v4
    doc_a_hash      TEXT NOT NULL,                   -- 文档 A 的 hash
    doc_b_hash      TEXT NOT NULL,                   -- 文档 B 的 hash
    doc_a_filename  TEXT NOT NULL,                   -- 文档 A 文件名
    doc_b_filename  TEXT NOT NULL,                   -- 文档 B 文件名
    threshold       REAL NOT NULL DEFAULT 0.75,      -- 匹配阈值
    mode            TEXT NOT NULL DEFAULT 'default',  -- 比对模式: 'default' | 'precise' | 'fast'
    result_json     TEXT,                            -- 比对结果 JSON（pairs 数组摘要）
    diff_ast        BLOB,                            -- 可选：gzip 压缩的 Diff AST（仅保存模式）
    stats_json      TEXT NOT NULL,                   -- 统计信息 JSON
    duration_ms     INTEGER NOT NULL,                -- 总耗时（毫秒）
    status          TEXT NOT NULL DEFAULT 'pending',  -- 状态: 'pending'|'running'|'completed'|'failed'|'cancelled'
    error_message   TEXT,                            -- 失败时的错误信息
    language        TEXT DEFAULT 'zh-CN',            -- 文档语言
    embedding_model TEXT,                            -- 使用的 embedding 模型
    created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 索引：按创建时间倒序（历史列表查询）
CREATE INDEX IF NOT EXISTS idx_compare_history_created
    ON compare_history(created_at DESC);

-- 索引：按文档 A hash 查询（查找某文档的所有比对记录）
CREATE INDEX IF NOT EXISTS idx_compare_history_doc_a
    ON compare_history(doc_a_hash, created_at DESC);

-- 索引：按文档 B hash 查询
CREATE INDEX IF NOT EXISTS idx_compare_history_doc_b
    ON compare_history(doc_b_hash, created_at DESC);

-- 索引：按状态查询（查找未完成的任务）
CREATE INDEX IF NOT EXISTS idx_compare_history_status
    ON compare_history(status) WHERE status IN ('pending', 'running');
```

**stats_json 字段结构**：
```json
{
  "total_a_nodes": 45,
  "total_b_nodes": 52,
  "total_pairs": 38,
  "avg_similarity": 0.82,
  "match_breakdown": {
    "one_to_one": 30,
    "one_to_many": 3,
    "many_to_one": 2,
    "many_to_many": 1,
    "unmatched_a": 4,
    "unmatched_b": 5
  },
  "order_changes": 2,
  "phase_times": {
    "parse_ms": 120,
    "chunk_ms": 50,
    "embed_ms": 2800,
    "match_ms": 800,
    "diff_ms": 430
  }
}
```

**diff_ast 持久化策略**：

`diff_ast` 字段存储 gzip 压缩后的 DiffAST JSON（类型定义参见 [[05-数据结构设计.md]] 第5.4节）。持久化行为受 `settings.auto_save_diff` 控制：

| 设置值 | 行为 | 适用场景 |
|--------|------|---------|
| `true` | 比对完成后自动将 DiffAST 存入 `diff_ast` BLOB | 需要反复查看历史结果 |
| `false`（默认） | 仅保存 `result_json` 摘要，不保存完整 DiffAST | 节省存储空间 |

**存储估算**：

| 文档规模 | DiffAST JSON 大小 | gzip 压缩后 | 200 条历史总占用 |
|---------|-------------------|-------------|-----------------|
| 50 页 × 2 | ~200KB | ~60KB | ~12MB |
| 200 页 × 2 | ~2MB | ~500KB | ~100MB |
| 1000 页 × 2 | ~15MB | ~3MB | ~600MB |

**读取流程**：
```
用户打开历史记录
  → 查询 compare_history 获取 diff_ast BLOB
  → gunzip 解压为 JSON
  → 反序列化为 DiffAST 对象
  → 通过 docAHash / docBHash 从 document_cache 获取原始 AST
  → 传递给前端渲染
```

> **注意**：如果 `diff_ast` 为空（用户未启用自动保存），系统需要从原始文档重新解析+比对。
> 建议在用户界面明确提示"此历史记录未保存完整结果，需要重新比对"。

### 7.3.4 `model_registry` — 模型管理表

**用途**：管理 AI 模型的下载状态、版本信息和存储路径。

```sql
CREATE TABLE IF NOT EXISTS model_registry (
    model_id          TEXT PRIMARY KEY,              -- 模型唯一标识，如 'bge-m3'
    name              TEXT NOT NULL,                 -- 显示名称，如 'BGE-M3 Multilingual'
    type              TEXT NOT NULL,                 -- 模型类型: 'embedding' | 'llm' | 'ocr' | 'reranker'

    -- 状态: 'not_downloaded' | 'downloading' | 'downloaded' | 'loading' | 'loaded' | 'error'
    file_path         TEXT,                          -- 本地文件路径
    file_size         INTEGER,                       -- 文件大小（字节）

    config_json       TEXT,                          -- 模型配置 JSON（维度、max_tokens 等）
    description       TEXT,                          -- 模型描述
    is_builtin        INTEGER NOT NULL DEFAULT 0,    -- 是否为内置模型（不可删除）
    last_used_at      TEXT,                          -- 最后使用时间
    created_at        TEXT NOT NULL DEFAULT (datetime('now')),
```

## 7.4 索引策略

### 7.4.1 主键索引

所有表的主键自动创建唯一索引：

| 表名 | 主键 | 类型 | 选择依据 |
|------|------|------|----------|
| document_cache | doc_hash | TEXT | SHA256 天然唯一，等值查询 O(log n) |
| embedding_cache | id | INTEGER AUTO | 自增整数，插入性能最优 |
| compare_history | id | TEXT UUID | 分布式 ID，无冲突 |
| model_registry | model_id | TEXT | 模型标识符，短字符串 |
| settings | key | TEXT | 配置键名，短字符串 |

### 7.4.2 二级索引分析

**高频查询模式与索引覆盖**：

```sql
-- 查询模式 1: 根据 doc_hash 获取 embedding（最频繁）
-- 频率: 每次比对执行 2-4 次
SELECT * FROM embedding_cache WHERE doc_hash = ? AND model_id = ?;
-- 索引: idx_embedding_cache_doc_hash(doc_hash, model_id) → 完全覆盖

-- 查询模式 2: 历史记录列表（按时间倒序）
-- 频率: 每次打开历史页面
SELECT * FROM compare_history ORDER BY created_at DESC LIMIT 50;
-- 索引: idx_compare_history_created(created_at DESC) → 覆盖

-- 查询模式 3: LRU 淘汰查询
-- 频率: 定期执行（缓存清理时）
SELECT doc_hash, ast_size FROM document_cache
ORDER BY access_count ASC, last_accessed_at ASC
LIMIT 10;
-- 索引: idx_document_cache_last_access(last_accessed_at) → 部分覆盖

-- 查询模式 4: 按模型查询 embedding 缓存
-- 频率: 模型卸载/清理时
SELECT COUNT(*), SUM(chunk_tokens) FROM embedding_cache WHERE model_id = ?;
-- 索引: idx_embedding_cache_model(model_id) → 覆盖
```

### 7.4.3 复合索引场景

```sql
-- 场景: 查找某文档使用特定模型的 embedding 缓存数量
CREATE INDEX IF NOT EXISTS idx_embedding_doc_model_count
    ON embedding_cache(doc_hash, model_id, chunk_index);

-- 场景: 按文档 hash 查找所有比对历史（两个文档可能互换位置）
CREATE INDEX IF NOT EXISTS idx_compare_both_docs
    ON compare_history(doc_a_hash, doc_b_hash, created_at DESC);
```

## 7.5 数据生命周期管理

### 7.5.1 `document_cache` LRU 淘汰

**淘汰策略**：基于 `access_count` + `last_accessed_at` 的加权 LRU。

```
淘汰优先级分数 = access_count * 0.3 + (1 / days_since_last_access) * 0.7

分数越低，越优先被淘汰。

示例：
- 文档 A: access_count=10, last_access=1天前 → 分数 = 3.0 + 0.7 = 3.7
- 文档 B: access_count=2,  last_access=30天前 → 分数 = 0.6 + 0.023 = 0.623
→ 文档 B 优先被淘汰
```

**淘汰触发条件**：
1. 缓存总大小超过 `max_cache_size_mb` 设置（默认 2GB）
2. 手动触发清理
3. 应用启动时的定期检查

### 7.5.2 `embedding_cache` 关联删除

```sql
-- 当 document_cache 记录被删除时，级联删除关联的 embedding_cache
-- 注意: SQLite 默认不支持 ON DELETE CASCADE（需外键约束）
-- BidLens 采用应用层级联删除策略

-- 删除文档缓存的事务
BEGIN;
  -- 1. 先删除 embedding 缓存
  DELETE FROM embedding_cache WHERE doc_hash = :doc_hash;
  -- 2. 再删除文档缓存
  DELETE FROM document_cache WHERE doc_hash = :doc_hash;
COMMIT;
```

### 7.5.3 `compare_history` 保留策略

```
保留规则：
1. 默认保留最近 200 条记录（可通过 settings.max_history_count 配置）
2. 状态为 'running' 或 'pending' 的记录永不自动删除
3. 用户手动标记为"收藏"的记录不参与自动清理

清理 SQL:
DELETE FROM compare_history
WHERE id NOT IN (
    SELECT id FROM compare_history
    WHERE status IN ('running', 'pending')
       OR id IN (SELECT key FROM settings WHERE key LIKE 'favorite_%')
    ORDER BY created_at DESC
    LIMIT :max_count
);
```

## 7.6 数据库迁移方案

### 7.6.1 Schema 版本管理

使用 SQLite 的 `user_version` PRAGMA 管理 Schema 版本：

```sql
-- 每次连接时检查版本
PRAGMA user_version;
-- 返回当前版本号（0 表示新建数据库）
```

**版本号规则**：`V1` = 初始版本，`V2` = 第一次迁移，以此类推。

### 7.6.2 迁移脚本设计

```typescript
// 迁移管理器
interface Migration {
  version: number;
  description: string;
  up: string[];      // 升级 SQL 列表
  down: string[];    // 降级 SQL 列表（可选）
}

const MIGRATIONS: Migration[] = [
  {
    version: 1,
    description: 'Initial schema',
    up: [
      `CREATE TABLE IF NOT EXISTS document_cache (...);`,
      `CREATE TABLE IF NOT EXISTS embedding_cache (...);`,
      `CREATE TABLE IF NOT EXISTS compare_history (...);`,
      `CREATE TABLE IF NOT EXISTS model_registry (...);`,
      `CREATE TABLE IF NOT EXISTS settings (...);`,
      // ... 所有建表语句和索引
    ],
    down: [
      `DROP TABLE IF EXISTS settings;`,
      `DROP TABLE IF EXISTS model_registry;`,
      `DROP TABLE IF EXISTS compare_history;`,
      `DROP TABLE IF EXISTS embedding_cache;`,
      `DROP TABLE IF EXISTS document_cache;`,
    ],
  },
  {
    version: 2,
    description: 'Add OCR support fields',
    up: [
      `ALTER TABLE document_cache ADD COLUMN has_images INTEGER DEFAULT 0;`,
      `ALTER TABLE document_cache ADD COLUMN ocr_text_blob BLOB;`,
      `INSERT INTO settings (key, value, type) VALUES ('ocr_enabled', 'false', 'boolean');`,
    ],
    down: [
      // SQLite 不支持 DROP COLUMN（< 3.35），需重建表
      `DELETE FROM settings WHERE key = 'ocr_enabled';`,
    ],
  },
  // V3, V4, ... 后续迁移
];
```

### 7.6.3 迁移执行流程

```
应用启动
    │
    ▼
读取 PRAGMA user_version → current_version
    │
    ▼
遍历 MIGRATIONS 数组
    │
    ▼
执行 version > current_version 的迁移
    │
    ▼
每个迁移在单个事务中执行
    │
    ▼
成功 → PRAGMA user_version = new_version
失败 → 事务回滚，报错，应用降级运行（只读模式）
```

### 7.6.4 向后兼容策略

- **新增字段**：使用 `DEFAULT` 值，旧数据自动兼容
- **删除字段**：不实际删除（SQLite 不支持），标记为 deprecated
- **字段重命名**：先新增，迁移数据，旧字段保留不使用
- **索引变更**：`CREATE INDEX IF NOT EXISTS`，幂等安全

## 7.7 并发访问策略

### 7.7.1 WAL 模式详解

```
┌─────────────────────────────────────────────────────────────────┐
│                    SQLite WAL 模式工作原理                       │
│                                                                 │
│  ┌──────────┐    读操作     ┌──────────────┐                   │
│  │ Reader 1 │─────────────►│              │                   │
│  └──────────┘              │   db.sqlite  │                   │
│  ┌──────────┐    读操作     │   (主数据)    │                   │
│  │ Reader 2 │─────────────►│              │                   │
│  └──────────┘              └──────────────┘                   │
│                                   ▲                            │
│  ┌──────────┐    写操作           │ checkpoint                 │
│  │  Writer  │─────────┐          │                            │
│  └──────────┘         │          │                            │
│                       ▼          │                            │
│              ┌──────────────┐    │                            │
│              │ db.sqlite-wal │───┘                            │
│              │ (预写日志)    │                                 │
│              └──────────────┘                                 │
│                                                                 │
│  优势：                                                        │
│  - 读操作完全不阻塞（可以并发多个 reader）                       │
│  - 写操作不阻塞读操作                                          │
│  - 写操作仍然串行（单写锁）                                     │
└─────────────────────────────────────────────────────────────────┘
```

### 7.7.2 读写分离设计

```rust
// Rust 侧使用 rusqlite 的连接池模式
// 注意: SQLite 单文件，"连接池"实际上是单连接 + 序列化访问

struct Database {
    conn: Mutex<Connection>,  // 互斥锁保证单写
}

impl Database {
    // 读操作：获取锁，执行查询，立即释放
    fn query<T, F>(&self, f: F) -> Result<T>
    where
        F: FnOnce(&Connection) -> Result<T>,
    {
        let conn = self.conn.lock().unwrap();
        f(&conn)
    }

    // 写操作：获取锁，执行事务，立即释放
    fn transaction<T, F>(&self, f: F) -> Result<T>
    where
        F: FnOnce(&Transaction) -> Result<T>,
    {
        let mut conn = self.conn.lock().unwrap();
        let tx = conn.transaction()?;
        let result = f(&tx)?;
        tx.commit()?;
        Ok(result)
    }
}
```

### 7.7.3 事务隔离级别

SQLite WAL 模式提供 **Snapshot Isolation**（快照隔离）：

- 读事务看到的是开始时刻的数据库快照
- 写事务的修改在 commit 前对其他事务不可见
- 不会出现脏读、不可重复读

### 7.7.4 死锁预防

```
规则: 所有写操作必须遵循统一的表访问顺序

访问顺序（固定）:
1. settings
2. model_registry
3. document_cache
4. embedding_cache
5. compare_history

禁止: 事务 A 先锁 document_cache 再锁 embedding_cache
      事务 B 先锁 embedding_cache 再锁 document_cache
→ 顺序固定，不可能死锁
```

## 7.8 备份与恢复

### 7.8.1 SQLite 在线备份

```rust
// 使用 SQLite Online Backup API
// 优势: 备份期间不阻塞读操作

use rusqlite::backup::Backup;

fn backup_database(source: &Connection, dest_path: &str) -> Result<()> {
    let dest = Connection::open(dest_path)?;

    // 每次复制 100 页（约 400KB），在页间释放锁
    let mut backup = Backup::new(&source, &dest)?;
    let mut step_count = 0;

    loop {
        match backup.step(100)? {
            rusqlite::backup::StepResult::Done => break,
            rusqlite::backup::StepResult::Ok => {
                step_count += 1;
                // 每 10 步 yield 一次，让出 CPU
                if step_count % 10 == 0 {
                    std::thread::yield_now();
                }
            }
            rusqlite::backup::StepResult::Busy => {
                std::thread::sleep(Duration::from_millis(10));
            }
            rusqlite::backup::StepResult::Locked => {
                return Err(Error::BackupLocked);
            }
        }
    }

    Ok(())
}
```
