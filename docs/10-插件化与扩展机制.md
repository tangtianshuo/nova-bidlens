## 10.1 插件化架构动机

### 10.1.1 从工具到平台的演进

BidLens 的核心定位不仅仅是一个文档比对工具，而是一个**AI 原生文档智能分析平台**。工具与平台的根本区别在于：

| 维度 | 工具模式 | 平台模式 |
|------|---------|---------|
| 功能边界 | 固定功能集，封闭开发 | 可扩展功能，开放生态 |
| 用户场景 | 单一场景（如标书比对） | 多场景（合同审查/法规检查/财务审计） |
| 迭代方式 | 团队内部开发，版本发布 | 核心稳定 + 插件扩展，独立迭代 |
| 商业模式 | 一次性授权 | 平台 + 插件市场 + 增值服务 |

插件化架构是实现这一演进的技术基石。通过将核心能力（文档解析、Embedding、向量匹配、Diff 计算）封装为稳定的 API，将业务场景（标书比对、合同审查、法规检查）实现为可插拔的 Workflow 插件，BidLens 可以在不修改核心代码的前提下，持续扩展新的文档智能分析场景。

### 10.1.2 多场景扩展需求

在实际业务中，文档智能分析的需求远不止标书比对。以下是已识别的主要扩展场景：

```
┌──────────────────────────────────────────────────────────────┐
│                    BidLens 平台核心                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│  │ 文档解析  │  │ Embedding│  │ 向量匹配  │  │ Diff计算  │     │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘     │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│  │ LLM 推理  │  │ OCR 识别  │  │ AST 操作  │  │ 配置管理  │     │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘     │
└────────────────────────┬─────────────────────────────────────┘
                         │ Plugin API
          ┌──────────────┼──────────────┐
          │              │              │
          ▼              ▼              ▼
  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
  │  标书比对插件  │ │  合同审查插件  │ │  法规检查插件  │
  │  (内置)      │ │  (V0.5)      │ │  (V0.5)      │
```

**典型场景说明：**

- **合同审查**：读取合同文本，与标准模板进行条款比对，标记风险条款，生成审查报告
- **法规检查**：将企业文档与最新法规条款进行语义匹配，标记不合规内容
- **财务审计**：解析财务报表，交叉验证数据一致性，标记异常数据
- **论文查重**：基于 Embedding 的语义相似度检测，而非简单的字符串匹配

## 10.2 插件类型体系

BidLens 定义了三种核心插件类型，每种类型对应不同的扩展点和职责边界。

### 10.2.1 Parser 插件（文档解析器）

**职责**：将特定格式的文档解析为标准 JSON AST。

**扩展点**：当需要支持新的文档格式（如 Excel、PPT、OFD、WPS）时，开发者只需实现 Parser 插件，无需修改核心代码。

**内置 Parser**：
- `@bidlens/parser-docx` — Word (.docx) 解析器（基于 mammoth/docx4js）
- `@bidlens/parser-pdf` — PDF 解析器（基于 pdf.js）

**典型扩展**：
- `@bidlens/parser-xlsx` — Excel 解析器，将工作表转为表格型 AST
- `@bidlens/parser-pptx` — PPT 解析器，将幻灯片转为页面型 AST
- `@bidlens/parser-ofd` — OFD 版式文档解析器

### 10.2.2 OCR 插件（文字识别服务）

**职责**：提供图片/PDF 扫描件的文字识别能力。

**扩展点**：当需要接入不同的 OCR 服务（如阿里云 OCR、百度云 OCR、腾讯云 OCR）时，开发者实现 OCR 插件即可。

**内置 OCR**：
- `@bidlens/ocr-paddle` — PaddleOCR 本地识别（默认，离线可用）

**典型扩展**：
- `@bidlens/ocr-aliyun` — 阿里云 OCR（高精度，适合复杂版式）
- `@bidlens/ocr-baidu` — 百度云 OCR（支持手写体识别）
- `@bidlens/ocr-tencent` — 腾讯云 OCR（票据识别）

### 10.2.3 Workflow 插件（业务流程）

**职责**：定义完整的业务分析流程，包含 UI 面板组件。

**扩展点**：Workflow 插件是最灵活的插件类型，它可以组合使用核心 API（Embedding、Diff、LLM、OCR）实现任意复杂的文档分析流程。

**内置 Workflow**：
- `@bidlens/workflow-bidding` — 标书比对工作流（核心场景）

**典型扩展**：
- `@bidlens/workflow-contract` — 合同审查工作流
- `@bidlens/workflow-compliance` — 法规合规检查工作流

```
│                  Plugin 类型体系                   │
├─────────────────────────────────────────────────┤

│  │ WorkflowPlugin├──────┐                          │
│  │  (业务流程)   │      │                          │
│  └─────────────┘      │                          │
│                        ▼                          │
│  ┌─────────────────────────────────────┐         │
│  │          Core API                    │         │
```

## 10.3 插件接口定义

### 10.3.1 PluginManifest（插件清单）

每个插件必须包含一个 `manifest.json` 文件，描述插件的元信息、依赖关系和配置 schema。

```typescript
/**
 * 插件清单接口定义
 * 每个插件包的根目录必须包含 manifest.json
 */
interface PluginManifest {
  /** 插件唯一标识符，采用 @scope/name 格式 */
  id: string;

  /** 插件显示名称 */
  name: string;

  /** 语义化版本号 */
  version: string;

  /** 插件类型 */
  type: 'parser' | 'ocr' | 'workflow';

  /** 插件作者信息 */
  author: {
    name: string;
    email?: string;
    url?: string;
  };

  /** 插件入口文件路径（相对于插件根目录） */
  entry: string;

  /** 最低支持的 BidLens 版本 */
  min_app_version: string;

  /** 依赖的其他插件列表 */
  dependencies?: Array<{
    id: string;
    version_range: string; // semver range, e.g., ">=1.0.0"
  }>;

  /** 插件配置的 JSON Schema（用于自动生成配置 UI） */
  config_schema?: Record<string, unknown>;

  /** 插件描述 */
  description?: string;

  /** 插件图标（base64 或相对路径） */
  icon?: string;

  /** 插件标签（用于市场搜索） */
  tags?: string[];

  /** 许可证 */
  license?: string;

  /** 插件仓库地址 */
  repository?: string;
}
```

### 10.3.2 ParserPlugin（文档解析器插件）

```typescript
/**
 * 文档解析器插件接口
 * 实现此接口可扩展新的文档格式支持
 */
interface ParserPlugin {
  /** 插件清单 */
  readonly manifest: PluginManifest;

  /**
   * 列出本插件支持的文件扩展名
   * @returns 支持的扩展名数组，如 [".xlsx", ".xls"]
   */
  getSupportedExtensions(): string[];

  /**
   * 检查是否能解析指定文件
   * @param filePath 文件路径
   * @param fileBuffer 文件内容（前 4KB 预读）
   * @returns 是否可以解析
   */
  canParse(filePath: string, fileBuffer: ArrayBuffer): boolean;

  /**
   * 解析文档为 JSON AST
   * @param filePath 文件路径
   * @param options 解析选项
   * @param context 插件上下文（提供 Core API 访问）
   * @returns 解析结果，包含 AST 和元数据
   */
  parse(
    filePath: string,
    options: ParseOptions,
    context: PluginContext
  ): Promise<ParseResult>;

  /**
   * 获取解析进度（可选）
   * @param taskId 任务 ID
   * @returns 解析进度 0-100
   */
  getProgress?(taskId: string): number;
}

interface ParseOptions {
  /** 是否提取批注 */
  extractComments?: boolean;
  /** 是否提取修订 */
  extractRevisions?: boolean;
  /** 是否提取图片 */
  extractImages?: boolean;
  /** 自定义选项（由插件自行定义） */
  custom?: Record<string, unknown>;
}

interface ParseResult {
  /** JSON AST 根节点 */
  ast: DocumentAST;
  /** 文档元数据 */
  metadata: DocumentMetadata;
  /** 解析警告（非致命错误） */
  warnings?: string[];
  /** 资源列表（图片、附件等） */
  resources?: ResourceDescriptor[];
}
```

### 10.3.3 OcrPlugin（文字识别插件）

```typescript
/**
 * OCR 文字识别插件接口
 * 实现此接口可接入不同的 OCR 服务
 */
interface OcrPlugin {
  /** 插件清单 */
  readonly manifest: PluginManifest;

  /**
   * 识别图片中的文字
   * @param image 图片数据（Buffer 或 URL）
   * @param options 识别选项
   * @param context 插件上下文
   * @returns 识别结果
   */
  recognize(
    image: ImageInput,
    options: OcrOptions,
    context: PluginContext
  ): Promise<OcrResult>;

  /**
   * 批量识别（可选，用于性能优化）
   * @param images 图片列表
   * @param options 识别选项
   * @param context 插件上下文
   * @returns 识别结果列表
   */
  recognizeBatch?(
    images: ImageInput[],
    options: OcrOptions,
    context: PluginContext
  ): Promise<OcrResult[]>;
}

type ImageInput =
  | { type: 'buffer'; data: Buffer; mime: string }
  | { type: 'file'; path: string }
  | { type: 'url'; url: string };

interface OcrOptions {
  /** 识别语言 */
  language?: string;
  /** 是否返回坐标信息 */
  returnCoordinates?: boolean;
  /** 是否返回置信度 */
  returnConfidence?: boolean;
  /** 自定义选项 */
  custom?: Record<string, unknown>;
}

interface OcrResult {
  /** 识别出的文本块列表 */
  blocks: OcrTextBlock[];
  /** 整体置信度 */
  confidence: number;
  /** 识别耗时（毫秒） */
  duration_ms: number;
}

interface OcrTextBlock {
  /** 文本内容 */
  text: string;
  /** 置信度 0-1 */
  confidence: number;
  /** 文本坐标（可选） */
  coordinates?: {
    x: number;
    y: number;
    width: number;
    height: number;
  };
}
```

### 10.3.4 WorkflowPlugin（业务流程插件）

```typescript
/**
 * 业务流程插件接口
 * 实现此接口可定义新的文档分析业务流程
 */
interface WorkflowPlugin {
  /** 插件清单 */
  readonly manifest: PluginManifest;

  /**
   * 执行分析流程
   * @param input 输入参数（文件路径、配置等）
   * @param context 插件上下文
   * @returns 分析结果
   */
  analyze(
    input: WorkflowInput,
    context: PluginContext
  ): Promise<WorkflowResult>;

  /**
   * 获取面板组件（React 组件）
   * 用于在主界面中展示插件的操作面板和结果
   * @returns React 组件
   */
  getPanelComponent(): React.ComponentType<WorkflowPanelProps>;

  /**
   * 获取结果展示组件（可选）
   * @returns React 组件
   */
  getResultComponent?(): React.ComponentType<WorkflowResultProps>;

  /**
   * 获取配置面板组件（可选）
   * @returns React 组件
   */
  getConfigComponent?(): React.ComponentType<WorkflowConfigProps>;
}

interface WorkflowInput {
  /** 输入文件列表 */
  files: string[];
  /** 用户配置 */
  config: Record<string, unknown>;
  /** 任务 ID（用于进度上报） */
  taskId: string;
}

interface WorkflowResult {
  /** 结果数据（由插件自行定义结构） */
  data: unknown;
  /** 结果摘要 */
  summary: string;
  /** 是否有风险/异常 */
  hasRisk: boolean;
  /** 风险等级 */
  riskLevel?: 'low' | 'medium' | 'high' | 'critical';
  /** 生成的报告（可选） */
  report?: {
    title: string;
    content: string;
    format: 'markdown' | 'html' | 'pdf';
  };
}

interface WorkflowPanelProps {
  /** 任务状态 */
  status: 'idle' | 'running' | 'completed' | 'error';
  /** 进度百分比 */
  progress: number;
  /** 结果数据 */
  result: WorkflowResult | null;
  /** 错误信息 */
  error: string | null;
  /** 触发分析 */
  onStart: (input: WorkflowInput) => void;
  /** 取消分析 */
  onCancel: () => void;
}
```

## 10.4 插件生命周期

### 10.4.1 生命周期阶段

插件从安装到卸载经历以下六个阶段：

```
┌──────────┐    ┌──────────────┐    ┌──────────────┐
│  下载     │───▶│  onInstall   │───▶│  onActivate  │
│  (.zip)  │    │  (安装注册)   │    │  (加载初始化) │
└──────────┘    └──────────────┘    └──────┬───────┘
                                           │
                                           ▼
                                    ┌──────────────┐
                                    │   正常使用     │
                                    │  (方法调用)    │
                                    └──────┬───────┘
                                           │
                   ┌──────────────┐    ┌────┴───────┐
                   │ onUninstall  │◀───│onDeactivate│
                   │ (卸载删除)    │    │ (卸载清理)  │
                   └──────────────┘    └────────────┘
```

**各阶段详细说明：**

| 阶段 | 触发条件 | 执行操作 | 状态存储 |
|------|---------|---------|---------|
| 下载 | 用户从市场安装或手动导入 | 下载 .zip 包到临时目录 | - |
| onInstall | 解压 .zip 包后 | 校验 manifest、检查依赖、复制到插件目录、注册到数据库 | `installed` |
| onActivate | 插件启用时 | 读取入口文件、执行初始化代码、注册到运行时 | `active` |
| 正常使用 | 插件已激活 | 通过 Plugin API 调用插件方法 | `active` |
| onDeactivate | 插件禁用时 | 释放内存资源、注销运行时注册 | `inactive` |
| onUninstall | 用户卸载插件时 | 删除插件目录、清除数据库记录、清除配置 | - |

### 10.4.2 生命周期状态机

```
                        ┌─────────────────────┐
                        │                     │
                        ▼                     │
┌──────┐  install  ┌──────────┐  activate  ┌──────────┐
│ 无    │─────────▶│ installed│──────────▶│  active   │
│ 状态  │          │ (已安装)  │           │  (已激活)  │
└──────┘          └──────────┘           └─────┬────┘

                        │                       │ deactivate
                        │                       ▼
                        │                 ┌──────────┐
```

## 10.5 插件沙箱与安全

### 10.5.1 安全威胁模型

插件运行在用户本地环境中，潜在安全威胁包括：

| 威胁类型 | 风险描述 | 严重等级 |
|---------|---------|---------|
| 文件系统越权 | 插件读取/修改/删除用户敏感文件 | 高 |
| 网络滥用 | 插件向外部服务器发送用户数据 | 高 |
| 资源耗尽 | 插件占用过多 CPU/内存导致应用卡死 | 中 |
| 代码注入 | 插件通过 eval 等方式执行恶意代码 | 高 |
| 数据泄露 | 插件将文档内容发送到外部 | 高 |
| 权限提升 | 插件通过子进程获取更高权限 | 严重 |

### 10.5.2 第一阶段：Node.js vm 模块沙箱（V0.5 早期过渡）

> **安全警告**：Node.js `vm` 模块**不是安全沙箱**（Node.js 官方文档明确声明）。
> 攻击者可能通过原型链污染（`this.constructor.constructor('return process')()`）绕过上下文限制。
> 因此，Phase 1 的安全策略必须遵循以下原则：
>
> 1. **仅允许 Parser 插件**使用 vm 沙箱（Parser 仅做格式转换，不访问核心 API）
> 2. **禁止 Workflow / OCR 插件**在 Phase 1 中使用 vm 沙箱运行
> 3. **所有外部插件**（非内置）必须在 V0.5 后期迁移到 Worker threads（Phase 2）
> 4. 内置插件（`@bidlens/parser-docx`）由于代码受控，可以放宽限制

在 V0.5 版本中，使用 Node.js 内置的 `vm` 模块创建受限的执行上下文（仅限内置 Parser 插件）：

```typescript
import vm from 'node:vm';
import { createContext } from './sandbox-context';

/**
 * 创建插件沙箱执行环境
 * @param plugin 插件实例
 * @param permissions 授权的权限列表
 * @returns 沙箱化的插件导出对象
 */
function createSandbox(
  plugin: PluginDescriptor,
  permissions: Permission[]
): PluginExports {
  // 1. 构建受限的全局对象
  const sandboxContext = createContext({
    // 允许使用的全局对象
    console: createScopedConsole(plugin.id),
    setTimeout: safeSetTimeout,
    setInterval: safeSetInterval,
    clearTimeout,
    clearInterval,
    Buffer,
    TextEncoder,
    TextDecoder,
    URL,
    URLSearchParams,

    // 禁止使用的全局对象
    // process, require, import, eval, Function 均不可用
  });

  // 2. 创建 V8 上下文
  const context = vm.createContext(sandboxContext);

  // 3. 加载并执行插件代码
  const pluginCode = fs.readFileSync(plugin.entry, 'utf-8');
  const wrappedCode = `
    (function(exports, require, module, __filename, __dirname) {
      ${pluginCode}
    });
  `;

  const script = new vm.Script(wrappedCode, {
    filename: plugin.entry,
    timeout: 5000, // 编译超时 5 秒
  });

  const wrapper = script.runInContext(context, {
    timeout: 3000, // 执行超时 3 秒
  });

  // 4. 注入受限的 API
  const pluginExports = {};
  const pluginModule = { exports: pluginExports };

  wrapper(
    pluginExports,
    createRestrictedRequire(plugin, permissions),
    pluginModule,
    plugin.entry,
    path.dirname(plugin.entry)
  );

  return pluginModule.exports;
}
```

### 10.5.3 第二阶段：Worker 线程隔离（V0.5 后期，所有外部插件必须使用）

在 V0.5 的后期迭代中，将所有非内置插件迁移到 Worker 线程隔离。Worker threads 提供进程级别的内存隔离（`resourceLimits`），插件崩溃不会影响主进程。**这是外部插件的唯一安全执行方式。**

```typescript
import { Worker } from 'node:worker_threads';

/**
 * 在 Worker 线程中运行插件
 * 提供进程级别的隔离，插件崩溃不会影响主进程
 */
class PluginWorker {
  private worker: Worker;
  private pendingTasks = new Map<string, {
    resolve: Function;
    reject: Function;
    timeout: NodeJS.Timeout;
  }>();

  constructor(plugin: PluginDescriptor, permissions: Permission[]) {
    this.worker = new Worker(
      path.join(__dirname, 'plugin-worker-bootstrap.js'),
      {
        workerData: {
          pluginEntry: plugin.entry,
          permissions,
          resourceLimits: {
            maxOldGenerationSizeMb: 256,  // 最大堆内存 256MB
            maxYoungGenerationSizeMb: 64,
            codeRangeSizeMb: 32,
            stackSizeMb: 8,
          },
        },
        // 不继承主进程的环境变量（防止信息泄露）
        env: {},
      }
    );

    this.worker.on('message', this.handleMessage.bind(this));
    this.worker.on('error', this.handleError.bind(this));
    this.worker.on('exit', this.handleExit.bind(this));
  }

  /** 调用插件方法 */
  async call(method: string, args: unknown[]): Promise<unknown> {
    const taskId = crypto.randomUUID();

    return new Promise((resolve, reject) => {
      const timeout = setTimeout(() => {
        this.pendingTasks.delete(taskId);
        reject(new Error(`Plugin call timeout: ${method}`));
      }, 30000); // 30 秒超时

      this.pendingTasks.set(taskId, { resolve, reject, timeout });
      this.worker.postMessage({ type: 'call', taskId, method, args });
    });
  }

  /** 终止 Worker */
  terminate(): void {
    this.worker.terminate();
  }
}
```

### 10.5.4 权限控制体系

```typescript
/**
 * 插件权限声明
 * 插件在 manifest.json 中声明所需权限，用户在安装时确认
 */
type Permission =
  | { type: 'file:read'; paths: string[] }      // 读取指定路径
  | { type: 'file:write'; paths: string[] }      // 写入指定路径
  | { type: 'network'; domains: string[] }       // 访问指定域名
  | { type: 'core:embed' }                        // 使用 Embedding API
  | { type: 'core:llm' }                          // 使用 LLM API
  | { type: 'core:ocr' }                          // 使用 OCR API
  | { type: 'core:diff' }                         // 使用 Diff API
  | { type: 'core:config:read' }                  // 读取配置
  | { type: 'core:config:write' };                // 修改配置

// Phase 1 (vm 沙箱) 权限限制：
// - 内置 Parser 插件: 允许 file:read, core:embed
// - 外部 Parser 插件 (vm 沙箱): 仅允许 file:read（禁止 core:* 系列权限）
// - 所有 Workflow/OCR 插件: 必须使用 Worker threads（Phase 2），不得使用 vm 沙箱

/**
 * 权限检查中间件
 * 在每次 API 调用前检查插件是否具有对应权限
 */
function checkPermission(
  pluginId: string,
  requiredPermission: Permission
): void {
  const granted = getGrantedPermissions(pluginId);
  const hasPermission = granted.some(p =>
    matchPermission(p, requiredPermission)
  );

  if (!hasPermission) {
    throw new PermissionDeniedError(
      `Plugin ${pluginId} does not have permission: ${JSON.stringify(requiredPermission)}`
    );
  }
}
```

### 10.5.5 资源限制

| 资源类型 | 限制值 | 超限处理 |
|---------|-------|---------|
| CPU 时间（单次调用） | 30 秒 | 超时终止，返回错误 |
| 堆内存 | 256 MB | OOM 终止，重启 Worker |
| 磁盘空间（插件目录） | 500 MB | 禁止写入 |
| 网络请求速率 | 100 次/分钟 | 限流，返回 429 |
| 文件句柄数 | 64 | 禁止打开新文件 |

```typescript
class PluginManager {

    pluginId: string,

  async runWorkflow(
    pluginId: string,
    input: WorkflowInput

    const plugin = this.getPlugin(pluginId, 'workflow') as WorkflowPlugin;
    const context = this.createContext(pluginId);

  }
```

### 10.8.2 插件发布流程

```
开发者注册 → 创建插件项目 → 开发调试 → 本地打包
    → 上传 .zip 包 → 自动化安全审核 → 人工审核（首次）
    → 审核通过 → 发布到市场 → 用户可见
```

**安全审核内容：**

1. **静态代码扫描**：检查是否包含 `eval`、`new Function`、`child_process` 等危险调用
2. **权限声明校验**：检查插件声明的权限是否与其功能匹配
3. **依赖审计**：检查第三方依赖是否存在已知漏洞
4. **行为沙箱测试**：在隔离环境中运行插件，监控其文件/网络行为
