## 1.1 产品定位与目标

### 1.1.1 产品定位

**Nova BidLens** 是一款 AI 原生的文档智能比对桌面平台，专注于招投标文档（标书）的智能审核与差异分析。

**一句话定位：**

> 以 JSON AST 为核心数据模型，以 Embedding 语义匹配为主要比对手段，以 Rust 为计算引擎，以 Electron + React 为交互层，支持 1000 页以上大型文档的全保真（L5）智能比对，帮助标书审核人员快速发现两份文档之间的内容差异、格式变化和语义偏移。

**产品核心价值主张：**

| 价值维度 | 传统工具 | BidLens |
|---------|---------|---------|
| 比对能力 | 仅文本逐行对比 | 语义级比对，结构不同也能比 |
| 文档保真 | 丢失格式/批注/修订 | L5 全保真，完整保留 |
| 大文档支持 | 50 页以上卡顿 | 1000 页以上流畅 |
| AI 能力 | 无 | 本地 AI + 外部 AI 双模式 |
| 扩展性 | 固定功能 | 插件化架构，可扩展场景 |

### 1.1.3 核心使用场景（Use Cases）

**场景 1：标书版本比对（核心场景）**

```
触发条件：用户拖入两份 Word 文档（标书新旧版本）
前置条件：已完成文档解析和 Embedding
主要流程：
  1. 用户拖入文档 A（旧版）和文档 B（新版）
  2. 系统自动解析为 Document AST
  3. 系统执行 Chunk → Embedding → 向量匹配 → Diff
  4. 系统展示差异结果（三种展示方式可选）
  5. 用户逐条审核差异，标记重要/忽略
  6. 用户导出审核报告
异常流程：
  - 文档格式不支持 → 提示用户
  - AI 模型未下载 → 引导下载或切换外部模型
  - 比对超时 → 可取消，降级到文本 Diff
```

**场景 2：结构重组文档比对**

```
触发条件：两份标书章节顺序完全不同（如重新排版）
核心挑战：传统逐行比对完全失效
BidLens 方案：
  1. 基于 Embedding 语义匹配，不依赖位置
  2. 识别"拆分"（一段变两段）、"合并"（两段变一段）
  3. 标记"顺序调整"（第 3 章移到第 5 章）
  4. 以匹配对而非位置为主线展示差异
```

**场景 3：扫描件与电子文档比对（V0.4）**

```
触发条件：一份为扫描件 PDF，一份为电子 Word
BidLens 方案：
  1. 扫描件通过 PaddleOCR 识别文字
  2. 识别结果构建 Document AST
  3. 后续流程与标准比对一致
  4. OCR 不确定的文字标记置信度
```

```
│  领域层（Domain Layer）                                   │
│  Diff Engine / Embedding / Vector / Chunk                │
│  职责：核心业务逻辑、算法实现、领域模型                     │
├─────────────────────────────────────────────────────────┤
│  基础设施层（Infrastructure Layer）                       │
│  SQLite / ONNX Runtime / llama.cpp / FileSystem          │
│  职责：技术基础设施、外部系统集成、持久化                   │
└─────────────────────────────────────────────────────────┘
```

### 1.3.1 表现层（Presentation Layer）

**所在进程**：Electron Renderer Process

**包含内容**：
- React 页面组件（Home / Compare / Settings / PluginStore）
- UI 基础组件（基于 shadcn/ui）
- Zustand 状态管理（UI 状态、比对配置、选中文档）
- TanStack Query hooks（RPC 调用、模型状态、历史记录）
- 虚拟滚动组件（大文档渲染）

**依赖规则**：仅依赖应用层暴露的 Hook 和 Service，不直接调用底层。

### 1.3.2 应用层（Application Layer）

**所在进程**：Electron Main Process + Preload

**包含内容**：
- IPC Handler（处理渲染进程的请求）
- Parser 调度器（选择合适的 Parser 适配器）
- RPC Client（与 Rust 子进程通信）
- 文件操作 Service（读取/写入文件）
- 自动更新 Service

**依赖规则**：依赖领域层的接口定义，通过依赖注入调用基础设施层。

### 1.3.3 领域层（Domain Layer）

**所在进程**：Rust Core Engine 子进程

**包含内容**：
- Document AST 类型定义和操作
- Diff 算法引擎（文本/段落/表格/格式 Diff）
- Embedding 推理引擎
- 向量检索引擎
- 文档分块引擎

**依赖规则**：不依赖任何外部特定实现，通过 Trait 抽象依赖基础设施层。

### 1.3.4 基础设施层（Infrastructure Layer）

**所在进程**：各进程按需

**包含内容**：
- SQLite 数据库访问
- ONNX Runtime 推理运行时
- llama.cpp 推理运行时
- 文件系统操作
- 网络请求（外部 API）
- 日志系统（tracing）

**依赖规则**：实现领域层定义的 Trait，提供具体技术实现。

### 1.3.5 层间依赖规则

```
表现层 ──→ 应用层 ──→ 领域层 ──→ 基础设施层
   │           │          │           │
   │           │          │           │
   ▼           ▼          ▼           ▼
 不反向依赖  不反向依赖  通过 Trait    不反向依赖
                          依赖抽象
```

**严格规则**：
- 上层可以依赖下层，下层不能依赖上层
- 领域层通过 Trait 接口依赖基础设施层（依赖倒置）
- 同层之间尽量不依赖，必要时通过接口通信
- 跨进程通信仅在应用层进行（IPC + JSON-RPC）

## 1.4 技术选型决策矩阵

### 1.4.1 前端技术选型

| 决策项 | 选择 | 备选方案 | 选择理由 | 风险与缓解 |
|--------|------|---------|---------|-----------|
| **桌面框架** | Electron | Tauri / Flutter Desktop | 生态最成熟，自动更新方案完善，企业级实践丰富（VS Code、Slack、Notion 均基于 Electron） | 包体积大（~150MB）→ 按需加载、模型不内置 |
| **UI 框架** | React 19 + TypeScript | Vue 3 / Svelte | 组件生态完善，TypeScript 支持优秀，适合复杂交互型应用 | 学习曲线 → 团队已有 React 经验 |
| **构建工具** | Vite | Webpack / Turbopack | 开发体验优秀（HMR < 100ms），构建速度快，配置简洁 | 生态略逊 Webpack → 足够成熟 |
| **CSS 方案** | TailwindCSS + shadcn/ui | styled-components / Ant Design | 快速构建现代桌面 UI，shadcn/ui 可完全自定义源码 | 类名冗长 → IDE 插件辅助 |
| **状态管理** | Zustand | Redux Toolkit / Jotai | 轻量（1KB）、无 Boilerplate、支持 React 19 | 中大型应用需规范 → 配合 devtools |
| **异步状态** | TanStack Query | SWR / RTK Query | 缓存、重试、乐观更新、后台刷新一应俱全 | 学习成本 → 仅用于 RPC 调用 |
| **虚拟列表** | tanstack-virtual | react-window / react-virtuoso | 动态高度支持好，Headless（无 UI 约束） | 需自行实现渲染逻辑 → 封装通用组件 |

### 1.4.2 后端引擎技术选型

| 决策项 | 选择 | 备选方案 | 选择理由 | 风险与缓解 |
|--------|------|---------|---------|-----------|
| **引擎语言** | Rust | C++ / Go | 内存安全、性能优异、ONNX/llama.cpp 原生支持 | 学习曲线陡峭 → 团队提前培训 |
| **异步运行时** | Tokio | async-std / smol | Rust 异步生态事实标准，IO 调度成熟 | 复杂度高 → 仅用于 IO 任务 |
| **CPU 并行** | Rayon | std::thread / crossbeam | 数据并行简单优雅（par_iter），适合文档处理 | 线程池管理 → 使用默认配置 |
| **序列化** | serde + serde_json | simd-json / sonic-rs | 生态最成熟，与 JSON AST 天然契合 | 大 JSON 慢 → 可切换 simd-json |
| **Embedding 运行时** | ONNX Runtime | tract / candle | 跨平台、模型兼容性好、Rust binding 成熟 | 模型格式转换 → ONNX 是通用格式 |
| **LLM 运行时** | llama.cpp | candle / vLLM | 本地 LLM 推理事实标准，GGUF 格式广泛 | Rust binding 不完善 → C FFI 封装 |
| **向量索引** | SQLite 暴力搜索 → HNSW | FAISS / Milvus | 第一阶段简单够用，后续 HNSW 无外部依赖 | 大规模慢 → 1000 页 chunk 数 < 10000 |

### 1.4.3 解析层技术选型

| 决策项 | 选择 | 备选方案 | 选择理由 | 风险与缓解 |
|--------|------|---------|---------|-----------|
| **运行环境** | Node.js (Electron Main) | Rust / Python | npm 生态有成熟 Word 解析库，开发效率高 | 非性能瓶颈（1-3秒 vs 分钟级 AI） |
| **Word 解析** | mammoth + docx4js | officegen / docx | mammoth 擅长 HTML 转换，docx4js 擅长结构提取，互补 | L5 支持需自研补充 → 预研验证 |
| **PDF 解析** | pdf.js（V0.4） | pdf-lib / pdf2json | Mozilla 出品，社区活跃，支持文本提取和坐标 | 扫描件需配合 OCR |
| **通信方式** | stdio JSON-RPC | napi-rs / gRPC | 进程隔离、崩溃不影响主进程、构建解耦 | 性能开销可忽略（300-500ms/1000页） |

### 1.4.4 基础设施技术选型

| 决策项 | 选择 | 备选方案 | 选择理由 | 风险与缓解 |
|--------|------|---------|---------|-----------|
| **数据库** | SQLite (rusqlite) | DuckDB / RocksDB | 嵌入式、零配置、跨平台、适合缓存和配置 | 并发写入弱 → WAL 模式 |
| **自动更新** | electron-updater | 自研 / Squirrel | Electron 生态标准方案，支持增量更新 | 需要 CDN → 内网部署场景可配置 |
| **打包** | electron-builder | electron-forge | 多平台打包成熟，签名支持完善 | 配置复杂 → 封装脚本 |
| **日志** | tracing (Rust) + electron-log (Node) | log4rs / bunyan | 结构化日志、span 追踪、性能好 | 学习成本 → 仅核心模块使用 |

## 1.6 跨平台兼容策略

### 1.6.1 平台差异对照表

| 功能点 | Windows | macOS | Linux | 处理策略 |
|--------|---------|-------|-------|---------|
| 文件路径分隔符 | `\` | `/` | `/` | 统一使用 Node.js `path` 模块 |
| 文件名大小写 | 不敏感 | 默认不敏感（可配置） | 敏感 | 统一使用小写文件名 |
| 最大路径长度 | 260 字符（传统） | 无限制 | 4096 字符 | 使用短路径 + hash |
| 文件锁定 | 强制锁定 | Advisory 锁 | Advisory 锁 | 读取时复制到临时目录 |
| 剪贴板 | Win32 API | AppKit | X11/Wayland | Electron `clipboard` API 统一 |
| 通知 | Windows Notification Center | NSUserNotification | libnotify | Electron `Notification` API |
| Rust 二进制 | .exe | Mach-O | ELF | 分平台编译，electron-builder 打包 |
| ONNX Runtime | DirectML / CPU | CoreML / CPU | CPU | 按平台选择加速后端 |
| GPU 加速 | CUDA / DirectML | Metal | CUDA / Vulkan | V1.0 仅 CPU，V1.1 引入 GPU 加速 |

### 1.6.2 各平台特殊处理

**Windows 特殊处理：**
- 长路径支持：启用 `\\?\` 前缀或 manifest 声明 `longPathAware`
- UAC 兼容：安装目录使用用户级目录，无需管理员权限
- Windows Defender：首次启动 Rust 子进程可能触发安全扫描 → 添加签名

**macOS 特殊处理：**
- Gatekeeper：应用需 Apple 公证（Notarization）
- App Nap：长时间后台任务需调用 `powerSaveBlocker` 阻止系统休眠
- Apple Silicon：需同时支持 x64 和 arm64 构建

**Linux 特殊处理：**
- 发行版兼容：使用 AppImage 格式打包，兼容主流发行版
- 依赖库：静态链接 glibc 或使用 musl 构建
- Wayland/X11：Electron 默认 X11，Wayland 下可能需要 `--ozone-platform=wayland`

## 1.7 安全设计

### 1.7.1 Electron 沙箱安全

```
┌─────────────────────────────────────────────────────┐
│  Electron 安全模型                                    │
│                                                     │
│  Main Process（Node.js，完整权限）                    │
│    │                                                │
│    │  IPC（受控通道）                                 │
│    ▼                                                │
│  Preload Script（有限 API 暴露）                     │
│    │                                                │
│    │  contextBridge（最小接口）                       │
│    ▼                                                │
│  Renderer Process（Web 沙箱，无 Node.js 访问）       │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**安全策略清单：**

| 策略 | 实施方式 | 目的 |
|------|---------|------|
| `nodeIntegration: false` | BrowserWindow 配置 | 渲染进程无法直接访问 Node.js API |
| `contextIsolation: true` | BrowserWindow 配置 | Preload 脚本与渲染进程上下文隔离 |
| `sandbox: true` | BrowserWindow 配置 | 渲染进程运行在 OS 级沙箱中 |
| CSP 策略 | Content-Security-Policy header | 防止 XSS、限制资源加载来源 |
| 最小 API 暴露 | Preload 仅暴露必要接口 | 减少攻击面 |
| IPC 输入校验 | Main 进程校验所有 IPC 参数 | 防止注入攻击 |

### 1.7.2 IPC 安全

**Preload 暴露的 API 最小集：**

```typescript
// preload.ts — 仅暴露必要接口
contextBridge.exposeInMainWorld('bidlens', {
  // 文件操作
  openFile: () => ipcRenderer.invoke('dialog:openFile'),
  saveFile: (data) => ipcRenderer.invoke('dialog:saveFile', data),

  // 比对任务
  startCompare: (params) => ipcRenderer.invoke('compare:start', params),
  cancelCompare: (taskId) => ipcRenderer.invoke('compare:cancel', taskId),
  onCompareProgress: (callback) => {
    ipcRenderer.on('compare:progress', (_, data) => callback(data));
  },

  // 模型管理
  getModelStatus: () => ipcRenderer.invoke('model:status'),
  downloadModel: (modelId) => ipcRenderer.invoke('model:download', modelId),

  // 配置
  getSettings: () => ipcRenderer.invoke('settings:get'),
  setSettings: (settings) => ipcRenderer.invoke('settings:set', settings),
});
```

**IPC 参数校验规则：**
- 所有 `invoke` 参数在 Main 进程进行 Schema 校验（使用 zod）
- 文件路径参数进行路径遍历检查（防止 `../../etc/passwd`）
- 数组/对象参数限制大小（防止 DoS）

### 1.7.3 AI 模型安全

| 威胁 | 风险 | 缓解措施 |
|------|------|---------|
| 模型文件被篡改 | 恶意模型执行任意代码 | SHA256 校验 + 签名验证 |
| 模型下载被中间人攻击 | 下载恶意文件 | HTTPS 下载 + 证书校验 |
| 外部 API 泄露文档内容 | 隐私泄露 | 明确提示用户；本地模式优先 |
| LLM 输出注入 | 恶意 prompt | 输出过滤 + 沙箱执行 |
| OCR 误识别 | 错误信息 | 置信度标注 + 人工确认 |

### 1.7.4 数据安全

**静态数据安全：**

| 数据 | 存储位置 | 保护措施 |
|------|---------|---------|
| 文档缓存 | SQLite (用户数据目录) | 文件系统权限（仅当前用户可读写） |
| Embedding 缓存 | SQLite (用户数据目录) | 同上 |
| 比对历史 | SQLite (用户数据目录) | 同上；用户可清除 |
| API Key | SQLite settings 表 | AES-256 加密存储（使用系统 Keychain 保护密钥） |
| 本地模型文件 | 用户数据目录/models | 文件系统权限 |

**传输数据安全：**

| 通信路径 | 保护措施 |
|---------|---------|
| Node ↔ Rust (stdio) | 进程内通信，无需额外加密 |
| Node ↔ 外部 API (HTTPS) | TLS 1.2+，证书校验 |
| Electron ↔ 更新服务器 (HTTPS) | TLS 1.2+，SHA256 校验更新包 |

**隐私设计原则：**
- 默认本地处理，文档不出设备
- 使用外部 API 时明确告知用户并获得确认
- 不收集用户文档内容（仅收集匿名使用统计，需用户同意）
- 用户可随时清除所有缓存和历史数据
