## 2. 当前仓库事实

- 当前渲染层入口仍是“打开示例比对”。
- `compare:start` 返回硬编码演示结果，没有串联真实解析器与 Rust 引擎。
- 当前报告导出 IPC 返回空路径，真实报告生成器尚未接入。
- 仓库已有 DOCX、文本型 PDF 解析器及相关测试，但未接入客户端闭环。
- Rust 引擎当前实际生效的公开配置只有 `similarity_threshold`。
- TypeScript 中的 fast/standard/precise、Embedding provider/model、topK 尚无对应引擎实现。
- `DiffItem.diffDetail` 已预留细粒度文本差异结构，但当前引擎不会填充。
- 项目已有 SQLite 数据库设计文档，但尚未落地；现有设计缺少独立审核记录表。
- `ui-style-preview.html` 定义了 Swiss Minimalism 风格、明暗主题和差异语义色，但只是原生 HTML/CSS，并未使用正式组件库。
- Desktop 文档规定 React 19、Tailwind CSS 4、shadcn/ui；当前依赖尚未完整安装。

## 3. 已确认决策

### D01 V0.2.2 顶层范围

状态：已确认

V0.2.2 聚焦真实可操作的单次文档审核闭环：

`选择两个真实文档 -> 校验 -> 配置 -> 查看/取消进度 -> 获取真实结果 -> 筛选与逐项复核 -> 添加审核记录 -> 导出报告 -> 开始新任务`

必要的主进程、IPC、任务编排和错误处理属于范围。模型商店、插件市场、自动更新、账号和云服务不属于本版本。

### D02 输入支持矩阵

状态：已确认

- DOCX 与 DOCX：按实际能力比对内容、表格、格式、批注和修订。
- 文本型 PDF 与文本型 PDF：比对内容和页码。
- DOCX 与文本型 PDF：仅比较双方都具备的内容能力，并明确能力降级。
- 不支持旧版 DOC、加密文件和扫描型 PDF。
- 单文件上限暂定 100 MB。
- 支持声明必须通过真实文件端到端测试验证。

### D03 信息架构

状态：已确认

- 单窗口、任务驱动的三阶段结构：文件准备、处理中、复核结果。
- 启动后直接进入新建比对，不建设营销式首页。
- 比对设置位于准备阶段，复核筛选位于结果工作台。
- 不建设无真实能力支撑的插件、模型等空页面。
- 少量应用设置使用对话框或抽屉。
- 未完成任务或未保存输入离开时需要确认。

### D04 文档角色命名

状态：暂定

- 暂用“基准文档 / 待审文档”，替代“文档 A / 文档 B”。
- 新增和删除均以基准文档为参照解释。
- 支持交换两份文档。
- 最终文案仍可在后续讨论中调整。

### D05 比对配置

状态：已确认

- 默认零配置，文件就绪即可开始。
- 高级设置仅提供严格、标准、宽松三级匹配灵敏度。
- 不显示尚未生效的模式、Embedding、模型、外部 API 和 topK。
- 支持的内容维度默认全部检测，跨格式时自动降级并解释。
- Embedding 接入前不宣传为 AI 语义比对。

### D06 进度与取消

状态：已确认

- 使用真实阶段进度，不伪造平滑百分比和预计剩余时间。
- 阶段包括文件校验、解析两份文档、比对和整理结果。
- 展示已用时间和非阻塞警告。
- 取消必须真实终止任务并释放资源。
- 取消或失败后保留文件选择，支持重试。

### D07 结果工作台定位

状态：已确认

- 采用差异优先复核，不承诺 Word 页面级复刻。
- 左侧为差异导航，中间为双栏原文与上下文，右侧为详情和审核操作。
- 支持上下文展开、上一项、下一项和键盘导航。
- 表格使用专用双表格视图。
- 页码仅在来源可靠时显示。

### D08 差异分类与筛选

状态：已确认，审核筛选由 D10 修订

- 变化类型：新增、删除、修改、移动、拆分、合并、低置信匹配。
- 完全相同默认隐藏，但保留统计和显式开关。
- 维度筛选：正文、表格、格式、批注、修订。
- 默认保持原文顺序。
- 搜索覆盖两侧文本、摘要和审核备注。
- 筛选项显示数量，并显示当前结果数与总数。

### D09 行内细粒度差异

状态：已确认

- 修改项默认显示行内细粒度差异。
- 中文按字符、英文按单词处理，数字和标点保留边界。
- 基准版删除内容使用红色浅底和删除线；待审版新增内容使用绿色浅底和下划线。
- 高亮始终可见，不依赖 hover。
- 提供隐藏细节选项。
- 超长段落允许退化为段落级展示，并明确提示。

### D10 审核记录模型

状态：已确认

- 审核状态：未审核、已确认、待确认、已忽略。
- 重要标记为独立布尔字段，可与任意审核状态组合。
- 支持可选备注并自动保存。
- 工作台展示已处理进度和“下一条未审核”。
- 审核筛选由状态筛选和“仅看重要”组合构成。
- 导出报告包含审核状态、重要标记和备注。

### D11 本地历史与恢复

状态：已确认

- 使用本地持久化保存任务、结果和审核进度。
- 应用重启后可恢复未完成复核。
- 顶部入口访问最近比对，不单独建设首页。
- 支持重新打开、删除和清空记录。
- 不复制原始文档文件。
- 用户进一步指定可使用 SQLite，并要求参考现有数据库设计。

### D12 SQLite 保存与清理策略

状态：暂定执行，必须写入正式开发文档

- 保存压缩后的 Document AST、Diff AST、任务摘要、耗时和错误状态。
- 审核信息使用独立 `review_annotations` 表，不写入算法结果。
- 不复制原始文件，只保存必要元数据与哈希。
- 默认保留最近 20 个完整任务。
- 数据库与缓存总上限默认 1 GB，按最久未访问顺序清理。
- 正在复核或用户标记保留的任务不自动清理。
- 失败和取消任务只保存诊断摘要。
- Schema 使用版本化迁移；数据库损坏时隔离备份并新建数据库。
- SQLite 仅由 Electron 主进程访问。

### D13 正式支持平台

状态：已确认

- Windows 10/11 x64 为 V0.2.2 唯一 P0 验收平台。
- dev 与打包客户端都必须跑通真实流程。
- 安装包包含 Rust、SQLite 和解析依赖，不要求用户安装开发环境。
- 中文路径、空格路径、长文件名和非管理员安装纳入测试。
- macOS/Linux 保持代码可移植，但不作为本版本发布阻塞条件。
- Windows 7/8/8.1 不属于支持范围；不为其降级 Electron 或维护遗留发布分支。

### D14 本地内容加密

状态：已确认

- Document AST、Diff AST、审核备注和详细错误上下文使用 AES-256-GCM 加密后写入 SQLite。
- 随机主密钥通过 Electron `safeStorage` 和 Windows DPAPI 保护。
- 列表所需的非敏感元数据可保持明文。
- 日志禁止记录正文、备注、密钥和完整路径。
- 密钥不可用或数据被篡改时隔离记录，不阻止应用启动。
- 清空历史时同步删除缓存和相关密钥材料。
- V0.2.2 不引入 SQLCipher。

### D15 报告导出

状态：已确认

- 正式支持 HTML 和 Markdown，HTML 为主要格式。
- 可导出全部差异、当前筛选、仅重要或仅待确认。
- 默认不导出完全相同内容。
- 报告包含文档信息、统计、逐项差异、审核状态、重要标记和备注。
- HTML 为离线单文件并支持打印；暂不直接生成 PDF/DOCX。
- 使用系统保存对话框，完成后可打开报告或所在目录。
- 导出基于 SQLite 中已保存的任务快照。

### D16 视觉依据

状态：原推荐已被用户要求替换

- 正式视觉实现以仓库根目录 `ui-style-preview.html` 为参考。
- 需要沿用其视觉 tokens、明暗主题、差异语义色和主要布局语言。
- 仍需讨论预览稿中的侧边栏、卡片列表等结构如何适配已确认的信息架构。

### D17 UI 技术栈

状态：已确认

- React 19。
- Tailwind CSS 4。
- shadcn/ui + Radix Primitives。
- Lucide React。
- Sonner。
- `react-resizable-panels`。
- `@tanstack/react-virtual`。
- 将预览稿视觉变量映射为正式语义化主题 tokens。
- 不并用 Ant Design、MUI 等第二套组件体系。
- 字体必须离线可用，不从 Google Fonts 在线加载。
- 技术栈与组件使用规则必须更新到 Desktop 开发文档。

### D18 全局导航结构

状态：已确认

- V0.2.2 不使用常驻全局侧边栏。
- 应用采用紧凑顶栏承载品牌、新建比对、最近比对、主题和设置。
- 导出报告属于结果页的上下文操作，不作为全局导航项。
- 结果工作台继续保留左侧差异导航；它属于审核工具，不是全局导航。
- 沿用 `ui-style-preview.html` 的视觉 tokens、明暗主题、差异语义色、边框和面板语言，但不沿用其侧边栏信息架构。

### D19 新建比对页结构

状态：已确认

- 使用单屏、双文档并列的工作台布局，不设置欢迎页、功能介绍或统计卡片。
- 顶部沿用紧凑全局顶栏；主区域标题为“新建比对”。
- “基准文档”和“待审文档”使用两个等宽文件槽，支持点击选择和拖放，中间提供交换操作。
- 文件选中后在原位置展示文件名、格式、大小、页数或解析能力，以及更换和移除操作。
- 不支持、超限、加密和能力降级提示紧贴对应文件槽展示。
- 文件槽下方展示本次实际检测维度；跨格式不可用维度置灰并解释原因。
- 高级设置默认收起，只提供严格、标准、宽松三级匹配灵敏度。
- 底部操作区提供“开始比对”主操作；文件未就绪时禁用并说明缺失条件。
- 窄窗口下两份文档改为上下排列，保证文件名、状态与错误信息可读。
- 视觉沿用预览稿的克制边框、低圆角和语义色，避免多层卡片堆叠。

### D20 处理中页面结构

状态：已确认

- 使用聚焦任务状态的阶段清单，不使用大号总百分比或装饰性动画页面。
- 顶部保留紧凑全局顶栏；任务运行期间离开当前任务需要确认。
- 主区域顶部展示两份文档的简要信息和当前匹配灵敏度。
- 纵向展示文件校验、解析基准文档、解析待审文档、执行比对和整理结果等真实阶段。
- 已完成、当前活动和未开始阶段使用明确但克制的状态区分。
- 仅在引擎提供真实完成量时显示阶段百分比，不推算或伪造总进度。
- 固定展示已用时间、当前阶段处理量，以及可展开的非阻塞警告。
- 提供“取消比对”操作并进行确认；终止过程中禁用重复操作并显示正在终止。
- 成功后自动进入结果工作台。
- 失败后原页显示可操作错误，保留文件选择并提供“重试”和“返回调整”。
- 不显示预计剩余时间、循环宣传文案或伪造的平滑进度。

### D21 历史记录页面结构

状态：已确认

- 顶部“最近比对”入口打开同一窗口内的独立历史视图，不使用侧边抽屉或卡片墙。
- 历史视图沿用紧凑全局顶栏，页面顶部显示“最近比对”和“新建比对”主操作。
- 主体使用紧凑表格，每个任务占一行。
- 主要列包括文档组合、任务状态、差异摘要、审核进度和最后访问时间。
- 文档组合突出文件名，格式和文件大小等信息作为次要文本。
- 已完成、复核中、失败和已取消等状态使用克制的语义标签。
- 差异摘要只显示关键数量，不在列表中展开正文。
- 点击整行恢复任务，低频操作收纳到行末更多菜单。
- 空状态只提供“新建比对”，不展示示例数据或营销内容。
- 窄窗口优先保留文档组合、状态、审核进度和时间，允许隐藏次要列。

### D22 结果工作台结构与视口扩展

状态：已确认

- 使用三栏联动工作台，不沿用预览稿以差异卡片连续流作为中心内容的结构。
- 全局顶栏下方设置紧凑任务栏，展示文档组合和审核进度，并提供“新建比对”和“导出报告”。
- 任务栏下方使用单行统计与筛选区，展示当前结果数、差异类型数量、审核状态筛选、重要筛选和搜索。
- 左栏为可虚拟滚动的差异导航列表，展示类型、章节或摘要、审核状态及重要标记。
- 中间定义为可替换的中央比对视口，不与单一文本布局绑定。
- V0.2.2 提供段落差异审阅视图和表格专用审阅视图；段落视图并排显示基准文档、待审文档及必要上下文。
- 右栏展示差异详情和审核操作，包括匹配信息、格式或结构变化、审核状态、重要标记和备注。
- 左侧选中项驱动中央视口与右侧详情，并以同一 `matchId` 维持选择和定位状态。
- 上一项、下一项和“下一条未审核”放在稳定工具栏位置，不随正文滚动。
- 统计使用紧凑数字与标签，不使用占据较大高度的统计卡片。
- 完全相同内容默认不进入导航列表，开启后按普通条目展示。
- 后续全保真模式可作为新的中央视口接入，与差异审阅共用导航、筛选、选择状态和审核记录。
- 全保真视口预期支持双文档并排、同步滚动、差异锚点和面板折叠；其实现依赖页面版式、节点坐标映射和 DOCX 渲染方案，不属于 V0.2.2。
- V0.2.2 只预留视口边界和模式切换位置，不显示尚未实现的“全文预览”入口。

### D23 工作台面板尺寸与小窗口策略

状态：已确认

- 应用窗口默认保持 `1280 x 800`，最小尺寸设为 `1024 x 700`。
- 三栏默认宽度为左侧差异导航 `280px`、右侧详情 `320px`，中央视口占据剩余空间。
- 左栏允许在 `240–360px` 间调整，右栏允许在 `280–420px` 间调整，中央视口至少保留 `560px`。
- 左右面板均支持通过图标按钮折叠和恢复。
- 窗口宽度低于约 `1120px` 时自动收起右侧详情；详情通过覆盖式面板临时打开。
- 左侧差异导航不随窗口宽度自动消失，但允许用户主动折叠。
- 后续全保真预览可使用同时折叠左右面板的专注模式。
- 用户设置的面板宽度和折叠偏好保存在本地；响应窗口宽度的临时收起不覆盖用户偏好。
- 各面板独立纵向滚动，页面整体不产生横向滚动；超宽表格仅在中央表格视口内部横向滚动。
- 顶栏、任务栏和审核导航保持固定，正文滚动时关键操作不移出视口。

### D24 表格差异交互

状态：已确认

- 中央视口使用双表格对照，不尝试将任意表格合成为单一结果表。
- 默认并排显示基准表格和待审表格；空间不足时可通过分段控件切换为上下排列。
- 根据行列对齐结果插入占位，使两侧对应内容保持在相同视觉位置。
- 两侧默认同步纵向滚动；横向滚动各自独立，并提供滚动同步开关。
- 表头、行号和必要的首列保持固定。
- 新增内容仅在待审侧使用绿色，删除内容仅在基准侧使用红色。
- 修改单元格在两侧分别使用删除红和新增绿，并在单元格内部显示细粒度文本差异。
- 合并和拆分单元格使用独立蓝色边框或标记，不与内容增删颜色混淆。
- 点击变化单元格后同时选中两侧对应单元格；右侧详情展示旧值、新值、相似度和合并范围变化。
- 提供“上一处变化 / 下一处变化”，按变化单元格或结构变化导航。
- 新增、删除的整行或整列同时使用边缘标记和浅色背景，避免只依赖颜色表达范围。
- 嵌套表格默认显示变化标记，按需在单元格内展开或进入详情，不默认展开所有层级。

### D30 SQLite 库、迁移与备份

状态：已确认

- V0.2.2 以 D12 的 Electron 主进程数据库所有权为准，替代旧数据库设计中由 Rust `rusqlite` 管理历史库的方案。
- 使用 `better-sqlite3` 作为 Desktop 生产依赖，不依赖当前 Electron 运行时中的内置 `node:sqlite`。
- 不引入 ORM；使用参数化 SQL、薄仓储层、显式事务和共享 IPC 契约。
- 数据库服务由 Electron 主进程拥有，连接及压缩、加密、大型 BLOB 处理运行在主进程管理的专用 Worker 中，写操作串行执行。
- Rust 引擎不直接访问历史数据库，只通过 JSON-RPC 返回计算结果。
- V0.2.2 只创建任务、文档快照、差异快照、`review_annotations`、设置和迁移记录等真实需要的表。
- 使用递增版本的 TypeScript 迁移模块；每次迁移在单个事务中执行，成功后更新 `PRAGMA user_version`。
- 迁移记录保存版本、名称、校验值和应用时间；生产环境只支持前向迁移。
- 启动时执行 `quick_check`，并配置外键、WAL、`busy_timeout` 和合理同步级别。
- Schema 迁移前使用 SQLite Online Backup API 创建一致性备份，不在 WAL 模式下直接复制单个数据库文件。
- 最多保留 5 个通过校验的自动备份，备份仅包含加密后的 BLOB，不产生明文快照。
- 迁移失败时回滚并保留迁移前备份，不在失败 Schema 上继续写入。
- 数据库损坏时隔离数据库及 WAL/SHM 文件，新建数据库并保留诊断摘要，不阻止应用启动。
- 备份依赖当前 Windows 用户通过 `safeStorage` 保护的主密钥，V0.2.2 不承诺跨设备恢复。
- 清空历史时同步清理备份和对应密钥材料。
- `electron-builder` 必须显式处理原生 `.node` 文件，并针对 Electron 33 Windows x64 重建 `better-sqlite3`。
- CI 必须包含安装包启动测试；单元测试使用内存数据库，迁移、备份、损坏恢复和加密测试使用临时真实文件。
- 正式开发文档必须同步修订旧 `rusqlite` 历史库方案。

### D31 Rust 子进程生命周期与恢复

状态：已确认

- Electron 主进程维护一个长生命周期 Rust 子进程，并管理停止、启动中、就绪、忙碌、重启中、不可用状态。
- V0.2.2 只允许一个活动比对任务；并发请求返回明确的 `ENGINE_BUSY`，不静默排队。
- 开发环境从明确配置的 Cargo 输出路径启动；打包环境固定从 `process.resourcesPath/engine/bidlens-engine.exe` 启动。
- Rust 二进制位于 ASAR 外，Windows 启动时隐藏控制台窗口。
- 发布构建先执行 `cargo build --release --locked`，再将二进制和 SHA-256 清单暂存到 Desktop 构建目录并通过 `extraResources` 打包。
- 构建或安装包验收发现二进制缺失、校验不符或无法执行时直接失败。
- 保持 stdio JSON-RPC；stdout 只允许协议消息，日志只写 stderr。
- 启动后握手返回引擎版本、协议版本和能力列表；协议不兼容时禁止开始任务。
- Rust 主循环改为可响应的任务协调器，比对计算在独立任务中运行。
- 协议增加真实阶段进度通知、`compare.cancel` 和 `shutdown`，请求使用唯一 ID 关联响应并设置分场景超时。
- 空闲时定期 `ping`；活动任务期间协调器保持响应，避免把长计算误判为崩溃。
- 应用退出时先发送 `shutdown`，超时后终止子进程，确保不遗留后台进程。
- Rust 算法在主要循环和阶段边界检查取消令牌并释放中间资源。
- 取消在短超时内无响应时由 Electron 终止并重启引擎，取消后丢弃迟到结果且不保存完整历史快照。
- 子进程异常退出时，所有待处理 RPC 以结构化 `ENGINE_EXITED` 失败；当前任务保存脱敏诊断但不自动重跑。
- 引擎服务使用有限次数指数退避自动重启；连续失败后进入不可用状态，历史查看和报告导出仍可使用。
- stderr 日志不得包含正文、完整路径、审核备注或 AST。
- 测试至少覆盖协议握手、版本不兼容、真实取消、强制终止、迟到结果、崩溃重启、退出清理、打包路径和 Windows 安装包真实流程。

### D32 国产化服务形态与 Rust Web 扩展

状态：已确认，属于未来架构约束

- 国产化不等同于必须服务化；国产 Linux 桌面交付可继续使用 Electron 与本地 Rust 子进程。

### D33 性能目标、测试集与验收指标

状态：已确认

- 参考环境为 Windows 11 x64、16 GB 内存、SSD、Intel Core i5-1240P 或同级 Ryzen 5，并使用正式安装包和冷缓存测试。

- UI 不出现超过 100 ms 的主线程长任务；滚动目标为 60 FPS，P95 帧时间不超过 16.7 ms。
- 测试集包含固定版本和 SHA-256 的真实脱敏 DOCX、文本型 PDF、跨格式文档及 50、200、1000 页规模。
- 覆盖纯正文、复杂表格、格式、批注、修订、重复条款、完全不同、基本相同、顺序重排等场景。
- 覆盖超宽表格、长段落、嵌套表格、中英数字标点混排、中文路径、空格路径、长文件名、损坏文件和接近 100 MB 文件。
- 建立人工标注黄金数据集，覆盖新增、删除、修改、移动、拆分、合并和低置信匹配。
- 各主要分类精确率和召回率均不低于 90%，法务关键条款小型验收集不允许漏报已标注变化。
- 表格单元格、格式、批注和修订的确定性夹具必须 100% 命中预期结果。
- 同一输入、配置和版本重复运行时，结果顺序和匹配 ID 必须稳定。
- PR CI 跑小型正确性与性能回归，每夜跑 200 页基准，发布候选在固定 Windows 机器跑 1000 页和接近 100 MB 验收。
- 性能较基线回退超过 10% 时告警，超过硬门槛时阻止发布。
- 第一轮实测后更新旧文档中未经验证且互相冲突的性能指标，并保留原始测量报告。

### D36 Swiss Minimalism 设计哲学

状态：已确认

- 正式 UI/UX 采用瑞士极简主义（Swiss Minimalism）作为设计哲学，而不是仅参考其表面配色。
- 视觉风格必须干净、克制、功能导向，无多余装饰。
- 信息层级主要通过字体大小、字重和中性色深浅建立，不依赖装饰性容器。
- 使用严格网格、清晰对齐和一致间距形成秩序；功能分组优先使用留白与分隔线。
- 保留充足留白，避免信息过载；结果工作台可保持专业信息密度，但必须具备明确分区和稳定扫描路径。
- 色彩以低饱和度中性色为主，只保留一个克制的产品强调色用于主要操作、焦点和选择状态。
- 差异语义色仅用于新增、删除、修改、结构变化和必要状态，不作为装饰色。
- 不使用渐变、装饰性插画、光斑、拟物效果或无功能意义的色块。
- 控件与工具面板使用 1px 边框和 4–6px 低圆角；阴影仅用于模态对话框、Toast 等真实浮层。
- 字体必须离线可用，拉丁字符优先使用 Arial/Segoe UI，中文回退到 Microsoft YaHei UI，并保持字距为 `0`。
- 明暗主题均保持同一信息层级与语义色含义，深色主题使用中性炭灰而非大面积蓝黑色。
- Lucide 图标只用于识别命令和状态，不作为装饰；动效只用于状态反馈并保持短促克制。
- `v022-ui-ux-prototype.html` 是当前视觉评审基线，后续原型调整必须同步本决策、Spec 与任务验收。
