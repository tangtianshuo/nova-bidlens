# BidLens V0.3 多投标文件雷同性风险审查 PRD

> 文档版本：1.0
> 决策日期：2026-07-21
> 状态：产品主干已确认，作为 V0.3 实施与验收的唯一产品依据
> 适用版本：V0.3.0、V0.3.1、V0.3.2

## 1. 文档权威性

本 PRD 定义 BidLens V0.3 的产品定位、范围、业务语义、用户流程和验收标准。

文档优先级如下：

1. 本 PRD：产品目标、范围、流程和验收。
2. `docs/architecture.md` 及分层架构文档：运行边界和技术设计。
3. `docs/roadmap.md`：版本拆分、进度和交付顺序。
4. `AGENTS.md`、`CLAUDE.md` 和分层 `AGENT.md`：开发约束和文档导航。

开发指导文件不得复制并重新定义 Shared 契约；真实类型以 `packages/shared/src/` 为准。

## 2. 产品定位

BidLens 是面向招标人、招标代理机构和审计复核人员的本地桌面工具。产品以招标项目为单位分析多份投标文件，发现文本、表格、实体和关键事实的雷同性风险，并提供可追溯证据供人工判断。

BidLens 的核心责任是发现、组织和解释风险线索，不认定串标、围标或其他法律结论。

### 2.1 核心价值

- 将 2-8 份投标文件纳入同一项目审查。
- 通过招标基线过滤公共内容，降低误报。
- 以原文件、页码、章节、AST 节点和原文证据解释风险。
- 完全离线运行，敏感数据不离开本机。
- 支持人工确认、忽略、标记重要、备注和报告归档。
- 为后续本地语义增强保留稳定的数据和业务协议。

### 2.2 目标用户

| 用户 | 核心任务 | 主要痛点 |
|---|---|---|
| 招标人 | 对同一项目的多份投标文件进行风险筛查 | 人工逐份检查耗时且容易遗漏跨文件关系 |
| 招标代理机构 | 形成可复核的雷同性审查材料 | 缺少统一证据、文件关系和报告流程 |
| 审计与复核人员 | 复核风险依据并形成意见 | 结果缺少原文定位和算法解释 |

投标人内部自查、企业知识库和 AI 编标不进入 V0.3 产品范围。

## 3. 产品责任边界

### 3.1 产品必须做到

- 输出低、中、高风险线索或“结果不完整”。
- 保存每条风险的原始证据、检测依据和规则版本。
- 区分算法风险和人工复核意见。
- 明确分析完整性、招标基线状态和降级原因。
- 报告声明结果仅供审查参考。

### 3.2 产品禁止做到

- 输出“串标概率”或“围标概率”。
- 使用“已认定串标”等法律定性文案。
- 让人工意见覆盖或修改原始算法证据。
- 在未完成规定检测阶段时显示正常低风险。
- 在 V0.3.0 宣称具备大幅语义改写检测能力。
- 将生产文档、原文、路径或个人信息写入非加密日志。

## 4. 版本拆分

### 4.1 V0.3.0：基础雷同性审查闭环

V0.3.0 不依赖 Embedding，交付完整业务闭环：

- 真实项目和多文件导入。
- DocumentAst 解析和 ReviewNode 提取。
- 文本、表格、实体、关键事实四类检测器。
- 招标公共内容过滤。
- 四路稀疏召回和确定性精排。
- 风险聚合、证据复核、历史、恢复和报告。
- 完全离线和加密持久化。

### 4.2 V0.3.1：本地语义增强

- BGE-M3 ONNX 本地 Provider。
- 稳定 Chunk 和加密向量缓存。
- 项目内语义 Top-K 召回。
- 语义、词法、结构、实体和事实混合精排。
- 模型安装、校验、替换、删除和降级。

V0.3.1 必须等待模型许可和运行证据门槛通过，不得静默绕过门槛。

### 4.3 V0.3.2：质量与发布加固

- 金标数据集和盲测。
- 阈值和预设校准。
- F1、明显错误率和 Recall@TopK 验收。
- 大项目性能、安全、打包、签名和安装验证。

## 5. V0.3.0 范围

### 5.1 范围内

- 每个项目 2-8 份投标文件。
- 0-1 份招标基线。
- DOCX 和文字版 PDF。
- 单份文件最多 1000 页，项目累计最多 4000 页。
- 文本完全一致、轻度修改和局部重排检测。
- 表格行、关键单元格和参数组合检测。
- 人名、身份证、手机号、邮箱、统一社会信用代码和公司名称提取。
- 金额、比例、日期、工期、编号、资质等级、否定和承诺事实提取。
- 文件对风险、关系矩阵和项目风险。
- 人工复核、审计记录和报告导出。

### 5.2 范围外

- Embedding 和大幅语义改写检测。
- OCR、扫描版 PDF 和图片指纹。
- 文件制作环境和设备硬件指纹。
- 经济标清单、定额和人材机规则。
- 旧版 `.doc`、加密文件和专有投标文件包。
- LLM、RAG、企业知识库、AI 编标和 Agent。
- 云端账号、在线处理和局域网服务端。

## 6. 核心领域模型

```text
AnalysisProject
├── TenderBaseline 0..1
├── Submission 2..8
│   └── DocumentVersion
│       └── DocumentAst
│           └── ReviewNode
├── RiskFinding
│   └── Evidence
├── FilePairAssessment
├── ProjectRiskAssessment
├── ReviewDecision
├── AnalysisCheckpoint
└── ExportedReport
```

### 6.1 数据语义

- `DocumentAst`：不可变的解析事实和原文定位依据。
- `ReviewNode`：检测器使用的统一分析单元。
- `RiskFinding`：多个检测器聚合后的一条风险发现。
- `Evidence`：文件、AST 节点、原文和算法依据。
- `ReviewDecision`：人工意见，与算法结果独立。
- `RiskAssessment`：版本化确定性规则产生的文件对或项目评估。

真实字段以 Shared 契约为准，本 PRD 不替代 TypeScript 类型定义。

## 7. ReviewNode 和关键节点

ReviewNode 必须至少保存：

- 稳定 ID 和来源 AST 节点 ID。
- 节点类型：标题、段落、列表项、表格行或表格单元格。
- 章节路径、文档顺序、页码范围。
- 原始文本、规范化文本和内容 Hash。
- 提取的实体、关键事实和业务标签。
- 表格位置和关键单元格信息。

### 7.1 首版业务标签

```text
bidder-identity
authorization
qualification
personnel
performance
technical-solution
schedule
quality
equipment
commercial
commitment
generic
```

一个节点可以有多个标签。标签用于召回优先级和解释，不删除未标记节点。

### 7.2 关键节点判定

满足以下任一条件的节点可标记为关键节点：

- 章节路径命中业务词典。
- 包含强实体。
- 包含两个以上关键事实。
- 表头、关键列或参数组合命中。
- 文本具有较高项目内稀有度。
- 与其他 Submission 形成候选关系。

普通节点仍参与基础文本召回，防止关键节点规则造成漏检。

## 8. 检测器设计

### 8.1 文本检测器

- 完全一致文本。
- 增删词、局部替换和轻度调序。
- 长文本稀有片段重复。
- 章节和节点类型结构相似。

### 8.2 表格检测器

- 表头和列语义规范化。
- 表格行和关键单元格组合。
- 设备名称、型号、数量和参数组合。
- 忽略纯样式变化，保留表、行、单元格定位。
- 同一表格重复行不得重复累计风险。

### 8.3 实体检测器

- 身份证、手机号、邮箱、信用代码使用规范化精确匹配。
- 公司名称保留规范化值和原始值。
- 人名属于弱实体，必须结合章节、邻近文本和其他事实。

### 8.4 关键事实检测器

- 金额、比例、日期、工期、编号和资质等级。
- 否定、承诺和限制性表达。
- 相同事实组合提高证据强度。
- 文本相似但事实不同生成“事实冲突”证据，不直接删除候选。

### 8.5 招标公共内容过滤

- 有基线时识别投标文件引用的招标原文。
- 被过滤内容保留 Evidence 和过滤原因。
- 被过滤内容不直接累计高风险。
- 无基线时允许分析，但新建、处理、结果、历史和报告持续显示误报警告。

## 9. 候选召回和确定性精排

V0.3.0 使用四路稀疏召回：

1. 内容 Hash 索引。
2. 字符 n-gram 倒排索引。
3. 实体和关键事实索引。
4. 表格签名索引。

禁止无条件构造所有节点的完整 `N x M` 矩阵。只有不同 Submission 之间的节点可以形成风险候选。

精排必须保留以下贡献：

- 精确一致分。
- 词法分。
- 结构分。
- 实体分。
- 关键事实分。
- 招标公共内容折扣。
- 通用模板折扣。
- 关键事实冲突惩罚。
- 最终分数和规则版本。

V0.3.0 的严格、标准和宽松预设使用版本化工程阈值，不宣传为经过金标校准的准确率结论。

## 10. 风险聚合

```text
Detector Candidate
-> RiskFinding
-> FilePairAssessment
-> ProjectRiskAssessment
```

### 10.1 RiskFinding

- 同一原文被多个检测器命中时合并为一个 Finding。
- 保留全部检测依据，不重复累计数量。
- 至少包含两份不同 Submission 的 Evidence。
- 保存方向覆盖率、对称相似度、置信度和贡献明细。

### 10.2 文件对风险

- 保留 A 到 B、B 到 A 两个方向覆盖率。
- 提供用于矩阵展示的对称相似度。
- 保存主要 Finding、各风险数量、规则版本和完整性。

### 10.3 项目风险

项目风险综合：

- 高价值 Finding。
- 涉及文件数量和关系网络。
- 强实体和跨章节稀有内容。
- 招标公共内容和模板折扣。
- 分析完整性。

项目风险不得简单按发现数量求和。任一关键检测阶段失败时，项目风险为 `incomplete`，不得显示正常低风险。

## 11. 人工复核

人工复核状态与算法风险分离：

- `pending`：待确认。
- `confirmed`：确认作为风险线索。
- `ignored`：人工忽略。
- `important`：独立布尔标记，不属于状态枚举。
- `note`：人工备注。

人工意见不修改算法分数、不删除 Evidence、不覆盖规则版本。重新分析生成新的分析结果版本，历史结果和人工意见保留。

## 12. 用户工作流和信息架构

V0.3.0 使用单一项目式信息架构：

```text
项目列表
├── 新建项目
├── 项目处理
├── 项目结果
└── 设置
```

### 12.1 新建项目

1. 填写项目名称并选择可选招标基线。
2. 添加 2-8 份投标文件并完成格式、大小、页数和重复 Hash 校验。
3. 选择检测预设，确认文件和基线状态后开始分析。

### 12.2 处理页面

- 展示真实项目阶段、每份文件状态、已用时间、警告和结构化错误。
- 不显示虚构百分比和固定阶段耗时。
- 提供取消、替换文件、继续无基线和恢复等上下文操作。

### 12.3 结果页面

一级视图：

- 风险概览。
- 关系矩阵。
- 发现项。
- 报告。

点击 RiskFinding 进入证据复核工作台：左侧为发现项导航，中间为原文证据，右侧为检测依据、定位和人工复核。

### 12.4 双文档 Diff 定位

双文档版本比对不再作为独立产品入口。DiffAst、表格 Diff、格式、批注和修订能力保留为 RiskFinding 和文件对的深入证据工具。V0.3.0 不维护独立的新建比对、历史和报告产品流程。

## 13. 状态、阶段和恢复

项目状态、分析阶段和单文件状态必须分离。

### 13.1 项目状态

```text
draft
running
ready
partial
interrupted
failed
cancelled
```

### 13.2 V0.3.0 分析阶段

```text
validating
parsing
extracting-nodes
extracting-entities
filtering-tender-content
recalling-candidates
detecting
aggregating
persisting
completed
```

### 13.3 检查点

以下阶段完成后加密持久化：

- 项目创建。
- 文件校验。
- 文档解析。
- ReviewNode 提取。
- 招标内容过滤。
- 候选召回。
- 检测器完成。
- Finding 聚合。
- 项目完成。

检查点保存输入 Hash、处理版本、已完成检测器、中间结果引用、阶段、警告和错误。

### 13.4 异常处理

| 异常 | 产品行为 |
|---|---|
| 投标文件校验失败 | 启动前阻止分析 |
| 单份投标文件解析失败 | 暂停项目，允许替换或移除 |
| 招标基线解析失败 | 用户确认后允许切换到无基线模式 |
| 单个检测器失败 | 标记 Partial，不显示正常低风险 |
| 用户取消 | 保留 AST 和可复用缓存，不生成正式报告 |
| 应用异常退出 | 标记 Interrupted，从最近兼容检查点恢复 |
| 原文件丢失 | 历史证据可读，重新分析需重新定位并校验 Hash |
| 完成项目重开 | 读取快照，不重新分析 |

## 14. 本地持久化和安全

V0.3.0 默认不复制用户原始投标文件。产品保存：

- 加密文件路径。
- 文件 Hash、名称、格式、大小和版本。
- 加密 DocumentAst 和 ReviewNode。
- 实体、关键事实、Evidence 和检查点。
- Finding、风险评估、人工意见和报告记录。

原文件移动或删除后，历史证据仍可查看。重新分析必须重新定位文件并校验 Hash；Hash 不一致时创建新的 DocumentVersion。

### 14.1 加密范围

路径、AST、ReviewNode、实体、事实、Evidence 原文、人工备注、检查点和未来向量必须使用 AES-256-GCM 加密。密钥由 Windows `safeStorage` 保护。

### 14.2 清理策略

- 清理缓存：仅删除可重建数据。
- 删除项目：删除项目结果、证据、复核和孤立缓存。
- 永久保留：项目不参与自动保留上限清理。
- 共享数据仅在没有其他项目引用相同文档版本时删除。

## 15. 报告和审计

### 15.1 报告格式

- PDF：正式归档、流转和签批。
- HTML：本机查看完整证据。
- Markdown：兼容和二次整理，不作为正式主报告。

### 15.2 报告必含内容

- 责任边界声明。
- 项目和文件范围。
- 招标基线状态。
- 完整、降级或部分分析状态。
- 解析器、节点提取器、检测器和规则版本。
- 项目风险、关系矩阵和文件对风险。
- RiskFinding、Evidence 和人工意见。
- 未覆盖能力、失败检测器和警告。
- 报告范围、结果 Hash 和生成时间。

报告范围可以选择全部、已确认、重要或当前筛选结果，但不得移除完整性、基线、版本、失败说明和责任声明。

### 15.3 审计事件

记录项目创建、文件增删替换、无基线确认、取消、恢复、接受 Partial、人工复核、备注变更、报告导出、项目删除和缓存清理。

## 16. V0.3.0 发布验收

### 16.1 真实业务闭环

必须使用真实文件走通：

```text
新建项目
-> 导入文件
-> 校验
-> 解析和提取
-> 过滤和召回
-> 四类检测
-> 风险聚合
-> 关系矩阵
-> 证据复核
-> 人工意见
-> 历史重开
-> 报告导出
```

覆盖 DOCX、文字版 PDF、混合格式、有/无基线、2/8 份文件、取消、失败、恢复、Partial 和原文件重新定位。

### 16.2 检测和追溯

- 完全一致文本不得漏检。
- 强实体规范化精确匹配不得误配。
- 表格 Evidence 定位到表、行和单元格。
- 关键事实区分相同和冲突。
- 招标公共内容不直接累计高风险。
- 跨检测器命中不重复计数。
- 所有 Finding 至少有两份不同文件的 Evidence。
- 相同输入和版本产生相同结果。
- 每条 Finding 可追溯到文件、章节、页码、AST/ReviewNode、原文、检测依据、规则版本和人工意见。

### 16.3 状态、安全和性能

- 处理页只显示真实阶段和事件。
- 取消、Interrupted、恢复和 Partial 语义正确。
- 完全断网运行，不产生隐式外部请求。
- 敏感数据加密，日志不泄漏路径、原文和个人信息。
- 正式构建不包含可达测试项目或 Fixture 数据。
- 目标设备为 Windows 10/11、4 核 CPU、16GB 内存、无独立显卡。
- 4000 页项目不得 OOM，不构造完整 `N x M` 矩阵。
- 1000+ Finding 的筛选、选择和滚动保持可用。
- 记录解析、提取、召回、检测、聚合和持久化耗时。

V0.3.0 不以语义匹配 F1、Recall@TopK 或 BGE-M3 指标作为发布门槛。

## 17. 未来扩展边界

企业知识库和 AI 编标可以复用文档资产、DocumentAst、ReviewNode、Chunk、引用和模型运行时，但不进入 V0.3。

- 雷同性项目索引与知识库索引必须隔离。
- 不同索引用途使用独立 Chunk Profile 和 Index Namespace。
- AI 生成内容使用独立 DraftDocument，不修改来源 DocumentAst。
- 生成内容必须通过 Citation 引用来源文档版本和 AST 节点。

## 18. 完成定义
