## 13.1 双模式架构概述

### 13.1.1 动机与背景

BidLens 当前采用 Electron 架构，捆绑 Chromium 渲染引擎。在国产化环境中，这一架构面临严重的兼容性问题：

| 痛点 | 根因 | 影响范围 |
|------|------|---------|
| Chromium 启动失败 | 国产 OS 内核版本低，缺少 Chromium 需要的系统调用 | 麒麟 V10、统信 UOS（旧版） |
| GPU 渲染异常 | 显卡驱动（飞腾集显/龙芯集显）不支持 Chromium 需要的 Vulkan/OpenGL ES 版本 | ARM64 飞腾、LoongArch 龙芯 |
| 沙箱机制报错 | Chromium 的 seccomp-bpf / user namespaces 需要较高内核版本 | 龙芯（Linux 4.19 内核） |
| 字体/输入法异常 | Chromium 自带的 FreeType 版本与国产 OS 预装字体库冲突 | 所有国产 Linux |

**核心矛盾**：应用层无法修复固件/驱动层的不兼容。唯一可行的策略是**不捆绑渲染引擎**，将渲染引擎的选择权交给操作系统厂商。

### 13.1.3 核心设计理念

**渲染引擎选择权交给 OS 厂商**：国产操作系统（麒麟、统信 UOS）均预装了经过适配的系统浏览器（Firefox / Chromium 定制版），由 OS 厂商确保浏览器与底层固件/驱动的兼容性。Web UI 模式利用这一事实，绕过了 Electron 自带 Chromium 的兼容性问题。

**双模式共享同一套前端代码**：React UI 层通过 Transport Adapter 抽象通信接口，同一套代码既可以运行在 Electron Renderer 中（通过 IPC 通信），也可以运行在浏览器中（通过 HTTP/WebSocket 通信）。零代码分叉，零功能差异。

**关键变化**（与 Electron 模式对比）：

| 层级 | Electron 模式 | Web UI 模式 | 变化说明 |
|------|-------------|------------|---------|
| 表现层 | React → Electron Renderer | React → 浏览器标签页 | 同一套代码，运行环境不同 |
| 通信层 | IPC (ipcMain/ipcRenderer) | HTTP REST + WebSocket | 通过 Transport Adapter 抽象 |
| 应用层 | Electron Main Process | Rust HTTP Server (axum) | Rust 直接承载 HTTP，不再经过 Node.js Main |
| 解析层 | Node.js 子进程 | Node.js 子进程（不变） | 无变化 |
| 引擎层 | Rust stdio 子进程 | Rust 直接内嵌（同一进程） | 无 IPC 开销，性能更优 |

### 13.2.2 通信层设计

Web UI 模式下，前端与后端的通信从 Electron IPC 替换为 HTTP REST + WebSocket：

| 通信场景 | Electron 模式 | Web UI 模式 | 说明 |
|---------|--------------|------------|------|
| 请求-响应（如获取设置） | `ipcRenderer.invoke()` | `fetch('/api/settings')` | HTTP REST |
| 流式推送（如比对进度） | `ipcRenderer.on()` | WebSocket `onmessage` | WebSocket 长连接 |
| 文件上传（如提交文档） | IPC 传文件路径 | `multipart/form-data` | HTTP 文件上传 |
| 大文件传输 | IPC + 临时文件 | HTTP Streaming / 分片上传 | 流式传输 |

**与 Electron 模式的关键差异**：
- Rust Core Engine 作为 HTTP Server 的一部分直接运行在同一个 Tokio Runtime 中，**省去了 stdio JSON-RPC 的 IPC 开销**
- Node.js Parser 仍作为子进程运行（通过 stdio JSON-RPC 通信），保持现有的解析层架构不变
- 前端静态资源（Vite 构建产物）由 axum 的 `ServeDir` 直接提供，无需额外的 Web Server

### 13.2.5 文件系统访问

Web UI 模式下，浏览器无法直接访问本地文件系统。所有文件操作通过后端 API 代理：

| 操作 | Electron 模式 | Web UI 模式 | API 端点 |
|------|-------------|------------|---------|
| 选择文件 | Electron `dialog.showOpenDialog()` | HTML `<input type="file">` | `POST /api/documents/upload` |
| 保存文件 | Electron `dialog.showSaveDialog()` | 后端写入 + 下载 | `GET /api/export/:id/download` |
| 读取文件 | Node.js `fs.readFile()` | axum 文件服务 | `GET /api/documents/:id/content` |
| 目录浏览 | Electron `dialog` + `fs.readdir()` | 后端 API 列目录 | `GET /api/files/browse?path=` |

**安全限制**：
- 文件浏览 API 必须限制在用户数据目录和工作目录内，防止路径遍历攻击
- 上传文件大小限制（默认 200MB，可配置）
- 上传文件类型白名单（.docx, .pdf, .md, .html）

### 13.2.6 PWA 增强

为弥补浏览器模式下缺失的桌面集成能力，通过 PWA（Progressive Web App）技术进行增强：

| 能力 | 无 PWA | 有 PWA 增强 | 实现方式 |
|------|--------|------------|---------|
| 安装到桌面 | 需手动打开浏览器 | 桌面图标一键启动 | Web App Manifest + `beforeinstallprompt` |
| 离线运行 | 断网不可用 | 核心 UI 离线可用 | Service Worker 缓存策略 |
| 窗口外观 | 浏览器标签页 | 独立窗口（无地址栏） | PWA standalone display mode |
| 通知 | 无 | 浏览器通知 | Notification API |
| 全屏体验 | 受浏览器 UI 限制 | 全屏无干扰 | PWA + CSS 安全区域适配 |

**Service Worker 缓存策略**：

```
请求类型          策略              说明
─────────────────────────────────────────────
API 请求         Network First     优先网络，离线返回缓存
静态资源 (JS/CSS)  Cache First      优先缓存，后台更新
前端 HTML         Stale-While-     先返回缓存同时后台更新
                  Revalidate
AI 模型文件       不缓存            体积太大，由后端管理
```

## 13.3 通信层抽象设计

### 13.3.1 Transport Adapter 接口定义

通信层是实现代码共享的核心抽象。所有 UI 代码通过统一的 `Transport` 接口与后端通信，不直接依赖 Electron IPC 或 HTTP：

```typescript
// transport/types.ts — 通信层统一接口

/** 通用请求-响应调用 */
interface RequestResponse {
  invoke<TArgs, TResult>(
    channel: string,
    args: TArgs
  ): Promise<TResult>;
}

/** 事件订阅（服务端推送） */
interface EventSubscriber {
  on<T>(event: string, callback: (data: T) => void): () => void; // 返回取消订阅函数
}

/** 文件操作 */
interface FileTransport {
  uploadFile(file: File | Buffer, metadata: FileMetadata): Promise<UploadResult>;
  downloadFile(fileId: string): Promise<Blob>;
}

/** 统一传输层接口 */
interface Transport extends RequestResponse, EventSubscriber, FileTransport {
  /** 当前传输模式 */
  readonly mode: 'electron' | 'http';

  /** 连接状态 */
  readonly connected: boolean;

  /** 连接状态变化监听 */
  onConnectionChange(callback: (connected: boolean) => void): () => void;

  /** 销毁（清理资源） */
  destroy(): void;
}
```

### 13.3.2 Electron IPC Adapter 实现

```typescript
// transport/electron-ipc.ts

class ElectronIPCTransport implements Transport {
  readonly mode = 'electron';

  get connected(): boolean {
    return true; // Electron IPC 始终连接
  }

  async invoke<TArgs, TResult>(channel: string, args: TArgs): Promise<TResult> {
    return window.bidlens.invoke(channel, args);
    // 底层调用 ipcRenderer.invoke()
  }

  on<T>(event: string, callback: (data: T) => void): () => void {
    const handler = (_: any, data: T) => callback(data);
    window.bidlens.on(event, handler);
    return () => window.bidlens.off(event, handler);
  }

  async uploadFile(file: File, metadata: FileMetadata): Promise<UploadResult> {
    // Electron 模式下直接传文件路径
    return window.bidlens.invoke('file:upload', { path: file.path, metadata });
  }

  async downloadFile(fileId: string): Promise<Blob> {
    const buffer = await window.bidlens.invoke('file:download', { fileId });
    return new Blob([buffer]);
  }

  onConnectionChange(_: (connected: boolean) => void): () => void {
    return () => {}; // Electron 模式无需监听
  }

  destroy(): void { /* noop */ }
}
```

### 13.3.3 HTTP/WebSocket Adapter 实现

```typescript
// transport/http-ws.ts

class HTTPWSTransport implements Transport {
  readonly mode = 'http';
  private ws: WebSocket | null = null;
  private listeners = new Map<string, Set<Function>>();
  private _connected = false;
  private baseUrl: string;
  private authToken: string;

  constructor(config: { baseUrl: string; authToken: string }) {
    this.baseUrl = config.baseUrl;
    this.authToken = config.authToken;
    this.connectWebSocket();
  }

  get connected(): boolean {
    return this._connected;
  }

  async invoke<TArgs, TResult>(channel: string, args: TArgs): Promise<TResult> {
    const response = await fetch(`${this.baseUrl}/api/${channel}`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${this.authToken}`,
      },
      body: JSON.stringify(args),
    });

    if (!response.ok) {
      throw new TransportError(response.status, await response.text());
    }

    return response.json();
  }

  on<T>(event: string, callback: (data: T) => void): () => void {
    if (!this.listeners.has(event)) {
      this.listeners.set(event, new Set());
    }
    this.listeners.get(event)!.add(callback);
    return () => this.listeners.get(event)?.delete(callback);
  }

  async uploadFile(file: File, metadata: FileMetadata): Promise<UploadResult> {
    const formData = new FormData();
    formData.append('file', file);
    formData.append('metadata', JSON.stringify(metadata));

    const response = await fetch(`${this.baseUrl}/api/documents/upload`, {
      method: 'POST',
      headers: { 'Authorization': `Bearer ${this.authToken}` },
      body: formData,
    });

    return response.json();
  }

  async downloadFile(fileId: string): Promise<Blob> {
    const response = await fetch(`${this.baseUrl}/api/export/${fileId}/download`, {
      headers: { 'Authorization': `Bearer ${this.authToken}` },
    });
    return response.blob();
  }

  onConnectionChange(callback: (connected: boolean) => void): () => void {
    return this.on('connection:change', callback);
  }

  private connectWebSocket(): void {
    this.ws = new WebSocket(`${this.baseUrl.replace('http', 'ws')}/ws`);

    this.ws.onopen = () => {
      this._connected = true;
      this.emit('connection:change', true);
    };

    this.ws.onmessage = (msg) => {
      const { event, data } = JSON.parse(msg.data);
      this.emit(event, data);
    };

    this.ws.onclose = () => {
      this._connected = false;
      this.emit('connection:change', false);
      // 自动重连
      setTimeout(() => this.connectWebSocket(), 3000);
    };
  }

  private emit(event: string, data: any): void {
    this.listeners.get(event)?.forEach(cb => cb(data));
  }

  destroy(): void {
    this.ws?.close();
    this.listeners.clear();
  }
}
```

### 13.3.4 运行时自动检测与切换

```typescript
// transport/index.ts — 自动选择适配器

function createTransport(): Transport {
  // 检测是否运行在 Electron 环境中
  if (typeof window !== 'undefined' && window.bidlens?.isElectron) {
    console.log('[Transport] Electron IPC 模式');
    return new ElectronIPCTransport();
  }

  // Web UI 模式：从 meta 标签或 URL 获取服务地址
  const serverUrl = import.meta.env.VITE_BIDLENS_SERVER_URL || window.location.origin;
  const authToken = sessionStorage.getItem('bidlens_token') || '';

  console.log('[Transport] HTTP/WebSocket 模式 →', serverUrl);
  return new HTTPWSTransport({ baseUrl: serverUrl, authToken });
}

export const transport = createTransport();
```

**业务代码无感知**：整个应用中只有一处 `createTransport()` 调用。所有组件/服务通过 `transport.invoke()` 和 `transport.on()` 通信，完全不关心底层是 IPC 还是 HTTP：

```typescript
// 业务代码示例 — 两种模式完全一致
const settings = await transport.invoke('settings:get', {});
transport.on('compare:progress', (data) => updateProgressBar(data));
```

## 13.4 Rust 后端双模式运行

### 13.4.1 子进程模式（Electron 场景）

现有的 stdio JSON-RPC 模式不变，Rust 二进制作为 Electron Main Process 的子进程运行：

```
Electron Main Process
  │
  │  spawn('rust-engine', { stdio: ['pipe', 'pipe', 'pipe'] })
  │
  ▼
Rust Engine (stdio 模式)
  ├─ stdin  ← NDJSON 命令
  ├─ stdout → NDJSON 响应
  └─ stderr → 日志
```

### 13.4.2 HTTP Server 模式（Web UI 场景）

新增 HTTP Server 运行模式，Rust 引擎作为独立服务进程运行：

```rust
// Rust 后端模式选择伪代码

#[tokio::main]
async fn main() {
    let config = Config::from_args();

    match config.mode {
        RunMode::Stdio => {
            // 现有模式：stdio JSON-RPC（Electron 子进程）
            run_stdio_server(config).await;
        }
        RunMode::Http => {
            // 新增模式：HTTP + WebSocket Server（Web UI 独立服务）
            run_http_server(config).await;
        }
    }
}

async fn run_http_server(config: Config) {
    let engine = Arc::new(CoreEngine::new(config.clone()).await);
    let state = AppState { engine };

    let app = Router::new()
        // API 路由
        .route("/api/documents/upload", post(upload_document))
        .route("/api/documents/:id", get(get_document))
        .route("/api/compare", post(start_compare))
        .route("/api/compare/:id/status", get(compare_status))
        .route("/api/settings", get(get_settings).put(update_settings))
        .route("/api/models", get(list_models))
        .route("/api/history", get(list_history))
        .route("/api/health", get(health_check))
        // WebSocket
        .route("/ws", get(ws_handler))
        // 前端静态资源
        .fallback_service(ServeDir::new(config.static_dir))
        // 安全中间件
        .layer(AuthMiddleware::new(config.auth_token))
        .layer(CorsLayer::permissive())  // localhost 无需严格 CORS
        .with_state(state);

    let listener = TcpListener::bind(format!("127.0.0.1:{}", config.port))
        .await
        .expect("Failed to bind port");

    println!("BidLens Web UI: http://127.0.0.1:{}", config.port);

    axum::serve(listener, app)
        .await
        .expect("Server error");
}
```

### 13.4.3 启动参数与模式选择

| 参数 | 子进程模式（Electron） | HTTP Server 模式（Web UI） |
|------|---------------------|--------------------------|
| 启动方式 | `rust-engine --mode stdio` | `rust-engine --mode http --port 9527` |
| 通信方式 | stdin/stdout NDJSON | HTTP REST + WebSocket |
| 前端资源 | 不提供 | `--static-dir ./dist` 提供 SPA |
| 认证 | 不需要（进程内） | `--auth-token <random>` Bearer Token |
| 绑定地址 | N/A | `127.0.0.1` only（安全） |
| 日志 | stderr → Electron 主进程收集 | stdout/stdout → systemd journal |

### 13.5.2 各方案可行性与限制

- 适用：麒麟 V10 x86_64、统信 UOS x86_64、Ubuntu/Debian 系
- 限制：ARM64 架构需要交叉编译 Electron + Rust；龙芯/鸿蒙完全不支持
- 优势：桌面集成体验最完整

**方案二：Web UI 模式（国产 Linux）**
- 适用：所有有系统浏览器的国产 OS（麒麟、统信、龙芯）
- 限制：需要 Rust 引擎交叉编译到目标架构（Rust 对 LoongArch 支持已进入 Tier 2）
- 优势：绕过 Chromium 兼容性问题；包体积更小；支持远程访问（局域网多人共用）

**方案三：鸿蒙适配（远期）**
- 现状：OpenHarmony 上的 ArkWeb 引擎基于 Chromium 内核，理论上可访问 Web UI
- 风险：鸿蒙应用分发机制与传统 Web 不同；ArkWeb 兼容性未充分验证
- 建议：作为远期探索方向，需在鸿蒙真机上进行 POC 验证

```
  ├─ Rust 交叉编译到 LoongArch (龙芯)

  └─ 安全加固（局域网多用户场景）

Phase 4（远期 — 探索）
  ├─ 鸿蒙 OpenHarmony + ArkWeb POC
  └─ 国产 OS 应用商店上架流程
```

## 13.6 安全设计（Web UI 模式扩展）

### 13.6.1 本地 HTTP Server 安全

Web UI 模式下，Rust HTTP Server 运行在用户本机，浏览器通过 `localhost` 访问。虽然攻击面比 Electron 模式大，但通过以下措施可将风险控制在合理范围：

```
┌──────────────────────────────────────────────────────────┐
│  Web UI 模式安全模型                                       │
│                                                          │
│  浏览器 ──── HTTP Request ────┐                          │
│                               ▼                          │
│                    ┌───────────────────┐                 │
│                    │  安全中间件栈      │                 │
│                    │                   │                 │
│                    │  1. localhost 检查 │ ← 拒绝非本地请求 │
│                    │  2. Token 认证    │ ← 随机启动生成   │
│                    │  3. CSRF 防护     │ ← Origin 校验   │
│                    │  4. 请求限流      │ ← 防暴力攻击     │
│                    │  5. 输入校验      │ ← zod Schema    │
│                    │  6. 路径遍历防护   │ ← 文件操作限定   │
│                    └───────┬───────────┘                 │
│                            ▼                             │
│                    业务处理逻辑                            │
└──────────────────────────────────────────────────────────┘
```

**安全策略清单：**

| 策略 | 实施方式 | 目的 |
|------|---------|------|
| localhost-only 绑定 | `TcpListener::bind("127.0.0.1:PORT")` | 阻止外部网络直接访问 |
| Bearer Token 认证 | 启动时生成随机 Token，写入配置文件；浏览器首次访问时注入 | 防止本机其他应用/网页冒用 |
| CSRF 防护 | 校验 `Origin: http://127.0.0.1:PORT` | 防止跨站请求伪造 |
| 请求限流 | Tower `RateLimit` 中间件，单 IP 60 req/s | 防止暴力攻击 |
| CSP Header | `Content-Security-Policy: default-src 'self'` | 防止 XSS |
| 文件路径校验 | 所有文件操作参数经过 `Path::canonicalize()` + 白名单目录 | 防止路径遍历 |

### 13.6.2 与 Electron 安全模型的对比

| 安全维度 | Electron 模式 | Web UI 模式 | 评估 |
|---------|-------------|------------|------|
| 渲染沙箱 | Chromium 沙箱 + `contextIsolation` | 浏览器自带安全沙箱 | 平手 |
| 通信安全 | IPC（进程内，不可嗅探） | HTTP localhost + Token（可被本机进程嗅探） | Electron 略优 |
| 攻击面 | Preload 暴露有限 API | HTTP Server 暴露 REST API | Electron 略优 |
| XSS 风险 | CSP + contextIsolation | CSP + same-origin | 平手 |
| 文件访问 | Node.js（需路径校验） | 后端代理（需路径校验） | 平手 |
| 远程攻击 | 不可能（纯本地） | 不可能（localhost-only） | 平手 |

**结论**：Web UI 模式的安全模型略弱于 Electron（主要在通信层），但通过 localhost 绑定 + Token 认证 + CSRF 防护的组合，可将风险控制在"本地桌面应用"的等效水平。对于标书比对这一非高敏感场景，安全等级足够。

### 13.7.1 Electron 模式部署（现有）

```
用户下载安装包 → 双击安装 → 启动应用 → 自动更新
  └─ Windows: .exe (NSIS) / .msi

```

现有部署流程不变，参见 [[12-发布升级与运维方案]]。

### 13.7.2 Web UI 模式部署

Web UI 模式提供两种部署方式：

**方式一：桌面快捷方式（单机版）**

```
用户安装 BidLens-WebUI 包
  ├─ /opt/bidlens/                  ← 安装目录
  │   ├── bidlens-server            ← Rust HTTP Server 二进制
  │   ├── parsers/                  ← Node.js 解析层
  │   ├── dist/                     ← React 前端构建产物
  │   └── config.json               ← 默认配置（含随机 auth token）
  │
  ├─ systemd user service           ← 开机自启（用户级）
  │   └── bidlens.service
  │
  └─ 桌面快捷方式                    ← 打开浏览器访问 localhost:9527
      └── bidlens.desktop
```

**systemd 服务文件**：

```ini
# ~/.config/systemd/user/bidlens.service
[Unit]
Description=BidLens Document Intelligence Service
After=network.target

[Service]
Type=simple
ExecStart=/opt/bidlens/bidlens-server --mode http --port 9527 --static-dir /opt/bidlens/dist
Restart=on-failure
RestartSec=5
Environment=RUST_LOG=info

[Install]
WantedBy=default.target
```

**方式二：局域网服务版（多人共用）**

```
一台服务器部署 BidLens
  ├─ 监听 0.0.0.0:9527（需管理员显式配置）
  ├─ 启用用户认证（多用户 Token / LDAP）
  └─ 局域网内其他电脑通过浏览器访问 http://server-ip:9527
```

> **安全提醒**：默认情况下，Web UI 仅绑定 `127.0.0.1`（本机访问）。绑定 `0.0.0.0`（局域网访问）需管理员在配置文件中显式开启，并强制要求用户认证。

## 13.8 技术选型补充

### 13.8.1 HTTP 框架选型

| 决策项 | 选择 | 备选方案 | 选择理由 | 风险与缓解 |
|--------|------|---------|---------|-----------|
| **HTTP 框架** | axum | actix-web / warp | Tokio 官方出品，与现有 Rust 引擎共享 Runtime；类型安全的中间件系统；社区活跃 | 生态相对年轻 → 核心功能稳定 |
| **WebSocket** | axum::extract::ws | tokio-tungstenite | axum 内置，无需额外依赖 | 高级功能有限 → 够用 |
| **静态文件服务** | tower-http::ServeDir | axum::routing::get_service | axum 生态标准方案 | 大文件场景性能待测 |
| **认证中间件** | 自研 axum Layer | tower-http::Auth | 简单 Token 认证无需外部依赖 | 多用户场景需扩展 |
| **请求校验** | axum + serde + zod-rs | validator | 与前端共享 Schema 概念 | Rust 验证库成熟度 |

### 13.8.2 前端适配层技术选型

| 决策项 | 选择 | 备选方案 | 选择理由 | 风险与缓解 |
|--------|------|---------|---------|-----------|
| **HTTP 客户端** | 原生 `fetch` API | axios / ky | 零依赖，浏览器原生支持，足够简单 | 无自动重试 → 自行封装 |
| **WebSocket** | 原生 `WebSocket` API | socket.io-client | 零依赖，浏览器原生支持 | 无自动重连 → 自行实现 |
| **前端构建** | Vite (同一配置, 多入口) | 两套构建 | 最大化代码共享 | 需配置多入口构建 |
| **运行时检测** | `window.bidlens?.isElectron` | 环境变量注入 | 运行时自动切换，构建时无需区分 | 类型需扩展 Window 接口 |

## 本章小结

双模式架构通过**通信层抽象（Transport Adapter）**实现了 Electron 桌面模式与 Web UI 浏览器模式的代码共享，避免了功能分叉。Web UI 模式将渲染引擎的选择权交给操作系统厂商，有效绕过了 Electron 捆绑 Chromium 在国产化环境中的固件/驱动兼容性问题。

**核心收益**：

| 维度 | 收益 |
|------|------|
| 产品覆盖 | 从 3 平台（Win/Mac/Linux 标准）扩展到 7+ 平台（含麒麟/统信/龙芯） |
| 代码共享 | 前端 100% 共享（仅 Transport 层不同），后端 Rust 引擎 100% 复用 |
| 安装包体积 | Web UI 包 ~20MB vs Electron 包 ~150MB |
| 部署灵活性 | 支持单机安装 + 局域网服务两种部署模式 |
| 技术风险 | 通信层抽象使得未来迁移到 Tauri 等框架也仅需新增一个 Adapter |

**核心代价**：

| 维度 | 代价 |
|------|------|
| 开发工作量 | Transport Adapter 抽象层 + Rust HTTP Server 模式 + PWA 配置（预估 2-3 周） |
| 桌面集成 | Web UI 模式缺失原生菜单/托盘/全局快捷键（通过 PWA 部分弥补） |
| 安全模型 | localhost HTTP 的安全等级略低于 Electron IPC（通过 Token+CSRF 缓解） |
