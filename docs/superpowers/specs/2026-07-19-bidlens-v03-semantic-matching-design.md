# BidLens V0.3 本地语义匹配设计

> 日期：2026-07-19
> 状态：V0.3.1 技术设计参考，受正式 PRD 和模型证据门槛约束
> 版本目标：以完全离线、可解释的混合语义匹配提升雷同性审查召回和精排能力

> **适用边界：** 产品范围和版本拆分以 `docs/product/PRD-v0.3-similarity-risk-review.md` 为准。本文只约束 V0.3.1 本地语义增强，不定义 V0.3.0 的基础业务闭环，也不代表模型许可和金标门槛已经通过。

## 1. 背景

BidLens V0.2.2 已形成文件验证、DOCX/文字版 PDF 解析、Rust 比对、审核工作台、历史记录、加密持久化和报告导出的完整桌面工作流。当前 Rust 主链路仍以字符级 Jaccard 相似度和一对一贪心匹配为主，尚未接入真实 Embedding 运行时，也未真正识别段落拆分、合并和移动。

现有规划同时列出 Embedding、OCR、LLM 和语义搜索，但这些能力分别属于语义匹配、文档解析、生成式辅助和知识检索，无法在一个版本内用同一质量指标验收。V0.3 因此收敛为“本地语义匹配”版本。

## 2. 目标与验收口径

V0.3 的目标是：在 Windows 普通办公电脑上，通过完全离线的混合语义匹配，显著提升招投标文档段落对应关系的准确率，并使结果可解释、可降级、可复现。

目标设备：

- Windows 10/11
- 4 核 CPU
- 16GB 内存
- 无独立显卡
- Embedding 相关峰值内存不超过 2GB

核心质量门槛：

- 在固定盲测金标集上，匹配 F1 相比当前 Jaccard 基线提升至少 15 个百分点。
- 明显错误匹配率不高于 5%。
- 完全一致内容的识别能力不得低于现有基线。
- 相同 AST、模型、切分和匹配器版本产生一致结果。
- 完全断网环境能够安装模型并完成比对、复核、历史重开和报告导出。

指标计算口径：

- 一个匹配关系由排序后的 `nodeIdsA`、`nodeIdsB` 和 `matchType` 组成；一对一、拆分、合并和移动均按完整关系计分。
- 总体 Precision、Recall 和 F1 使用盲测集全部匹配关系的 micro average；新增和删除作为无对应关系的分类结果单独报告，不混入匹配关系 F1。
- “明显错误匹配”指人工裁决为语义无关却被系统建立对应关系的结果；错误匹配率以全部预测对应关系为分母。
- 当前基线和 V0.3 必须使用同一份 AST、同一盲测集和同一计算脚本，禁止通过更换解析结果制造提升。

处理耗时在 V0.3 不作为发布阻断项，但必须建立冷缓存和热缓存性能基线。

## 3. 产品范围

### 3.1 范围内

- BGE-M3 官方本地模型的 ONNX CPU 推理。
- 词法、结构、关键事实与 Embedding 的混合匹配。
- 一对一、拆分、合并和移动段落识别。
- 加密 Embedding 持久缓存。
- 官方离线模型包的安装、校验、替换和删除。
- 模型异常时明确提示并降级到传统匹配。
- 匹配依据、模型来源、算法版本和降级原因展示。
- 金标数据集、准确率、资源和离线发布验证。

### 3.2 范围外

- 扫描版 PDF OCR。
- LLM 摘要、风险结论或二次裁决。
- 文档问答式 RAG 和通用语义搜索。
- 自动执行审核任务的 AI Agent。
- 外部模型 API 的生产实现和配置界面。
- 任意第三方 ONNX 模型导入。
- 表格与段落之间的跨结构语义重组。

表格继续使用现有专用表格差异引擎。V0.3 不用生成式模型代替人工审核，也不输出法律或投标合规结论。

## 4. 架构决策

### 4.1 选定方案

Rust 拥有语义匹配算法和统一推理契约；V0.3 交付 Rust ONNX 本地 Provider；未来 Python Sidecar 和云端 Provider 可通过同一契约接入。

```text
React Renderer
    | typed Electron IPC
Electron Main
    |-- TaskOrchestrator
    |-- ModelManager
    |-- EmbeddingCacheRepository
    |-- SQLite Worker + Encryption
    |
    +-- stdio JSON-RPC v2
Rust Engine
    |-- diff-engine
    |-- semantic-matcher
    |-- embedding-core
    +-- embedding-onnx (BGE-M3)
```

边界约束：

- Renderer 只展示状态和结果，不接触模型文件、密钥或 Node API。
- Electron Main 继续拥有文件、设置、密钥、数据库和模型生命周期。
- Rust 负责切分、推理、候选召回、混合评分和匹配冲突消解。
- Rust 不直接访问 SQLite，不持久化外部模型凭据。
- `embedding-core` 不依赖 ONNX、Python 或云端 SDK。
- 原有词法匹配保留为显式降级路径。

### 4.2 未选方案

未选择独立 Python AI 服务作为 V0.3 默认运行时，因为它会增加 Python 环境、进程协议、依赖打包和升级边界。未来需要 Python 模型生态时，应使用独立 Sidecar Provider，而不是把 Python 解释器嵌入 Rust 进程。

未选择 Electron Main 直接执行 ONNX 推理，因为重计算、大内存对象和 Node 原生模块会扩大主进程故障面，并破坏 Rust 计算引擎的现有职责。

云端模型通常通过 HTTP/JSON 接入，不依赖特定 Rust SDK。未来 OpenAI-compatible Provider 可复用统一契约，密钥仍由 Electron `safeStorage` 管理。

## 5. 模块设计

### 5.1 Electron Main

`ModelManager`：

- 安装 `.bidlens-model` 离线模型包。
- 验证受信任清单、签名、SHA-256、模型版本、许可证元数据和运行时兼容性。
- 使用临时目录解包并原子切换版本。
- 保留上一可用模型，安装失败不得破坏现有版本。
- 提供缺失、安装中、可用、无效和不兼容状态。

`EmbeddingCacheRepository`：

- 通过现有 SQLite Worker 查询和写入缓存。
- 使用现有本地密钥体系加密 Float32 向量。
- 管理缓存版本失效、损坏恢复、占用统计和清理。
- 删除相关历史或模型时按保留策略联动清理。

`TaskOrchestrator` 扩展：

- 编排模型准备、向量化、匹配和降级。
- 将真实阶段进度发送给 Renderer。
- 统一处理取消、引擎心跳、重试和错误映射。
- 将实际算法来源写入结果、历史和报告。

### 5.2 Rust Crates

`embedding-core`：

- 定义 `EmbeddingProvider`、批次输入、向量输出、能力和错误。
- 契约包含模型身份、版本、维度、最大 token、批量限制、离线/联网属性、取消和进度。
- Provider 输出必须维持输入顺序，并返回已归一化的固定维度向量。

`embedding-onnx`：

- 实现 BGE-M3 tokenizer 和 ONNX Session。
- 实现动态批处理、L2 归一化、CPU 线程和内存控制。
- 校验输入、输出维度和非有限数值。
- V0.3 同一时间只加载一个认证模型。

`semantic-matcher`：

- 生成稳定语义 Chunk。
- 合并哈希、n-gram 和向量检索候选。
- 计算混合评分和关键事实冲突。
- 执行稀疏全局匹配以及拆分、合并和移动识别。
- 输出可供 `diff-engine` 消费的匹配关系和证据。

`diff-engine`：

- 消费最终匹配关系并生成统一 `DiffAst`。
- 保留局部文字差异和现有表格比对能力。
- 不直接依赖具体模型运行时。

## 6. 跨层契约

共享类型增加以下概念，实际字段名在实施计划中以兼容现有类型为准：

```typescript
interface ComparisonProvenance {
  mode: 'hybrid_ai' | 'lexical_fallback';
  matcherVersion: string;
  modelId: string | null;
  modelVersion: string | null;
  semanticCoverage: number;
  degradedReason: string | null;
}

interface MatchEvidence {
  semanticSimilarity: number | null;
  lexicalSimilarity: number;
  structuralScore: number;
  criticalFactConflicts: CriticalFactConflict[];
}

interface ModelStatus {
  state: 'missing' | 'installing' | 'ready' | 'invalid' | 'incompatible';
  modelId: string;
  version: string | null;
}
```

每个差异项保存匹配依据；任务结果保存模型、匹配器、语义覆盖率和降级原因。历史快照重新打开时不重新推理，也不得将旧模型结果显示成当前模型结果。

Rust 协议升级为双向 JSON-RPC v2。Rust 可在比对请求仍进行时向 Electron 批量发起缓存查询和写入请求。向量采用 Base64 编码的 Little-Endian Float32 二进制块，禁止使用 JSON 数字数组。所有请求均支持任务 ID、批次 ID、取消和结构化错误。

Renderer 导入新增类型时必须继续使用 `@bidlens/shared/types-only`。

## 7. 数据流

```text
验证模型
-> 解析两份 DocumentAst
-> 生成稳定 Chunk
-> 查询加密向量缓存
-> 对缺失 Chunk 批量推理
-> 构建候选集合
-> 混合评分
-> 全局一对一匹配
-> 拆分/合并补偿匹配
-> 移动识别
-> 生成 DiffAst 与匹配依据
-> 加密保存结果和新向量
```

Chunk 以段落和列表项为基本单位，保留节点 ID、章节路径、页码和文档顺序。章节标题作为模型上下文，但不改变原始正文。超长段落按固定 token 上限切分，向量结果聚合回原节点，保证 UI 可定位原文。

缓存键至少包含：

```text
documentHash
+ nodeTextHash
+ sectionContextHash
+ modelId/version
+ tokenizerVersion
+ chunkingVersion
+ normalizationVersion
```

模型、上下文、切分或归一化规则变化时，旧向量不得复用。

## 8. 混合匹配算法

### 8.1 候选召回

候选集合由三路结果合并：

- 完全一致文本哈希。
- n-gram/词法索引。
- BGE-M3 向量 Top-K 检索。

大文档使用近似向量索引或分层检索，禁止无条件构造完整的 `N x M` 相似度矩阵。完全一致结果优先锁定，减少不必要的推理和冲突。

### 8.2 混合评分

```text
综合分数 =
  语义相似度
  + 词法相似度
  + 章节/位置结构分
  + 精确一致奖励
  - 关键事实冲突惩罚
```

关键事实至少覆盖金额、百分比、日期、期限、编号、否定词和资质等级。冲突用于降低置信度和生成证据，不直接生成风险或合规结论。

混合权重通过金标校准，不向普通用户开放。宽松、平衡、严格仅映射到经过验证的阈值组合。

### 8.3 匹配关系

1. 锁定完全一致和高置信度一对一关系。
2. 在稀疏候选图上执行确定性的全局冲突消解。
3. 对未匹配节点检查相邻 2-4 段组合，识别拆分和合并。
4. 根据最终节点顺序和章节路径识别移动。

所有关系必须保留来源节点列表和匹配证据。V0.3 不执行段落与表格之间的跨结构重组。

## 9. 任务状态、资源和降级

任务阶段扩展为：

```text
validating
-> parsing_baseline
-> parsing_review
-> preparing_model
-> embedding
-> matching
-> finalizing
-> ready
```

由于 V0.3 不设置固定耗时 SLA，现有固定总超时应改为“持续心跳 + 阶段无响应超时”。只要引擎持续报告有效进度，长任务就不应被误杀。

资源规则：

- 默认保留一个逻辑核心给 Electron 和操作系统。
- 批次大小根据可用内存动态调整。
- 内存压力升高时先缩小批次并重试一次。
- 推理、索引和组合匹配循环必须检查取消信号。
- 模型 Session 可在任务间复用，空闲后允许卸载。

错误处理：

| 错误 | 行为 |
|---|---|
| 缓存缺失 | 重新计算 |
| 单条缓存损坏 | 删除该条并重新计算 |
| 模型包缺失 | 明确提示，可继续传统比对 |
| 签名或哈希失败 | 禁止加载，标记无效 |
| 模型不兼容 | 禁止加载，提示安装兼容版本 |
| 推理内存不足 | 缩小批次重试一次，失败后降级 |
| 空白或不可编码节点 | 该节点仅使用词法特征，降低语义覆盖率 |
| Provider 崩溃或维度错误 | 丢弃本次语义结果，重新执行传统匹配 |
| 引擎无心跳或异常退出 | 任务失败并保存诊断，不伪造降级成功 |

Provider 发生致命错误时不得混用部分语义结果。降级任务必须记录 `lexical_fallback`、原因、模型状态、语义覆盖率和匹配器版本，并在处理页、结果页、历史和报告中持续可见。

## 10. UI/UX 设计

V0.3 UI 必须在设计和施工阶段使用 `ui-ux-pro-max`。设计方向采用 Swiss Minimalism、Data-Dense Dashboard 和 Comparative Analysis，延续现有审核工作台，不创建营销式 AI 页面，也不引入第二套设计系统。

### 10.1 设置

“本地语义模型”区域展示：

- 模型状态、名称、版本、体积和许可证摘要。
- 从本地 `.bidlens-model` 文件安装。
- 重新校验、替换和删除。
- Embedding 缓存占用及单独清理命令。

不展示 API Key、模型切换或任意 ONNX 导入。

### 10.2 新建与处理

- 模型可用时显示“本地语义增强已就绪”。
- 模型不可用时显示明确警告，但允许开始传统比对。
- 保留现有宽松、平衡和严格敏感度，不增加 AI 开关。
- 处理页展示用户可理解的阶段、进度和取消状态。
- 模型加载、缓存命中和向量化详情按需展开，不挤占主进度区域。

### 10.3 结果与解释

- 主列表展示匹配类型、综合置信度和关键事实冲突标记。
- 详情面板使用现有 Radix Tabs 增加“匹配依据”。
- 匹配依据展示语义、词法、结构、事实冲突和关联节点。
- 拆分/合并作为一个逻辑差异项，可展开多个来源段落。
- 移动结果显示原章节和新章节。
- 工具栏持续显示“本地语义增强”“传统比对”或“语义覆盖不完整”。

### 10.4 UI 质量约束

- 使用现有设计令牌、Inter、8px 间距体系和 Lucide 图标。
- AI 状态必须同时使用图标、短文本和状态色。
- 控件和加载状态保持稳定尺寸，不产生布局跳动。
- 动效限制在 150-250ms 的颜色或透明度变化，并尊重 `prefers-reduced-motion`。
- 错误使用 `role="alert"` 或 `aria-live`；弹窗关闭后焦点回到触发控件。
- 验证浅色、深色、高对比、键盘和屏幕阅读器状态。
- Playwright 截图覆盖 1024x768、1280x800、1440x900 和 1920x1080。

施工前先更新 `apps/desktop/UI-SPEC.md`，然后再修改组件。UI 变更同步更新 Desktop 层文档和测试。

## 11. Agent/RAG 未来兼容性

V0.3 只在设计层考虑未来 Agent/RAG，不实现功能、不增加任何 Agent/RAG 专用 IPC，也不纳入验收：

- V0.3 的向量召回称为语义候选检索，不称为 RAG。
- Chunk 和匹配证据保留稳定来源信息，未来可作为 RAG 引用。
- 匹配引擎保持确定性，不引入 Agent 自主决策。
- 未来 Agent 只能通过受控工具读取差异、检索证据和生成建议。
- Agent 不得绕过审核流程直接修改差异事实或人工审核状态。

LLM、知识库、提示词、Agent 工具协议和问答 UI 均不进入 V0.3 实施计划。

## 12. 金标集和验收测试

金标集最低要求：

- 至少 30 组真实或脱敏招投标文档对。
- 至少 3,000 条人工标注关系。
- 覆盖一对一、拆分、合并、移动、新增、删除和关键事实冲突。
- 两名标注者独立标注，分歧经过裁决。
- 按文档对划分校准集和盲测集，禁止内容泄漏。
- 测试数据必须拥有内部使用许可，客户敏感文档不得提交到仓库。

测试层次：

- Rust 单元测试：切分、归一化、Top-K、评分、事实提取和复杂匹配。
- Provider 合约测试：维度、顺序、归一化、取消、超时和错误。
- 跨语言契约测试：字段映射、双向 JSON-RPC 和向量编码。
- 缓存测试：加密读写、版本失效、损坏恢复和联动清理。
- 集成测试：模型缺失、损坏、OOM、Provider 崩溃、重启和降级。
- E2E：断网模型安装、比对、复核、历史重开和导出。
- UI：按 `ui-ux-pro-max` 约束执行截图、焦点、键盘、对比度和布局检查。
- 发布：干净 Windows 环境安装应用和离线模型包，确认不存在隐式网络请求。

性能报告记录解析、向量化、索引、匹配和最终化耗时、缓存命中率、CPU 峰值、Rust RSS 和整机内存。记录 200 页及 1000 页文档对的冷/热缓存 P50/P95 数据。

## 13. 实施阶段

### Phase 0：可行性与证据门槛

- 审查 BGE-M3、tokenizer、ONNX Runtime 和转换产物的商业许可。
- 在目标 Windows CPU 环境验证最小 ONNX 原型。
- 验证中文效果、内存、维度、归一化和离线打包。
- 建立 Jaccard 基线和首版金标集。
- 任一硬门槛失败时暂停并重新决策模型，不静默替换。

### Phase 1：合约与模型生命周期

- 定义 Provider、模型清单、状态、错误和结果溯源契约。
- 设计并实现 JSON-RPC v2 双向通信。
- 实现官方离线模型包生命周期。
- 扩展设置、数据库迁移和 API 文档。

### Phase 2：本地推理与加密缓存

- 建立 `embedding-core` 和 `embedding-onnx`。
- 实现 tokenizer、批处理、归一化和资源控制。
- 实现加密向量缓存和双向缓存 RPC。
- 完成取消、崩溃、损坏和离线测试。

### Phase 3：混合一对一匹配

- 实现稳定 Chunk、三路召回和 Top-K。
- 实现混合评分和关键事实冲突。
