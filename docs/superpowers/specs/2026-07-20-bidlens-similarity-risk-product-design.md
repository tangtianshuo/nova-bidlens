# BidLens 多投标文件雷同性风险审查产品设计

> 日期：2026-07-20｜状态：历史讨论稿
> 主产品模式：多份投标文件的雷同性风险审查｜辅助产品模式：双文档版本差异比对

> **权威文档变更：** 本文已由 `docs/product/PRD-v0.3-similarity-risk-review.md` 取代，不再单独作为实施和验收依据。本文保留用于追溯产品调研和早期设计决策；发生冲突时以正式 PRD 为准。

## 1. 背景与调研结论

BidLens V0.2.2 已具备 DOCX/文字版 PDF 解析、双文档差异计算、表格/格式/批注/修订展示、人工标记、历史记录、加密持久化和报告导出能力。V0.3 正在建设完全离线、可解释的本地语义匹配，但当前产品对象仍是“基准文档 + 审查文档”。

2026-07-20 使用 Playwright 对参考产品“多版本文件比对系统”完成游客态、登录态、历史记录和真实结果页调研。调研确认其核心不是版本 Diff，而是把多家投标文件放在同一项目中，围绕资信、技术、经济和设备特征形成项目风险、文件关系矩阵和可追溯证据。

BidLens 的最终定位因此调整为：

> 本地化、可解释的投标文件雷同性风险审查工作台。

现有双文档比对不被废弃，而是降为辅助产品模式，并作为雷同性发现项的两两证据复核能力继续使用。

## 2. 目标用户与产品责任

### 2.1 首要用户

- 招标人；
- 招标代理机构；
- 审计和复核人员。

投标人内部自查不进入首版主场景，可在后续版本作为次级模式评估。

### 2.2 产品责任边界

BidLens 负责发现、组织和解释雷同性风险线索，不负责认定串标、围标或其他法律结论。

- 系统可以输出低、中、高三级风险；
- 系统不得把相似度包装为“串标概率”；
- 所有风险必须能追溯到原文件、AST 节点、原文和检测依据；
- 报告必须声明结果仅作为人工审查线索；
- 人工裁决不得覆盖或修改原始检测证据。

## 3. 产品模式

### 3.1 雷同性风险审查（默认主模式）

以招标项目为单位，导入可选招标基线和 2-8 份投标文件，执行跨文件语义、表格和实体检测，输出项目风险、文件关系和证据。

### 3.2 版本差异比对（辅助模式）

保留现有“基准文档 + 审查文档”工作流，继续支持正文、表格、格式、批注和修订差异。该模式不输出项目级雷同性风险。

两个模式共享解析器、Document AST、模型运行时、加密存储、证据审阅组件和导出基础设施，不建设两套割裂系统。

## 4. 首版范围

### 4.1 范围内

- 2-8 份投标文件的项目级分析；
- 0-1 份招标基线文件；
- 文本语义雷同和改写检测；
- 表格文字雷同检测；
- 人名、身份证、手机号、邮箱、统一社会信用代码等关键实体重复；
- 招标公共内容过滤；
- 本地 BGE-M3 ONNX Embedding；
- 项目内向量 Top-K 候选召回；
- 词法、语义、结构和实体混合精排；
- 项目风险、文件对风险、关系矩阵和发现项；
- 人工确认、待确认、忽略、重要标记和备注；
- PDF、HTML 和辅助 Markdown 报告；
- 完全离线运行、加密持久化、取消和恢复。

### 4.2 范围外

- 图片指纹查重；
- 图片文字 OCR；
- 扫描版 PDF；
- 设备和文档硬件指纹；
- 经济标清单、定额和人材机规则；
- 旧版 `.doc`；
- 加密文件和专有投标文件包；
- 问答式 RAG；
- LLM 参与候选检测、相似度计算或风险评级；
- 云端账号、次数权益和在线处理；
- 局域网私有化服务端（仅预留边界）。

图片/OCR、设备指纹和经济标属于不同技术体系，后续分别作为独立阶段设计，不能阻塞首版文本主链。

## 5. 输入与容量约束

### 5.1 支持格式

投标文件和招标基线首版均仅支持：

- `.docx`；
- 文字版 `.pdf`。

不支持、加密、扫描版或损坏文件必须在分析前明确拦截并说明原因。未来格式通过现有解析器适配器扩展。

### 5.2 项目容量

- 每个项目 2-8 份投标文件；
- 可附加 1 份招标基线；
- 单份文件最多 1000 页；
- 项目累计最多 4000 页；
- 目标设备为 Windows 10/11、4 核 CPU、16GB 内存、无独立显卡。

超限必须在分析前拦截，不允许在处理阶段因资源耗尽崩溃。

## 6. 招标基线策略

招标基线可选但强烈推荐。

- 提供时：生成公共内容指纹，用于过滤投标文件引用的招标原文并降低误报；
- 未提供时：允许分析，但新建、处理、结果、历史和报告必须持续显示误报风险警告；
- 基线解析失败时：允许用户明确确认后切换到无基线模式；
- 后续经济标等依赖基线的检测器必须把基线设为强制前置条件。

被过滤内容不删除，而是以“已过滤证据”保存并允许审阅。

## 7. 领域模型

### 7.1 AnalysisProject

一个招标项目及其完整分析上下文：

- 项目 ID、名称和时间；
- 模式、检测预设和状态；
- 招标基线引用；
- Submission 列表；
- 模型、解析器、切分器、匹配器和规则版本；
- 完整、降级或部分分析状态；
- 项目级 RiskAssessment。

### 7.2 Submission

一份投标文件及其处理状态：

- 文件 ID、名称、Hash、格式、页数和大小；
- Document AST 引用；
- 解析、过滤、向量化和检测状态；
- 能力、警告和结构化错误。

### 7.3 RiskFinding

一条合并后的风险发现：

- 检测类型和风险等级；
- 涉及 Submission 和 Evidence；
- 方向性覆盖率、对称相似度和综合置信度；
- 原始算法分和贡献因子；
- 规则版本；
- 人工审阅状态、重要标记和备注。

### 7.4 Evidence

RiskFinding 的可追溯依据：

- 文件和 AST 节点；
- 原始值、规范化值、文本片段和上下文；
- 词法、语义、结构或实体匹配依据；
- 招标过滤状态和原因；
- 可选 DiffAst 两两差异明细。

### 7.5 RiskAssessment

文件对或项目级风险评估：

- 低、中、高风险或“不完整”；
- 原始规则分；
- 主要贡献发现项；
- 检测预设和规则版本；
- 完整、降级或部分分析状态。

现有 DiffAst 保持为两份文档间的差异协议，不承担项目级风险结果。

## 8. 总体架构

现有 React Renderer、Electron Main、Rust Engine 三层架构保持不变。

### 8.1 Shared 契约层

新增项目、文件、检测配置、进度、发现、证据和风险类型。DocumentAst 和 DiffAst 保持兼容，Renderer 继续仅从 `@bidlens/shared/types-only` 导入。

### 8.2 Electron Main 编排层

新增项目分析编排器，负责：

- 文件验证和容量检查；
- Node 解析器调度；
- Document AST 和向量缓存；
- Rust 项目分析请求和进度转发；
- 取消、恢复和失败处理；
- SQLite 加密持久化；
- 历史、清理和报告导出。

### 8.3 Rust 分析层

Rust 新增职责单一的独立组件：

- `tender-filter`：招标公共内容识别；
- `semantic-detector`：文本语义雷同和改写检测；
- `table-detector`：表格文字和结构化行内容检测；
- `entity-detector`：关键实体提取、归一化和重复检测；
- `finding-aggregator`：跨检测器证据合并和去重；
- `risk-engine`：版本化确定性风险规则。

检测器只产生候选发现与证据，不直接生成法律或业务定性。

### 8.4 React Renderer

Renderer 提供项目列表、新建项目、处理进度、项目概览、文件关系矩阵、发现项列表、证据复核和报告导出。现有三栏审阅工作台复用为两两证据复核视图。

## 9. 语义检索与匹配链路

首版语义链路为：

```text
Document AST
  -> structure-aware Chunk
  -> local BGE-M3 ONNX Embedding
  -> encrypted project vector index
  -> cross-submission Top-K retrieval
  -> lexical/semantic/structure/entity reranking
  -> RiskFinding candidates
  -> deterministic aggregation and risk rules
```

### 9.1 Embedding

- BGE-M3 作为首版官方本地模型；
- 模型包必须校验清单、签名、SHA-256、版本和许可证元数据；
- 向量缓存键包含文档 Hash、解析器版本、切分版本和模型版本；
- 模型不可用时允许用户选择传统匹配，但必须显式记录降级。

### 9.2 向量召回

向量索引只覆盖当前项目，禁止把不同项目的文档内容混入召回。召回目标是为每个 Chunk 找到其他 Submission 的 Top-K 相似候选；同一 Submission 内的候选不形成跨主体风险。

该能力是跨文档语义候选召回，不是面向问答或生成的完整 RAG。

### 9.3 LLM 边界

LLM 不进入首版检测和评级主链。未来可作为独立的“证据辅助解释”能力，基于已确认 Evidence 生成摘要或报告草稿，但必须：

- 引用具体 Evidence；
- 不新增检测证据；
- 不修改相似度；
- 不覆盖风险评级；
- 失败时不影响原始结果。

## 10. 检测语义

### 10.1 文本

- 同时保留 A 到 B、B 到 A 的方向性覆盖率；
- 另给出用于关系矩阵的对称相似度；
- 语义、词法、结构和关键事实共同参与精排；
- 相同输入和版本必须产生相同结果。

### 10.2 表格

- 按表、行和关键单元格规范化；
- 忽略纯样式差异；
- 同一表格重复行不得重复累计风险；
- 保留表格位置和单元格证据。

### 10.3 实体

- 先处理空白、大小写、全半角和常见格式差异；
- 身份证、手机号、邮箱和信用代码使用规范化精确匹配；
- 人名等弱实体必须结合章节和邻近上下文，不能仅凭同名判高风险。

### 10.4 聚合去重

同一原文被多个检测器命中时合并为一个 RiskFinding，同时保留全部检测依据，防止风险数量虚高。

## 11. 风险规则与检测预设

风险引擎使用公开、可版本化的确定性规则。概念计算式为：

```text
finding weight
  x evidence strength
  x involved submission count
  x cross-section rarity
  - tender common-content discount
  - common-template discount
```

项目风险由高价值发现项和文件关系网络综合计算，不能只按发现数量累加。

首版提供三个固定预设：

- 严格：提高判定阈值，偏向高精度和较少误报；
- 标准：默认，平衡漏检和误报；
- 宽松：降低判定阈值，偏向高召回并产生更多待人工确认项。

预设对应版本化固定参数，语义方向必须与现有 `strict > standard > loose` 阈值顺序一致。首版不开放逐项滑块或任意阈值编辑，报告记录预设和规则版本。

人工确认、待确认或忽略只改变审阅状态和报告视图，不回写原始算法分。

## 12. 用户工作流

应用启动后直接进入项目列表。顶栏提供“雷同性审查”和“版本差异比对”，前者为默认入口。

### 12.1 新建项目

1. 填写项目名称；
2. 添加 2-8 份投标文件；
3. 可选添加招标基线；
4. 完成格式、页数、容量和解析能力检查；
5. 选择严格、标准或宽松预设；
6. 确认并开始分析。

### 12.2 处理阶段

```text
文件校验
  -> 文档解析
  -> 招标内容过滤
  -> 向量化
  -> 候选召回
  -> 混合精排
  -> 实体/表格检测
  -> 风险聚合
  -> 保存结果
```

每份文件显示独立状态，项目显示总体阶段、耗时、警告和取消命令。

### 12.3 结果信息架构

- 项目风险等级、预设、模型和规则版本；
- 文本、表格、实体风险摘要；
- 投标文件关系矩阵；
- 可筛选的 RiskFinding 列表；
- 两两 Evidence 复核工作台；
- 报告导出和历史重开。

项目概览必须区分算法原始发现数和人工确认风险数。

## 13. 状态、持久化与恢复

主要检查点为：

```text
validated
parsed
filtered
embedded
retrieved
detected
aggregated
ready
```

每个主要阶段完成后加密持久化。异常退出后项目标记为“已中断”，用户可从最近完整检查点恢复。完成项目从快照重开，不重新推理，也不能把旧模型结果显示为当前模型结果。

## 14. 错误与降级

- 文件错误在启动前拦截；
- 单份投标文件解析失败时暂停整个项目，等待移除或替换；
- 招标基线失败可经用户确认切换到无基线模式；
- 模型缺失可选择传统匹配，但所有页面和报告持续显示降级状态；
- 单个检测器失败时不能把部分结果包装成完整分析；
- 用户明确接受部分分析后，RiskAssessment 必须标记“不完整”，不得显示正常低风险；
- 取消任务保留可复用缓存，但不生成风险报告；
- 原文件丢失时历史证据仍可读，重新分析必须定位文件并校验 Hash。

## 15. 数据安全与清理

- 原文件、AST、Embedding、发现、裁决和报告均不离开本机；
- 继续使用 SQLite Worker、AES-256-GCM 和 Windows `safeStorage`；
- Embedding 缓存必须加密；
- 支持项目删除、保留上限和永久保留；
- 项目删除同步清理孤立 AST、向量和报告；
- 共享缓存只在没有其他项目引用时删除；
- 首版不要求账号或联网。

## 16. 报告

### 16.1 格式

- PDF：归档、流转和签批的主格式；
- HTML：本机交互查看完整证据；
- Markdown：兼容现有能力的辅助格式。

### 16.2 必含内容

- 项目范围和文件清单；
- 招标基线状态；
- 检测预设；
- 模型、规则、解析器和匹配器版本；
- 完整、降级或部分分析状态；
- 项目和文件对风险；
- 文件关系矩阵；
- RiskFinding、Evidence 和人工裁决；
- “仅供审查参考，不构成串标认定”的声明。

报告不得只输出不可追溯的摘要结论。

## 17. 质量门槛

- 固定金标集上，混合语义匹配 F1 比当前 Jaccard 基线提升至少 15 个百分点；
- 明显错误匹配率不高于 5%；
- 向量候选召回对金标关系的 Recall@TopK 不低于 95%；
- 身份证、手机号、邮箱和信用代码的规范化精确匹配不得产生算法误配；
- 完全一致内容识别能力不得低于现有基线；
- 相同文件、模型、规则和配置必须产生一致结果；
- 累计 4000 页项目在目标设备上不得 OOM；
- 处理进度必须持续可见，取消必须可响应。

## 18. 测试策略

- 单元测试：归一化、过滤、检测器、聚合和风险规则；
- 契约测试：Shared、Electron IPC、Rust JSON-RPC 和数据库迁移；
- 金标测试：文本改写、招标公共内容、表格和实体；
- 集成测试：2-8 文件离线分析、取消、恢复、历史重开和导出；
- E2E：雷同性主模式和版本差异辅助模式；
- 性能测试：目标容量、内存、缓存和取消；
- 安全测试：断网、加密、删除闭环和报告范围；
- 回归测试：现有双文档能力不得因主模式升级失效。

发布前必须人工复核金标集，确认高风险发现具备可理解证据，不能只依赖自动指标。

## 19. 选型结论

已比较三种总体路线：

1. 在现有架构上演进统一核心；
2. 并行新增独立雷同性子系统；
3. 按新定位整体重构。

选择路线 1。它最大限度复用现有解析、Diff、审阅、持久化和报告能力，同时通过新的项目级领域模型避免风险语义污染 DiffAst。路线 2 会形成重复任务和历史体系，路线 3 的交付风险与现阶段不匹配。

## 20. 后续能力顺序

首版主链稳定后，再分别设计：

1. 图片指纹和图片 OCR；
2. 文件与设备特征取证；
3. 经济标清单规则；
4. LLM 证据辅助解释；
5. 局域网私有化部署。

这些能力必须作为独立检测器或适配器接入统一 RiskFinding/Evidence 协议，不能破坏首版可解释、可复现的核心链路。
