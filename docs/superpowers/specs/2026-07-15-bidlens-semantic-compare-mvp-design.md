# Nova BidLens 语义比对 MVP 设计

> 日期：2026-07-15
> 状态：已讨论确认，待用户审阅
> 范围：AI 文档智能比对 MVP

## 1. 背景与目标

Nova BidLens 当前已有较完整的软件设计说明书，覆盖 Electron + React 前端、Node.js 文档解析层、Rust 核心引擎、SQLite 缓存、本地/外部 AI 双模式、插件化与国产化双模式等方向。

本轮规划将第一阶段范围明确收敛为“AI 文档智能比对”，暂不实现招标文件响应性审查、单份标书合规审查、风险规则库、资质/签章/报价/工期校验等标书审查能力。这些审查方向作为后续阶段规划，依赖当前比对产品沉淀的 Document AST、Diff AST、人工标记与结构化报告模型。

MVP 的核心目标是：让用户能在两份结构可能不同的标书之间快速看到“哪些内容对应、哪里变了、哪些内容移动/新增/删除”，并能导出可供人工审核复核的报告素材。

## 2. 产品边界

### 2.1 MVP 必做

- 支持两份 Word/docx 文档比对，定位为旧版/基准版与新版/比较版。
- 解析为 Document AST，第一阶段稳定覆盖章节、段落、列表、基础表格。
- 基于 Chunk + Embedding 做语义匹配，不依赖章节顺序。
- 识别新增、删除、修改、移动、拆分、合并、不确定匹配。
- 默认展示段落级差异，必要时展开局部字级差异。
- 支持 500-1000 页标书任务的进度展示、取消、错误恢复与结果审阅。
- 导出包含差异摘要、位置、原文片段、置信度与人工标记的报告素材。

### 2.2 MVP 不做

- 不做招标文件 vs 投标文件的响应性审查。
- 不做资质、签章、报价、工期、目录完整性等规则库审查。
- 不追求第一阶段 L5 全保真覆盖批注、修订、复杂版式。
- 不在第一阶段开放外部 Workflow 插件市场。
- 不让 LLM 直接生成不可追溯的审查结论。
- 不做像素级 Word 版式复刻，也不在结果页编辑原文。

## 3. 推荐路线

本轮讨论比较了三种路线：

- 语义比对 MVP：先打穿结构重排、段落改写、拆分合并、增删改和大文档性能。
- 全保真优先：优先处理内容、格式、表格、批注、修订的全面差异。
- 平台预埋优先：优先建设 Core API、Workflow 插件与后续审查扩展框架。

确认选择“语义比对 MVP”作为当前主线。全保真与平台化能力保留接口兼容，不进入 MVP 核心验收。

## 4. 核心数据流

MVP 主链路沿用现有总体架构，但叙述重心收敛到语义比对闭环：

```text
React UI
  -> Electron Main
  -> Node Parser
  -> Document AST
  -> Rust Chunk / Embedding / Match / Diff
  -> Diff AST
  -> React Review Workbench / Export
```

模块职责如下：

- React UI：文件选择、比对参数配置、进度展示、差异审阅、人工标记、报告导出。
- Electron Main：任务编排、IPC 路由、文件权限控制、Node Parser 与 Rust Engine 调度。
- Node Parser：docx 解析，输出 Document AST，MVP 稳定覆盖章节、段落、列表、基础表格。
- Rust Engine：执行 Chunk、Embedding、候选召回、冲突消解、局部 Diff、Diff AST 生成。
- SQLite Cache：缓存 Document AST、Embedding、比对历史、任务配置与人工处理状态。

## 5. 关键数据对象

### 5.1 Document AST

Document AST 是所有后续能力的基础结构。MVP 不要求 L5 全保真完整落地，但必须稳定输出：

- 文档元数据：文件名、hash、页数、字数、解析器版本。
- 章节与段落层级：用于 chunk、定位与结果导航。
- 列表结构：用于常见条款、评分项、要求项的上下文保留。
- 基础表格：用于报价、清单、参数表的初步比对。
- 节点 ID：所有结果必须能追溯到原始 AST 节点。

### 5.2 Chunk

Chunk 是 Embedding 与语义匹配的最小处理单元。MVP 采用章节优先、段落兜底的切分策略：

- 优先按章节、标题、自然段边界切分。
- 超长章节按段落继续拆分。
- 表格可作为独立 chunk 或按行/单元格分块，具体取决于表格规模。
- 每个 chunk 必须包含 nodeIds、原文文本、token 数、章节标题、页码范围。

### 5.3 Diff AST

Diff AST 是 UI 与报告导出的唯一结果协议，不直接夹带原始文档全文。建议稳定字段包括：

- `matchId`
- `matchType`
- `confidence`
- `similarity`
- `sourceA`
- `sourceB`
- `nodeIdsA`
- `nodeIdsB`
- `diffDetail`
- `summary`
- `reviewAnnotationId`

人工复核状态不直接写入算法结果。Diff AST 只保存 `reviewAnnotationId` 或可关联键，人工标记作为独立 review annotations 保存，避免污染算法结果。

## 6. 语义匹配算法

MVP 算法采用可解释的漏斗式流水线：

```text
AST 规范化
  -> Chunk 切分
  -> Embedding 向量化
  -> Top-K 候选召回
  -> 标题/编号/邻域增强
  -> 冲突消解
  -> 段落级与局部字级 Diff
  -> Diff AST
```

### 6.1 候选召回

- 对文档 A 与文档 B 的 chunk 分别生成 embedding。
- 对每个 A chunk 检索 B 的 Top-K 候选。
- 按模式使用不同阈值：快速模式偏召回，标准模式平衡，精确模式提高阈值并启用更多增强。
- Embedding 结果按 `doc_hash + model_name + chunk_index` 缓存。

### 6.2 候选增强

单纯向量相似度不足以稳定处理标书结构。MVP 需引入轻量增强因子：

- 标题相似度：章节标题相同或相近时加权。
- 编号相似度：如 `1.2.3`、`第二章` 等编号结构相近时加权。
- 邻域一致性：相邻 chunk 也匹配时提高置信度。
- 文本长度比：长度差异极端时降低置信度。

### 6.3 冲突消解

候选集合中可能出现一对多、多对一、多候选冲突：

- 默认使用相似度优先的贪心匹配。
- 对密集冲突区域可使用局部最优匹配策略。
- 对拆分、合并场景保留 1:N 与 N:1 结果，不强行压成 1:1。
- 低置信候选进入 `uncertain`，交给人工确认。

### 6.4 结果分类

MVP 结果类型固定为：

- `identical`：内容基本一致，可默认折叠。
- `modified`：语义对应但内容有变化。
- `added`：仅出现在 B 文档。
- `deleted`：仅出现在 A 文档。
- `moved`：语义匹配但位置偏移明显。
- `split`：A 的一段在 B 中拆成多段。
- `merged`：A 的多段在 B 中合并为一段。
- `uncertain`：低置信候选，需人工确认。

UI 不直接依赖向量分数，而依赖 Diff AST 中的 `matchType`、`confidence`、节点引用和差异详情。

## 7. 前端结果工作台

MVP 结果页采用三栏复核工作台，而不是完整 Word 版式复刻。

### 7.1 左栏：差异导航

- 展示统计摘要：修改、新增、删除、移动、拆分、合并、不确定。
- 支持类型筛选、置信度筛选、章节筛选。
- 默认按需关注程度排序：删除/新增、低置信、移动、修改。
- 点击差异项后，中栏跳转到对应原文位置。

### 7.2 中栏：双文档对照

- 展示 A/B 文档对应段落。
- 高亮当前差异。
- 支持同步定位与锚点跳转。
- 对移动内容显示来源章节与目标章节。
- 段落级默认展示，点击后展开局部字级 diff。
- 长文档必须使用虚拟滚动与必要的空白占位对齐。

### 7.3 右栏：差异详情

- 展示 `matchType`、`confidence`、`similarity`、摘要、节点引用。
- 支持人工标记：重要、忽略、待确认。
- 对 `uncertain` 支持确认匹配或拆分为新增/删除。

### 7.4 导出

导出不直接拼接 UI HTML，而应基于结构化 export model 生成报告。报告至少包含：

- 文档 A/B 元数据。
- 比对配置和模型版本。
- 差异统计。
- 差异列表。
- 原文片段。
- 置信度与差异类型。
- 用户人工标记。

## 8. LLM 使用边界

MVP 中 LLM 只作为可选辅助，不作为硬判定来源：

- 可用于 Top-N 关键差异的自然语言摘要。
- 可用于解释 `modified` 差异的变化点。
- 可用于报告中的“摘要说明”草稿。
- 不改变基础匹配结果。
- 不输出不可追溯的审查结论。

## 9. 错误处理与性能约束

### 9.1 错误处理

- 文档解析失败时返回具体阶段与文件名。
- 模型不可用时提示用户切换外部 API 或下载本地模型。
- 外部 API 失败时按配置降级本地模型或中止任务。
- 比对任务支持取消，取消后保存可复用的中间缓存。
- Diff AST 生成后必须校验节点引用完整性。

### 9.2 性能约束

- 500-1000 页文档必须有阶段进度反馈。
- AST、Embedding 与任务结果应缓存，重复比对不应全量重算。
- 相似度矩阵可分块计算，避免大文档内存峰值失控。
- 前端不渲染全量 DOM，差异列表与双栏文档均使用虚拟滚动。

## 10. 后续阶段路线

### 10.1 V0.1：语义比对 MVP

- docx 双文档语义比对。
- AST / Chunk / Embedding 主链路。
- 移动、拆分、合并、增删改识别。
- 三栏复核工作台。
- Markdown/HTML 报告导出。

### 10.2 V0.2：全保真增强

- 表格差异深化。
- 格式差异分层。
- 批注与修订接入。
- PDF 文本版支持。
- 报告模板化。

### 10.3 V0.3+：标书审查规划

- 招标文件 vs 投标文件响应性审查。
- 单标书规则审查。
- 风险规则库。
- Workflow 插件。
- 多项目审查台账。

后续审查能力应读取 Document AST、Diff AST、人工标记和报告模型，不应侵入或拖慢语义比对核心引擎。

## 11. 验收标准

- 用户能选择两份 docx 文档并启动比对。
- 系统能展示解析、向量化、匹配、Diff 生成等阶段进度。
- 系统能识别新增、删除、修改、移动、拆分、合并、不确定匹配。
- 用户能按类型、置信度、章节筛选差异。
- 用户能点击差异定位到 A/B 对照原文。
- 用户能对差异做重要、忽略、待确认等人工标记。
- 用户能导出包含差异摘要、位置、原文片段和人工标记的报告。
- 大文档结果页保持可交互，不因全量 DOM 渲染卡死。

## 12. V0.1 明确决策

- V0.1 只正式支持 docx 双文档比对，不保留 PDF 文本版实验入口，避免解析链路扩大。
- Embedding 默认优先本地模型；外部 OpenAI-compatible API 作为用户显式配置后的可选加速或备用方案。
- V0.1 不引入 rerank 模型。精确模式只调整阈值、Top-K、邻域增强和局部 Diff 策略。
- 报告首版输出 Markdown 与 HTML。PDF 导出放入 V0.2 报告模板化阶段。
