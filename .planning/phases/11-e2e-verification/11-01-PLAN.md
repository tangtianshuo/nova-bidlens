**Part D: DocumentAst → Rust 引擎全链路**

此测试验证 DocumentAst 经 toEngineDocumentAst 转换后能被 Rust 引擎处理。

1. 从 Task 1 的 mapper 测试中复用已构建的 DocumentAst（或重新从 content_list.json 构建）
2. 调用 `toEngineDocumentAst(ast)` 转换为 EngineDocumentAst 格式
3. 验证转换结果：
   - `blocks` 数组非空
   - 每个 block 有 `type`、`id` 字段
   - paragraph block 有 `runs` 数组，每个 run 有 `text` 字段
   - table block 有 `rows` 数组，每行有 `cells` 数组，每个 cell 有 `content` 数组

4. 启动 Rust 引擎进程并发送 risk.analyzeWithAst 请求：
   ```
   请求格式：
   {
     id: "test-1",
     method: "risk.analyzeWithAst",
     params: {
       projectId: "e2e-test-pdf",
       submissions: [{
         submissionId: "sub-pdf-1",
         fileHash: ast.sha256,
         ast: toEngineDocumentAst(ast)
       }, {
         submissionId: "sub-pdf-2",
         fileHash: ast2.sha256,
         ast: toEngineDocumentAst(ast2)
       }],
       baseline: null,
       preset: "standard"
     }
   }
   ```
   - 使用同一个扫描版 PDF 的 AST 作为两个 submission（模拟雷同检测场景）
   - 通过 stdin/stdout JSON-RPC 与引擎通信
   - 引擎二进制路径：`bidlens-engine/target/release/bidlens-engine.exe` 或 `bidlens-engine/target/debug/bidlens-engine.exe`

5. 验证引擎响应：
   - 响应包含 `findings` 数组
   - 至少产生 1 个 finding（相同文档应有 100% 相似度）
   - 每个 finding 有 `id`、`detectorType`、`riskLevel`、`involvedSubmissionIds`
   - `involvedSubmissionIds` 包含 "sub-pdf-1" 和 "sub-pdf-2"
   - 每个 finding 有 `evidence` 数组，且 evidence 中 `sourceOriginalText` 非空
   - 响应包含 `filePairAssessments` 数组
   - filePairAssessments 中 `symmetricSimilarity` > 0（相同文档）

**注意：**
- 如果引擎二进制不存在，在测试中标记 `it.skip` 并注明 "Engine binary not found, run: cargo build --manifest-path bidlens-engine/Cargo.toml"
- 使用 `node:child_process` 的 `spawn` 启动引擎
- JSON-RPC 通信通过 stdin 写入 JSON 行，stdout 读取 JSON 行
- 超时设置：引擎启动 5s，请求超时 300s（与 ENGINE_REQUEST_TIMEOUT_MS 一致）
- 测试完成后发送 shutdown 请求关闭引擎
- 如果 MINERU_API_TOKEN 未设置，mapper 部分使用已有的 content_list.json fixture（不需要 API 调用）
