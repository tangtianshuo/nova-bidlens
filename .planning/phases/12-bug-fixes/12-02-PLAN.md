**FIX-04: AbortSignal dead code in PDF path**

The signal never reaches MinerU's fetch calls. Three changes needed:

**1. Add `signal` to `ParseOptions` (types.ts)**

Add an optional field:
```typescript
export interface ParseOptions {
  // ... existing fields ...
  /** AbortSignal for cancellation */
  signal?: AbortSignal;
}
```

**2. Pass signal from parser-service.ts to MinerU parse**

In `parser-service.ts`, the PDF path (lines 93-98) calls `mineru.parse(input, options)` but `options` is a `ParseOptions` object that doesn't include `signal`. The `signal` is in `opts` (ParserServiceOptions) but not forwarded to `ParseOptions`.

Fix: When building `options` (line 78), include the signal:
```typescript
const options: ParseOptions = {
  // ... existing fields ...
  signal: opts?.signal,
};
```

This way, when the PDF path calls `mineru.parse(input, options)`, the signal is available.

**3. Use signal in MinerU parser (mineru/index.ts)**

In `MinerUParser.parse()`:
- Pass `options.signal` to `uploadFile`, `pollBatch`, and `downloadAndParseZip`
- Each method should accept an optional `signal` parameter
- Pass `signal` to all `fetch()` calls as `{ signal }` option

In `uploadFile()`:

**FIX-05: Hard timeout on pollBatch**

Current code: `const deadline = timeoutMs > 0 ? Date.now() + timeoutMs : Infinity;`

If `timeoutMs` is 0, deadline is Infinity — pollBatch runs forever.

Fix: Add a constant and enforce a floor:
```typescript
const MINERU_HARD_TIMEOUT_MS = 300_000; // 5 minutes max
```

In `pollBatch`:
```typescript
const effectiveTimeout = timeoutMs > 0 ? Math.min(timeoutMs, MINERU_HARD_TIMEOUT_MS) : MINERU_HARD_TIMEOUT_MS;
const deadline = Date.now() + effectiveTimeout;
```

Also add `signal?.throwIfAborted()` inside the polling loop (before each sleep) for faster cancellation response.
