**FIX-01: parserVersion hardcoded to '0.2.2'**

In `createProject()` (line 124), `parserVersion: '0.2.2'` is set at project creation time before any parsing happens. The actual parserVersion is only known after parsing completes (MinerU sets `'mineru-api-v4'`, docx4js sets its own version).

Fix: After the parsing loop completes (around line 357, after the `for` loop that populates `parsed[]`), update the project's parserVersion from the first parsed AST:

```typescript
// After parsing loop, update project parserVersion from actual parser
if (parsed.length > 0 && parsed[0].parserVersion) {
  this.projectRepo.updateParserVersion(projectId, parsed[0].parserVersion);
}
```

This requires adding an `updateParserVersion` method to the project repository. Check if it exists; if not, add it as a simple UPDATE statement.

Alternatively, if the repo doesn't have such a method, use a direct approach: after the parsing loop, call a helper that updates the project row's parser_version field. The simplest approach: add a method `updateParserVersion(projectId: string, parserVersion: string)` to the project repository that runs `UPDATE projects SET parser_version = ? WHERE id = ?`.

**FIX-03: file-validator doesn't recognize MinerU capabilities**

Current behavior: `validateFile()` uses `globalRegistry.findByExtension(ext)` which returns `pdf-parser` for PDFs. The MinerU parser is NOT in the global registry (it's instantiated lazily in parser-service.ts with a token). So PDF files always report `parserId: 'pdf-parser'` regardless of MinerU availability.

Fix: Add an optional `mineruAvailable` parameter to `validateFile`:

```typescript
export async function validateFile(filePath: string, options?: { mineruAvailable?: boolean }): Promise<FileValidationResult> {
```

When `mineruAvailable` is true and the file is a PDF:
- Set `parserId` to `'mineru-parser'` instead of `'pdf-parser'`
- Update capabilities: table dimension becomes 'supported' (MinerU extracts tables well), content stays 'supported'

In `detectCapabilities`, add handling for `mineru-parser`:
```typescript
if (parserId === 'mineru-parser') {
  // MinerU provides good table extraction for PDFs
  if (dimension === 'table') return { dimension, state: 'supported' as const };
  if (dimension === 'content') return { dimension, state: 'supported' as const };
  return { dimension, state: 'unsupported' as const, reason: 'MinerU不支持此维度' };
}
```

Update callers in risk-review-service.ts:
- Line 338: `validateFile(file.path)` -> `validateFile(file.path, { mineruAvailable: Boolean(getMinerUParser()) })`
  - But `getMinerUParser()` is in parser-service.ts. Either export it or pass the flag differently.
  - Simpler: pass `{ mineruAvailable: !!mineruConfig?.getToken() || !!process.env.MINERU_API_TOKEN }` using the existing `mineruConfig` reference.

Add a test for the new behavior in file-validator.test.ts:
```typescript
it('returns mineru-parser parserId for PDF when mineruAvailable is true', async () => {
  vi.mocked(fs.access).mockResolvedValueOnce(undefined as never);
  vi.mocked(fs.stat).mockResolvedValueOnce({ size: 1024 } as never);

  const result = await validateFile('/test/file.pdf', { mineruAvailable: true });

  expect(result.parserId).toBe('mineru-parser');
  expect(result.supported).toBe(true);
});
```
